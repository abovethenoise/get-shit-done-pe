# Phase 5: Documentation - Research

**Researched:** 2026-02-28
**Domain:** Documentation generation agent, code-reading pipeline, structured output for grep-optimized lookup
**Confidence:** HIGH

## Summary

Phase 5 creates a documentation agent that reads actual built code after review acceptance and generates structured reference docs in `.documentation/`. The agent operates in a single-agent pipeline (no gather-synthesize) with a 3-pass self-validation and user Q&A review before commit. All user decisions are locked: two doc layers (module index + flow narratives), strict heading templates for grep, section-level ownership tags (`[derived]`/`[authored]`), one-hop impact discovery, and gate docs scaffolded with universal seed content.

No new libraries. No new dependencies. The work is entirely markdown agent definitions, a workflow file, a command file, a `gsd-tools init doc-phase` CLI entry, and gate doc template scaffolding. The doc agent is a single agent (not gather-synthesize) because documentation is a sequential read-then-write task, not a parallel analysis task.

**Primary recommendation:** Build in 3 plans: (1) doc agent definition + gate doc scaffolding, (2) `init doc-phase` CLI command + docs template update, (3) doc-phase workflow + slash command with Q&A review loop.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- Two documentation layers: module index (navigation, lightweight) and flow narratives (primary docs, where the value lives)
- Function-level reference stays in the code itself (types, naming)
- Flow narrative template per flow: Trigger -> Input -> Steps -> Output -> Side effects
- Each flow gets an ASCII diagram first (visual overview), then structured steps with module ownership, then side effects
- Docs capture "what it does" primarily, with non-obvious "why" decisions as inline rationale (WHY blocks)
- Doc agent scope: flow narratives + module index (post-review). Gate docs scaffolded as infrastructure but human-maintained -- agent reads them as validation inputs, not generates them
- Directory structure: `.documentation/modules/` (flat, 1:1 with code), `.documentation/flows/` (capability-grouped, cross-module narratives), `.documentation/gate/` (scaffolded by agent, human-maintained content)
- Gate docs tagged `[manual]` to signal human ownership
- Heading templates (strict, for grep): Module docs use `## Module:`, `## Purpose:`, `## Exports:`, `## Depends-on:`, `## Constraints:`, `## WHY:`. Flow docs use `## Flow:`, `## Trigger:`, `## Input:`, `## Steps:`, `## Output:`, `## Side-effects:`, `## WHY:`
- Gate docs role: validation inputs, not agent outputs. `constraints.md` -> agent checks generated docs don't reference banned patterns. `glossary.md` -> agent enforces naming consistency. `state.md` -> agent verifies state references match
- Cross-referencing: one-way -- flows reference modules in Steps. Modules don't link back to flows. Flow steps reference modules by name, not individual functions
- Input contract: Agent reads code (what does this do), review findings (why is it this way), requirements/FEATURE.md (what was it supposed to do)
- Discovery scope: Primary = files directly modified in the reviewed change. Impact discovery = one hop via grep on existing flow docs for references to modified modules (flag only, don't auto-rewrite). Never full codebase scan
- Update strategy: section-level ownership -- `[derived]` sections overwrite freely, `[authored]` sections preserve and flag conflicts
- Impact handling: impacted flow docs flagged only -- not auto-rewritten
- Trigger model: doc agent triggers automatically after review acceptance -- final pipeline stage
- Review gate: agent presents generated docs for user Q&A review (same pattern as plan-phase and review-phase). User approves before commit
- Self-validation 3-pass: (1) structural compliance -- headings, ownership tags, anchors, timestamp. (2) referential integrity -- module names match code, exports exist, dependency references resolve. (3) consistency with gate docs -- glossary match, no banned patterns, state references match
- Gate doc seed content provided for constraints.md (7 universal constraints), glossary.md (5 universal terms), state.md (template)

- Exact wording of Purpose and Exports sections
- How to present impact flags to user during Q&A review

- Emergent docs (error taxonomy, change protocol, dependency manifest) -- grow with project, not generated by agent
- On-demand doc regeneration command -- possible future enhancement

### Pattern 1: Single-Agent Documentation Pipeline

**What:** One agent reads code + review findings + requirements, produces module docs + flow docs, runs 3-pass self-validation, writes output files.

**When to use:** Documentation is inherently sequential -- you must understand the code before writing about it. No benefit to parallelization.

**Flow:**
```
review-acceptance
       |
       v
  init doc-phase (CLI)
       |
       v
  context assembly (Layers 1-3)
       |
       v
  spawn doc agent (Task)
       |
       v
  agent reads code -> writes docs -> self-validates (3-pass)
       |
       v
  workflow reads output
       |
       v
  Q&A review with user (one-at-a-time)
       |
       v
  commit on approval
```

### Pattern 2: Section-Level Ownership Model

**What:** Each section in a doc file is tagged `[derived]` or `[authored]`. On update, agent regenerates `[derived]` sections from code, preserves `[authored]` sections, flags conflicts between old authored content and new code reality.

**When to use:** Every doc update after the initial generation.

**Example:**
```markdown
## Module: parser

## Purpose: [derived]
Extracts actions, players, amounts from PT4 hand history format.

## Exports: [derived]
- parseHand(raw: string): Hand
- parseAction(line: string): Action

## Constraints: [authored]
Must handle incomplete hand histories gracefully -- PT4 crashes mid-write are common.

## WHY: [authored]
Parser uses line-by-line streaming instead of full-text regex because PT4 files
can exceed 500MB and regex backtracking kills memory.
```

### Pattern 3: One-Hop Impact Discovery

**What:** After generating/updating module docs, grep existing flow docs for references to modified modules. Flag impacted flows for human review -- never auto-rewrite them.

**When to use:** Every doc generation run.

**Example:**
```bash
# Agent modified parser module doc
# Now check which flows reference parser
grep -r "parser" .documentation/flows/ --include="*.md"
# Result: flows/hand-import/import-flow.md references parser
# Output: "IMPACT: flows/hand-import/import-flow.md references parser -- review manually"
```

### Anti-Patterns to Avoid

- **Full codebase scan:** Agent should only read files from the reviewed change + one-hop impacts. Never crawl the entire source tree.
- **Spec rehash:** Documentation must come from reading actual code, not from paraphrasing FEATURE.md requirements. The agent reads requirements to understand intent, but writes about what the code actually does.
- **Auto-rewriting impacted docs:** Impact discovery is flagging only. Human decides if flow docs need updates.
- **Gate doc generation:** Agent scaffolds templates but never writes gate doc content. Gate docs are `[manual]`.

## Common Pitfalls

### Pitfall 1: Agent Hallucinates Code Structure

**What goes wrong:** Agent writes docs that describe functions, modules, or data flows that don't exist in the actual code.
**Why it happens:** LLMs can fabricate plausible-sounding code descriptions from training data rather than reading the actual files.
**How to avoid:** Self-validation Pass 2 (referential integrity) catches this: module names must match real code artifacts, listed exports must actually exist, dependency references must resolve.
**Warning signs:** Doc references to files that don't exist, function names not found in code, dependency chains that don't match import statements.

### Pitfall 2: Over-Documenting (Completeness Theater)

**What goes wrong:** Agent writes exhaustive docs for every function, variable, and edge case -- producing noise that obscures signal.
**Why it happens:** LLMs default to verbose, comprehensive output.
**How to avoid:** CONTEXT.md locks the scope: module-level docs (Purpose, Exports, Depends-on) and flow-level narratives. Function-level detail stays in code. Agent prompt must reinforce this boundary.
**Warning signs:** Docs longer than the code they describe. Individual function documentation in module docs. Copy-paste of type signatures.

### Pitfall 3: Stale Gate Doc References

**What goes wrong:** Agent's Pass 3 validation fails because gate docs haven't been populated yet (they're scaffolded empty, human fills them later).
**Why it happens:** Gate docs are templates at scaffolding time. Pass 3 checks against empty content = vacuous pass.
**How to avoid:** Pass 3 should handle empty/template gate docs gracefully: skip validation against empty sections, note "gate doc not yet populated" in output.
**Warning signs:** Pass 3 reports zero violations on a non-trivial codebase (suspicious).

### Pitfall 4: Update Conflicts Between Derived and Authored

**What goes wrong:** Agent overwrites `[authored]` sections because it fails to parse ownership tags, or flags false conflicts because code evolved but authored content is still valid.
**Why it happens:** The section-level ownership model requires the agent to parse existing docs, identify tags, and make judgment calls about conflicts.
**How to avoid:** Agent prompt must include explicit instructions for parsing `[derived]`/`[authored]` tags. When in doubt, preserve and flag -- never overwrite `[authored]`.
**Warning signs:** Authored content disappearing after updates. Excessive false-positive conflict flags.

### Pitfall 5: Docs Template vs CONTEXT.md Mismatch

**What goes wrong:** The existing `docs.md` template (v1) uses a different structure (design.md / features.md / lessons.md per capability) than CONTEXT.md specifies (modules/ + flows/ + gate/).
**Why it happens:** Template was created in Phase 1 before the user's doc-phase discussion decisions.
**How to avoid:** Template must be rewritten to match CONTEXT.md. The v1 template's 3-file structure (design, features, lessons) is superseded by the CONTEXT.md 3-directory structure (modules, flows, gate).
**Warning signs:** Agent producing docs matching the old template format instead of CONTEXT.md decisions.

## Constraints Discovered

- **Single agent, not gather-synthesize** -- Documentation is a sequential read-then-write task. The doc agent must understand the full code context before writing any doc. No benefit to parallel decomposition. This is a departure from the research (6 gatherers) and review (4 gatherers) patterns.

- **Existing docs.md template is misaligned** -- The v1 template produces `design.md`, `features.md`, `lessons.md` per capability. CONTEXT.md specifies `modules/`, `flows/`, `gate/` directories. Template must be rewritten to match CONTEXT.md decisions.

- **Gate docs are scaffolding-only** -- The agent writes the gate doc template files with seed content during initial setup. After that, gate docs are read-only inputs to the agent's Pass 3 validation. The agent never modifies gate doc content after scaffolding.

- **Q&A interaction only available in orchestrator** -- Same constraint as review-phase. The doc agent writes output files; the workflow reads them and presents to user via AskUserQuestion. Agent cannot interact with user.

- **AskUserQuestion header max 12 characters** -- Same constraint as plan-phase and review-phase. Doc review headers must be abbreviated (e.g., "Doc 1/3").

- **File-based result collection only** -- Orchestrator cannot receive agent output in context. Doc agent writes `.documentation/` files. Workflow verifies file existence, then presents for Q&A.

- **Context provided by orchestrator, not fetched by agent** -- Doc agent receives file paths and context at spawn time. Agent definition must NOT contain "read STATE.md" instructions. Consistent with AGNT-02 requirement.

- **CommonJS only for CLI tooling** -- Any doc-phase CLI code (init command) must use `.cjs` with `require()`. Consistent with Phase 2 constraint.

- **Git SHA via `git rev-parse HEAD`** -- The `built-from-code-at` frontmatter field requires the current git SHA. Agent obtains this via Bash tool. Workflow can also pass it in context.

- **Summary files are the artifact manifest** -- The doc agent discovers what was built by reading `*-SUMMARY.md` files from the phase directory. This is the established pattern from review-phase (step 3: Locate Phase Artifacts).

## Open Questions

1. **Doc agent role_type: executor or judge?**
   - What we know: The agent reads code and writes documentation. It makes judgment calls about what's important, but its primary task is content generation, not verdiction.
   - What's unclear: Whether the 3-pass self-validation (which includes judgment about referential integrity and gate doc consistency) pushes it into "judge" territory.
   - Recommendation: Use `role_type: executor` (Sonnet). The self-validation is structural checking, not subjective judgment. The user Q&A review is the actual quality gate. If validation accuracy is insufficient, can be revisited.

2. **How does the workflow discover which phase was just reviewed?**
   - What we know: Doc-phase triggers after review acceptance. The review-phase workflow completes with artifacts in `{phase_dir}/review/`.
   - What's unclear: Whether doc-phase receives the phase number as an argument or reads it from STATE.md.
   - Recommendation: Accept phase as a CLI argument (same as review-phase). `init doc-phase <phase>` resolves everything from there.

3. **Impact flag presentation in Q&A**
   - What we know: CONTEXT.md says impacted flow docs are "flagged only." User discretion area says "how to present impact flags."
   - What's unclear: Whether impact flags should be separate Q&A items or appended to the generated doc review.
   - Recommendation: Present as a separate section after doc review Q&A: "The following existing flow docs may be affected by these changes: [list]. Review them manually."
