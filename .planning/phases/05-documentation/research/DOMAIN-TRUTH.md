**Requirements addressed:** DOCS-01, DOCS-02, DOCS-03

## First Principles

- **FP-1: Documentation generated from code is only as accurate as the LLM's comprehension of the code it reads -- and LLMs understand syntax far better than semantics.** Research on LLM code analysis shows models excel at parsing structure (exports, imports, function signatures) but struggle with dynamic semantics -- runtime behavior, state transitions, side effects. The implication for doc generation: `[derived]` sections covering structural facts (Purpose, Exports, Depends-on) will be highly accurate. Sections requiring behavioral understanding (flow Steps, Side-effects) carry higher hallucination risk because they require the model to trace execution paths, not just read declarations. The 05-CONTEXT decision to use flow narratives with ASCII diagrams is a mitigation -- forcing the agent to articulate data flow step-by-step surfaces reasoning gaps that would hide in prose. [Source: [Large Language Models for Code Analysis: Do LLMs Really Do Their Job? -- USENIX Security 2024](https://www.usenix.org/system/files/sec24fall-prepub-2205-fang.pdf)] [Source: [LLMs: Understanding Code Syntax and Semantics for Code Analysis](https://arxiv.org/pdf/2305.12138)] [Connects to: DOCS-01 -- agent reads actual code; accuracy varies by what it reads for]

- **FP-2: A documentation agent that generates content and verifies it in the same pass will copy its own hallucinations.** Chain-of-Verification research (Meta, 2023; confirmed across 2024-2025 follow-ups) demonstrates that when an LLM verifies its own output while the draft is visible, it re-anchors to the same errors. The critical mechanism is independence: verification questions must be answered without conditioning on the draft. The 05-CONTEXT 3-pass self-validation is structurally sound IF each pass evaluates against an external anchor (headings template, actual code artifacts, gate docs) rather than against the generated text itself. Pass 1 checks structure against a template (external). Pass 2 checks references against real files (external). Pass 3 checks terminology against glossary (external). All three passes use external anchors -- this is correct and validated by CoVe research. [Source: [Chain-of-Verification Reduces Hallucination in Large Language Models -- ACL 2024](https://aclanthology.org/2024.findings-acl.212/)] [Source: [Chain of Verification in LLM Evaluations -- Rohan Paul](https://www.rohan-paul.com/p/chain-of-verification-in-llm-evaluations)] [Connects to: DOCS-01, DOCS-03 -- 3-pass validation architecture]

- **FP-3: The "what" vs "why" distinction is not a documentation style choice -- it is a cognitive task separation that determines accuracy.** "What does this code do" can be answered by reading the code. "Why was it built this way" cannot -- it requires access to decision context that does not exist in source files. An LLM asked "why" without access to decision records will confabulate plausible-sounding rationales. The 05-CONTEXT decision to source "what" from code, "why" from review findings, and "intent" from FEATURE.md is a three-input architecture that gives each question type its correct evidence source. WHY blocks that lack a traceable review finding should not be generated at all -- better to have no WHY than a hallucinated one. [First principles: asking an LLM to explain "why" code exists is asking it to reconstruct decisions it was not present for. Without the actual decision record as input, it will generate the most statistically likely explanation, which may be plausible but false.] [Connects to: DOCS-01 -- three input sources for three question types]

- **FP-4: Documentation that cannot be found is documentation that does not exist.** The value of reference docs is entirely in retrieval speed. A developer (human or AI) searching for "what does the parser module do" must find the answer in under 5 seconds of grep/search. This requires: (a) predictable file locations (flat directory, 1:1 with code modules), (b) canonical heading formats (grep for `## Module: parser` always works), (c) consistent terminology (glossary-enforced). The 05-CONTEXT heading templates and flat module directory directly serve this principle. Any heading variation -- even `## Module: Parser` vs `## Module: parser` -- breaks grep consistency. Case sensitivity must be enforced in Pass 1. [First principles: documentation ROI = (lookup frequency * time saved per lookup) - (generation cost + maintenance cost). If lookup time is not near-zero due to inconsistent structure, the ROI equation collapses regardless of content quality.] [Connects to: DOCS-03 -- optimized for quick lookups and mgrep searches]

- **FP-5: Incremental documentation must distinguish between "regenerate freely" and "preserve with care" -- section ownership is the mechanism.** Documentation systems that treat all content as regenerable lose human-authored context on every update cycle. Systems that treat all content as sacred become stale because nothing gets refreshed. The `[derived]`/`[authored]` ownership model in 05-CONTEXT solves this by making the boundary explicit at the section level. This maps to an established pattern in continuous documentation: derived content (API references, export lists) is cheap to regenerate and expensive to maintain manually; authored content (constraints, rationale) is cheap to maintain manually and dangerous to regenerate. The agent must parse by heading anchors, respect ownership tags, and never silently overwrite `[authored]` sections. [Source: [Shifting to Continuous Documentation -- InfoQ](https://www.infoq.com/articles/continuous-documentation/)] [Connects to: DOCS-02 -- section-level ownership in .documentation/ directory]

## Universal Constraints

- **UC-1: LLM cross-file reasoning degrades with the number of files in context.** Research benchmarks show dramatic performance degradation as context length increases for multi-file code understanding tasks. The 05-CONTEXT discovery scope constraint -- primary (modified files) + one-hop impact (grep existing docs for references) + never full codebase scan -- is not just a performance optimization. It is an accuracy constraint. An agent given 50 files to document will produce worse docs for each file than an agent given 5 files. The one-hop limit is the correct scope boundary: it keeps the agent's context focused on files it can reason about accurately while flagging (not rewriting) docs that may be affected downstream. [Source: [LoCoBench: A Benchmark for Long-Context LLMs](https://arxiv.org/pdf/2509.09614)] [Connects to: DOCS-01 -- discovery scope controls context size, which controls accuracy]

- **UC-2: Referential hallucination -- inventing exports, dependencies, or function names that do not exist -- is the highest-probability failure mode for code documentation.** Package hallucination research shows all 16 tested coding models hallucinated references at an average rate of 19.6%. For documentation specifically, this manifests as: listing exports that do not exist in the module, referencing dependencies the module does not import, naming functions with plausible but incorrect names. Pass 2 of the 3-pass validation (referential integrity check) directly targets this failure mode. It must verify: every export name exists in the actual file, every dependency reference resolves to a real module, every flow step references a real module in `.documentation/modules/`. This pass is the single highest-value validation step. [Source: [Package Hallucinations: Comprehensive Analysis -- USENIX 2025](https://www.usenix.org/publications/loginonline/we-have-package-you-comprehensive-analysis-package-hallucinations-code)] [Source: [Detecting and Correcting Hallucinations in LLM-Generated Code](https://arxiv.org/pdf/2601.19106)] [Connects to: DOCS-01, DOCS-03 -- referential integrity is prerequisite for lookup reliability]

- **UC-3: Git SHA staleness tracking is necessary but insufficient -- staleness is semantic, not temporal.** A file can change its formatting without changing its behavior (SHA changes, docs still valid). A file can change one function's internals without affecting its exports (SHA changes, module doc still valid). Conversely, a dependency's interface change can make docs stale even if the documented file's SHA has not changed. The `built-from-code-at:` git SHA in 05-CONTEXT is the right staleness signal for the common case (code changed, docs may be outdated), but the one-hop impact discovery covers the dependency case (upstream changed, downstream docs may need flags). Together, SHA tracking + impact flags provide two complementary staleness signals. [First principles: staleness is about semantic drift between code behavior and documentation claims, not about file modification timestamps. SHA is a proxy for semantic change -- a good proxy, but not perfect. The impact flag system compensates for the cases where SHA misses.] [Connects to: DOCS-02 -- git SHA staleness detection in .documentation/]

- **UC-4: One-way cross-references (flows reference modules, modules do not link back) prevent reference cycles and maintenance cascading.** If modules linked back to flows, every new flow referencing a module would require updating the module doc. This creates N-to-M maintenance where N flows reference M modules. The 05-CONTEXT one-way rule (flows -> modules only) means: adding a new flow never requires updating any module doc. Modifying a module triggers one-hop impact discovery on flow docs (grep for module name in flows/), but the module doc itself is self-contained. This is the correct asymmetry -- modules are stable reference points; flows are the navigational layer that changes more frequently. [First principles: in any documentation system, reference direction determines maintenance burden. Bidirectional references create O(N*M) maintenance. Unidirectional references create O(N+M) maintenance. The cost difference compounds with project growth.] [Connects to: DOCS-02, DOCS-03 -- directory structure and cross-referencing strategy]

- **UC-5: Gate docs as validation inputs (not agent outputs) enforce a human-authored constraint boundary the agent cannot cross.** The 05-CONTEXT decision that the agent reads gate docs but does not write them is a trust boundary. Constraints, glossary terms, and state definitions are human judgments -- the agent cannot author "no-premature-abstraction" because that is a design philosophy, not a code-derivable fact. Making the agent responsible for reading and enforcing gate doc content during Pass 3 (without modifying it) is the correct separation: the agent is an enforcer of human-defined standards, not an author of standards. The `[manual]` tag on every gate doc entry reinforces this boundary visually and structurally. [First principles: the agent's epistemic scope is "what the code does." Gate docs define "what the code should respect." These are different knowledge domains owned by different authorities. Conflating them produces either gate docs that parrot the code (useless) or code docs that invent constraints (dangerous).] [Connects to: DOCS-01 -- input contract specifying three distinct sources]

## Validated Assumptions

- **VA-1: The 3-pass self-validation architecture maps directly to Chain-of-Verification best practices, with the critical advantage that all three passes use external anchors.** CoVe research shows that self-verification accuracy depends on independence from the draft. The 05-CONTEXT passes are: (1) structural compliance against heading template, (2) referential integrity against actual code files, (3) consistency against gate docs. None of these passes ask "does this doc read well?" or "is this doc accurate?" -- they each ask a concrete, externally-verifiable question. This is structurally superior to a single "review your own output" pass, which CoVe research shows degrades to self-confirmation. The external-anchor approach achieves the 21-62% error reduction documented in code verification research. [Source: [Self-Verification Prompting -- LearnPrompting](https://learnprompting.org/docs/advanced/self_criticism/self_verification)] [Source: [Chain of Verification -- Rohan Paul](https://www.rohan-paul.com/p/chain-of-verification-in-llm-evaluations)] [Connects to: DOCS-01]

- **VA-2: Module-level granularity in flow references (not function-level) is the correct abstraction boundary for LLM-generated cross-module documentation.** LLMs struggle with precise function-level attribution across files but reliably identify module-level responsibilities. The 05-CONTEXT rule "parser -> extracts actions" (not "parser.parseHand()") matches the accuracy boundary: module-level claims are verifiable by checking exports; function-level claims require tracing internal implementation, which is where semantic comprehension degrades. This also improves doc stability -- internal function renames do not require flow doc updates. [First principles: documentation granularity should match the accuracy granularity of the generator. An LLM reading a module can reliably state what the module does; stating what each function does requires deeper semantic analysis that is less reliable. Module-level is the sweet spot between usefulness and accuracy.] [Connects to: DOCS-03 -- structured for lookups at the right granularity]

- **VA-3: The trigger model (doc agent fires after review acceptance) ensures the agent documents verified code, not draft code.** If documentation ran before review, the agent would document code that might be rejected or modified. If documentation ran during review, the agent's context would compete with reviewer context. Post-review-acceptance is the only correct trigger point because: (a) the code is in its final reviewed state, (b) review findings are available as WHY source material, (c) no further code changes will invalidate the docs before commit. The pipeline order execute -> review -> accept -> document is a dependency chain, not a preference. [First principles: documentation of unreviewed code creates docs that may describe rejected implementations. The review gate ensures the doc agent only documents code the project has accepted.] [Connects to: DOCS-01 -- reflects actual built code]

## Domain Risks

- **DR-1: Flow narrative generation requires multi-file reasoning, which is the documented weak point of LLM code comprehension.** Module docs are single-file tasks (read one file, describe it). Flow narratives require reading multiple modules and understanding how data passes between them. This is the cross-file reasoning task where LLMs show "dramatic performance degradation" as context grows. Risk: flow docs will have higher error rates than module docs, particularly for flows spanning 4+ modules. Mitigation: the 05-CONTEXT input contract provides FEATURE.md (what was intended) as a reasoning scaffold -- the agent traces the expected flow against actual code rather than discovering the flow purely from code. The ASCII diagram requirement also helps: forcing the agent to draw the flow before describing it creates a structural checkpoint that surfaces reasoning errors early. [Connects to: DOCS-01]

- **DR-2: The `[authored]` preservation mechanism depends on reliable heading-anchor parsing, which is brittle against formatting drift.** If the doc agent fails to parse an existing doc's heading anchors correctly -- due to extra whitespace, case variation, or non-standard formatting -- it may overwrite `[authored]` sections by treating them as new content. Risk: silent loss of human-authored rationale during doc regeneration. Mitigation: Pass 1 (structural compliance) must validate heading anchors before any content generation begins. If anchors cannot be parsed reliably, the agent must abort the update and flag the doc for manual review rather than proceeding with potentially destructive overwrites. [First principles: a safety mechanism that can be silently bypassed by formatting inconsistency is not a safety mechanism. Heading-anchor parsing must be a hard gate, not a best-effort attempt.] [Connects to: DOCS-02]

- **DR-3: WHY blocks sourced from review findings risk inheriting reviewer speculation as documented rationale.** Review findings may include the reviewer's interpretation of why code was written a certain way, which is not the same as the actual reason. If the doc agent transcribes a reviewer's speculation into a WHY block, that speculation becomes "documented rationale" that future agents and developers will treat as authoritative. Risk: compounding speculation across pipeline stages (reviewer guesses -> doc records guess -> future planner builds on recorded guess). Mitigation: WHY blocks should only be generated from review findings that explicitly cite decision context (e.g., "per CONTEXT.md decision X" or "review accepted with rationale Y"). Reviewer speculation without citations should not become WHY blocks. [First principles: documentation is an amplifier. It amplifies correct information (valuable) and incorrect information (dangerous) equally. The input quality of WHY sources determines the output quality of WHY blocks.] [Connects to: DOCS-01 -- review findings as input source]

- **DR-4: Impact discovery via grep is only as good as the consistency of module name references in existing flow docs.** One-hop impact discovery works by grepping flow docs for references to modified module names. If flow docs reference modules inconsistently (e.g., "the reader module" vs "reader" vs "Reader"), grep will miss impact relationships. Risk: modified modules that should trigger flow doc review flags are silently missed. Mitigation: Pass 3's glossary enforcement and the module-level naming convention (always use exact code module name, never natural language descriptions) must be established from the first generated doc onward. Retroactive consistency fixes are expensive; initial consistency is cheap. [Connects to: DOCS-02, DOCS-03]

- **DR-5: The gate doc scaffolding (seed content) will be the only gate doc content for a significant period, since project-specific entries require human authorship during development.** The universal seed constraints (no-implicit-state, no-unnecessary-deps, etc.) are valuable but generic. Project-specific constraints, glossary terms, and state entries will be sparse until the project reaches sufficient maturity. Risk: Pass 3 validation against sparse gate docs provides a false sense of completeness -- the agent checks against 7 universal constraints but misses project-specific naming conventions or state shapes that have not been documented yet. Mitigation: this is acceptable and should not be over-engineered. Gate docs grow organically. The validation pass should report what it checked against, making the sparse coverage visible rather than hiding it. [First principles: a validation pass that clearly reports "checked 7 constraints, 5 glossary terms, 0 state entries" is honest about its coverage. A pass that silently passes because there is nothing to check creates false confidence.] [Connects to: DOCS-02]
