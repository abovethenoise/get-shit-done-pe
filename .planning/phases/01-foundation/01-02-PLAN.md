  <name>Task 1: Create v2 artifact templates</name>
  <files>get-shit-done/templates/capability.md, get-shit-done/templates/feature.md, get-shit-done/templates/review.md, get-shit-done/templates/docs.md</files>
  <action>
    Create four template files. These are human-readable reference documents consumed by Claude agents (NOT by code — template.cjs hardcodes content inline). Each template must have YAML frontmatter with a `type` field as artifact discriminator (per research: Kubernetes `kind` pattern).

    **capability.md** — CAPABILITY.md template:
    Frontmatter: `type: capability`, `status: planning`, `created: {date}`.
    Body sections (per CONTEXT.md locked decisions):
    - `## Goal` — one sentence, what this capability delivers to the user
    - `## Domain Model` — shared entities, relationships, vocabulary (table format)
    - `## Invariants` — rules that apply across ALL features in this capability (numbered list)
    - `## Boundaries` — subsections: Owns, Consumes, Does Not Touch
    - `## Architecture Spine` — data flow, layer ownership, where logic lives
    - `## Dependencies` — cross-capability: Produces, Consumes (table format)
    - `## Features` — table: Feature | Status (planning/building/complete)
    - `## Decisions` — append-only log: Date | Decision | Context | Tradeoffs

    **feature.md** — FEATURE.md template with 3-layer requirements:
    Frontmatter: `type: feature`, `capability: {slug}`, `status: planning`, `created: {date}`.
    Body sections (per CONTEXT.md: trace table at TOP):
    - `## Trace Table` — table at TOP: REQ ID | Layer | Status | Plan | Review | Docs (per research: populated progressively)
    - `## End-User Requirements` — Layer 1 (EU-xx):
      For each requirement: Story (As a [who], I want [what], so that [why]), Acceptance Criteria (checklist), Out of Scope
    - `## Functional Requirements` — Layer 2 (FN-xx):
      For each requirement: Receives (inputs/triggers), Returns (outputs/side effects), Behavior (rules/logic/edge cases/errors)
    - `## Technical Specs` — Layer 3 (TC-xx):
      For each requirement: Intent (why this approach), Upstream, Downstream, Constraints, Example
    - `## Decisions` — section at bottom for notes

    **review.md** — REVIEW.md template:
    Frontmatter: `type: review`, `feature: {slug}`, `capability: {slug}`, `status: pending`, `verdict: pending`.
    Body sections (per CONTEXT.md):
    - `## Summary` — Verdict (PASS/PARTIAL/FAIL/BLOCKED), counts: pass/partial/fail
    - `## Per-Requirement Trace` — for each REQ ID: Verdict, Evidence (multi-dimensional: code, domain, integration), Gap Analysis, Fix Scope, Blocking Assessment
    - `## Reviewer Notes` — subsections: Domain Reviewer, Code Reviewer, Integration Reviewer (captures concerns not tied to specific REQs)

    **docs.md** — Documentation generation template:
    Frontmatter: `type: docs`, `capability: {slug}`, `built_from_code_at: {git-sha}`.
    Body: Instructions for generating three output files:
    - `design.md` — architecture decisions, data flow, why things are structured this way
    - `features.md` — per-feature reference: what it does, how to use it, key functions/modules
    - `lessons.md` — things learned, gotchas, patterns that worked/didn't
    Each output targets `.documentation/{capability}/` directory.
    Include guidance: YAML frontmatter on all output files, optimized for mgrep searches, structured for the four framings (new, extend, debug, refactor).

    Template files should use placeholder values like `{capability}`, `{feature}`, `{date}`, `{slug}` where appropriate.

  <name>Task 2: Add findCapabilityInternal and findFeatureInternal to core.cjs</name>
  <files>get-shit-done/bin/lib/core.cjs, tests/core.test.cjs</files>
  <behavior>
    - findCapabilityInternal(cwd, slug) returns { found: true, directory, slug, capability_path } when .planning/capabilities/{slug}/CAPABILITY.md exists
    - findCapabilityInternal(cwd, slug) returns { found: false } when capability does not exist
    - findCapabilityInternal(cwd, slug) returns { found: false } when directory exists but CAPABILITY.md is missing (partial creation)
    - findFeatureInternal(cwd, capSlug, featureSlug) returns { found: true, directory, slug, feature_path, capability_slug } when .planning/capabilities/{cap}/features/{feat}/FEATURE.md exists
    - findFeatureInternal returns { found: false } for missing feature
    - findFeatureInternal returns { found: false } for missing parent capability
    - generateSlugInternal handles empty-after-sanitization input (unicode → empty string) with error, not crash
    - generateSlugInternal prevents slash injection in slug
  </behavior>
  <action>
    Add two new internal helper functions to core.cjs following existing `findPhaseInternal` pattern:

    1. `findCapabilityInternal(cwd, capabilityInput)`:
       - Slugify input via `generateSlugInternal`
       - Check for edge case: empty slug after sanitization → return `{ found: false, reason: 'empty_slug' }`
       - Build path: `path.join(cwd, '.planning', 'capabilities', slug)`
       - Check directory exists AND `CAPABILITY.md` exists inside it (per research: detect partial creation)
       - Return: `{ found: true, directory, slug, capability_path }` or `{ found: false, slug, reason }`

    2. `findFeatureInternal(cwd, capabilitySlug, featureInput)`:
       - Slugify featureInput via `generateSlugInternal`
       - Validate parent capability exists via `findCapabilityInternal` first
       - If parent not found → return `{ found: false, reason: 'capability_not_found', capability_slug: capabilitySlug }`
       - Build path: `path.join(capDir, 'features', featureSlug)`
       - Check directory exists AND `FEATURE.md` exists
       - Return: `{ found: true, directory, slug, feature_path, capability_slug }` or `{ found: false, ... }`

    3. Harden `generateSlugInternal`:
       - After sanitization, if result is empty string → return empty string (callers check)
       - Reject input containing path separator (`/`, `\\`) before slugification — return empty string

    Add all three functions to `module.exports` at bottom of core.cjs.
    Add tests to core.test.cjs for the new functions using temp directory fixtures.
  </action>
