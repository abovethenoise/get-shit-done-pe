Apply Phase 2 goldilocks principles from DOMAIN-TRUTH.md and TECH-CONSTRAINTS.md to each agent. The pattern for each agent is identical:

**Step 1: Audit the current agent.** Count lines, sections, tables, MUST/CRITICAL instances. Identify what belongs in the agent definition (role, goal, success criteria, output format, reads/writes) vs what belongs in a reference file (tables, examples, anti-patterns, templates, detailed step procedures).

**Step 2: Extract to reference file.** Move all tables, examples, templates, anti-patterns, detailed procedures, and edge case catalogs to `get-shit-done/references/{agent}-reference.md`. The reference file is included via `@reference` by the orchestrator when spawning the agent -- it goes into the agent's 200k context window as prompt content, not into the agent definition itself.

**Step 3: Rewrite the agent definition.** The agent .md file should contain ONLY:
- YAML frontmatter with: name, description, tools, color, role_type (executor|judge), reads (list of file types this agent reads), writes (list of file types this agent writes)
- Role section: 3-5 sentences. Who you are, what you do, who spawns you.
- Goal section: 1-2 sentences. The success state. Goal-driven, not checklist-driven.
- Success criteria: 3-5 bullet points. Observable outcomes. Positive framing ("plans have X" not "plans MUST NOT lack X").
- Output format: Brief description of what the agent produces and where.
- Downstream consumers: 1-2 sentences. Who reads your output and what they need from it.

**Step 4: Add v2 awareness to each agent.** Every rewritten agent must reference:
- FEATURE.md as the requirements source (EU/FN/TC 3-layer structure)
- CAPABILITY.md for capability-level context (what + why + feature list)
- Feature directory model: `.planning/capabilities/{cap}/features/{feat}/`
- The capability/feature directory model replaces phase directories
