**slug-resolve CLI route (in core.cjs + gsd-tools.cjs):**

Add a new `slug-resolve` route to gsd-tools.cjs that calls a new function in core.cjs:

```
gsd-tools slug-resolve <input> [--type capability|feature|auto]
```

Returns JSON:
```json
{
  "tier": 1,
  "resolved": true,
  "type": "capability",
  "capability_slug": "coaching",
  "feature_slug": null,
  "full_path": "coaching",
  "candidates": [],
  "reason": "exact"
}
```

Implementation in core.cjs -- `resolveSlug(cwd, input, typeHint)`:
- **Tier 1 (Exact):** If input contains `/`, split into cap/feat and call `findFeatureInternal(cwd, cap, feat)`. If no `/`, call `findCapabilityInternal(cwd, input)`.
- **Tier 2 (Fuzzy):** If not found, load full capability + feature list. Run substring matching against all slug names. If unique match, return resolved. If multi-match, return `{resolved: false, candidates: [...], reason: 'ambiguous'}`.
- **Tier 3 (Fall-through):** If zero matches, return `{resolved: false, candidates: [], reason: 'no_match'}`.

Add route case in gsd-tools.cjs: `case 'slug-resolve':` calling the new function.

Per CONTEXT.md: "Steps 1-2 in CLI route, step 3 falls through to workflow" (workflow = Claude interprets intent).

**capability-orchestrator.md (~150 lines):**

Thin orchestration workflow that reads CAPABILITY.md, gets prioritized feature list, calls framing-pipeline per feature in DAG order. Per CONTEXT.md: "Same pattern as execute.md wave orchestration but for features."

Structure:
1. **Inputs:** CAPABILITY_SLUG, LENS (from invoking command)
2. **Read CAPABILITY.md:** Extract features table (Feature | Priority | Depends-On | Status)
3. **Build DAG:** Topological sort features by depends-on relationships
4. **Group into waves:**
   - Wave 1: features with no deps or all deps complete
   - Wave 2+: features whose deps are all in earlier waves
5. **For each wave (in order):**
   a. Display wave plan: "Wave N: feat-a, feat-b [parallel if enabled]"
   b. User confirms or adjusts
   c. For each feature in wave:
      - Check if DISCOVERY-BRIEF.md exists in feature dir
      - If not: invoke framing-discovery.md for this feature first
      - Invoke framing-pipeline.md with FEATURE_SLUG + CAPABILITY_SLUG + LENS
      - Wait for pipeline completion
      - If failed: pause, present error to user, ask whether to continue or abort
   d. Update feature status in CAPABILITY.md
6. **Completion:** Update capability status. Update ROADMAP.md focus group checklist if applicable.

Per CONTEXT.md: "Pipeline passes feature only. Stages derive capability from feature's directory path." The orchestrator passes FEATURE_SLUG and FEATURE_DIR; capability is derived from the path.

**get-shit-done/workflows/focus.md (~180 lines):**

Per CONTEXT.md: "/gsd:focus command with Q&A (goals, priority caps/features) -> quick dependency ordering -> ROADMAP.md update."

Structure:

1. **Initialize:** Call `gsd-tools init feature-progress` to load all capabilities + features with status. Read existing ROADMAP.md to get current active focus groups.

2. **Q&A (Goal):** Ask: "What is the goal of this sprint/focus? Give it a name and one-sentence goal."
   - Capture: focus_group_name, focus_group_goal
   - Validate name doesn't collide with existing focus groups

3. **Q&A (Scope):** Ask: "Which capabilities or features are in scope? List them in rough priority order."
   - User lists items (free text)
   - For each item: call `gsd-tools slug-resolve "$ITEM"` to resolve
   - If ambiguous: present candidates inline, user picks
   - If no_match: ask if user wants to create new capability/feature or re-describe
   - Build resolved_items list: `[{type, capability_slug, feature_slug, full_path}]`

4. **Dependency trace -- how the DAG is built:**
   For each resolved item, trace dependencies:

   **4a. Explicit dependencies (from metadata):**
   - Read CAPABILITY.md Features table `Depends-On` column for each feature
   - Read FEATURE.md frontmatter `depends_on` field (if present)
   - Build edges: `feature_a -> depends_on -> [feature_b, feature_c]`

   **4b. Implicit dependencies (from code overlap detection):**
   - For each feature in scope, read FEATURE.md TC-xx requirements for file paths mentioned (Upstream/Downstream fields)
   - Run `mgrep` across feature directories for shared file path references: files that appear in multiple features' TC requirements
   - If shared files found: these features have implicit coupling
   - Surface implicit deps to user: "Features X and Y both reference {file}. This suggests a dependency."

   **4c. DAG construction:**
   - Combine explicit + confirmed implicit edges into a directed acyclic graph
   - Topological sort to determine execution waves
   - Detect cycles: if cycle found, present to user for resolution (which dep to break?)
   - Produce wave groups: Wave 1 (no deps), Wave 2 (deps on Wave 1), etc.

5. **Overlap detection -- merge-vs-parallel logic:**
   - Read existing active focus groups from ROADMAP.md
   - For each existing group, extract its feature set
   - Compare new focus group's feature set against existing groups
   - **If overlap found:**
     a. Show: "Feature {X} is already in focus group '{existing_name}'. Options:"
        - (a) Merge into existing group (add new items, reprioritize)
        - (b) Create parallel group (both groups active, features shared)
        - (c) Remove overlapping item from new group
     b. User confirms per overlapping item
   - **If no overlap:** proceed to ordering

6. **Priority ordering:** Present final priority order based on:
   - User's original ordering (from Q&A step 3)
   - Adjusted by DAG constraints (if user put X before Y but X depends on Y, flag this)
   - User confirms or reorders

7. **Write to ROADMAP.md:** Add new focus group section using the exact format from CONTEXT.md:
   ```markdown
   ### Focus: {focus_group_name}
   **Goal:** {focus_group_goal}
   **Priority Order:**
   1. {cap}/{feat} -> depends: none
   2. {cap}/{feat} -> depends: {dep}
   **Status:**
   - [ ] {cap}/{feat} (not started)
   ```

8. **Update STATE.md:** Set `active_focus` to the new focus group name. If multiple focus groups exist, STATE.md tracks all of them under `## Active Focus Groups`.

9. **Commit:** `docs: create focus group '{focus_group_name}'`
