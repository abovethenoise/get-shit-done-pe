### First Principles

- **FP-1: Problem framing is not ceremony — it is the highest-leverage moment in the pipeline.** The cost of a requirements defect, if caught at the requirements phase, is effectively free. The same defect caught post-execution is 6-30x more expensive to fix. Software defects that originate in requirements account for 70-85% of total rework cost. This is the empirical justification for the framing discovery step before pipeline convergence: every token spent in discovery preventing a misdirected pipeline run pays back orders of magnitude. For AI-assisted coding specifically, where execution is fast and cheap but direction is hard to correct, this asymmetry is even more extreme. [Source: [The Cost of Finding Bugs Later in the SDLC — Functionize](https://www.functionize.com/blog/the-cost-of-finding-bugs-later-in-the-sdlc)] [Source: [What Is the Cost of a Requirement Error — StickyMinds](https://www.stickyminds.com/article/what-cost-requirement-error)] [Connects to: WKFL-01 through WKFL-05 — all four framing commands exist to ensure discovery happens before pipeline convergence]

- **FP-2: The framing of a problem structurally constrains the solution space, independent of the actual problem.** When the same problem is framed as a debugging task vs. a new feature task, different reasoning chains activate, different questions get asked, and different solution structures emerge. This is documented cognitive science: "Different communicative frames lead people to construct different situation models, which then inform judgments and predictions." For an LLM-powered system, this effect is amplified — framing at the start of a context window primes everything that follows. A `debug` entry point that establishes "symptom → hypothesis → root cause" as the cognitive mode will resist premature solution-jumping. A `new` entry point that establishes "problem space → constraints → shape" will resist over-scoping. The four framing entry points are not just UX affordances — they are mechanisms to enforce appropriate reasoning modes. [Source: [Framing (social sciences) — Wikipedia](https://en.wikipedia.org/wiki/Framing_(social_sciences))] [Source: [Source framing triggers systematic bias in large language models — Science Advances](https://www.science.org/doi/10.1126/sciadv.adz2924)] [First principles: LLMs are trained to predict the continuation of a context; the framing establishes the probability distribution for all continuations. Framing is not merely labeling — it is configuration.]

- **FP-3: Discovery must converge to a written artifact, not a shared mental model.** Conversations produce a fleeting shared understanding that dissipates at context reset. Written artifacts — the Discovery Brief — encode that understanding into a persistent, inspectable contract. This is why the Discovery Brief is defined as the handoff artifact: it ensures that what was understood during discovery is faithfully carried into research, requirements, plan, execute, and review. The one-sentence problem statement requirement ("if you can't, discovery isn't done") is a direct test of this principle — it forces externalization of understanding into verifiable form. A mental model only both parties think they share is not a contract; a written problem statement is. [First principles: Shared understanding between a human and a stateless LLM exists only in the context window. When the context resets, only written artifacts survive. Therefore, discovery that ends without a written artifact is a liability, not an asset.] [Connects to: WKFL-06 — pipeline convergence requires the Discovery Brief as handoff]

- **FP-4: Excessive context at the start of discovery increases ambiguity rather than reducing it.** The research on conversational AI workflows states explicitly: "Contextualization should be paired with user choice to filter and prioritize relevant dimensions. Excessive context actually increases ambiguity rather than reducing it." This validates the "fixed skeleton + adaptive muscles" architecture: a small set of anchor questions (3-5) establishes minimum viable understanding without triggering context overload. Adaptive branching then explores only the threads that the anchor answers reveal as relevant. This is progressive disclosure applied to conversational workflows — show only what is needed to decide and act at each step. [Source: [Agentic Workflows for Conversational Human-AI Interaction Design — arxiv 2501.18002](https://arxiv.org/html/2501.18002v1)] [Source: [Progressive Disclosure Matters: Applying 90s UX Wisdom to 2026 AI Agents — AI Positive](https://aipositive.substack.com/p/progressive-disclosure-matters)] [Connects to: all WKFL discovery requirements — layer discovery architecture]

- **FP-5: The orchestrator owns context decisions; agents do not self-select.** This principle from Phase 2 domain research (Truth 3) applies with particular force to workflows. The workflow is the orchestrator. It determines what context gets injected into each agent and when. The four framing workflows (debug, new, enhance, refactor) must each decide, based on the discovery brief, what framing-specific context to add to research agents, the planner, executors, and reviewers. Agents cannot self-select framing context they were not given — if the research agent doesn't receive lens metadata, it cannot adjust its investigation posture. The workflow is the intelligence that propagates framing through the entire pipeline. [Source: Phase 2 Domain Truth — `.planning/phases/02-agent-framework/research/DOMAIN-TRUTH.md`, Truth 3] [Connects to: WKFL-07 — framing context injection is a workflow responsibility]

### Universal Constraints

- **UC-1: Slash commands must stay orchestration-only; execution must go to subagents.** Official best practice and community research are unambiguous: slash commands that attempt to do token-intensive work in the main conversation thread cause "context pollution" — noisy, massive outputs that degrade reasoning for subsequent turns. The correct pattern: slash command = orchestration and user interaction; subagents = all token-intensive investigation and generation work. For GSD workflows, this means the `/debug`, `/new`, `/enhance`, `/refactor`, and `/init` commands must keep discovery interaction in the main thread (where user response is needed) and delegate all file reading, codebase analysis, and artifact generation to subagents. The distinction is not about complexity — it's about signal preservation. [Source: [Slash Commands vs Subagents: How to Keep AI Tools Focused — Jason Liu](https://jxnl.co/writing/2025/08/29/context-engineering-slash-commands-subagents/)] [Connects to: WKFL-01 through WKFL-05, INIT-01, INIT-02 — all workflows must enforce this boundary]

- **UC-2: Multi-turn discovery requires explicit dialogue state management; implicit state is unreliable.** LLM-based conversational agents lack reliable multi-turn disambiguation when conversation state is managed implicitly. When faced with ambiguity, they tend to overhedge or guess rather than ask clarifying questions. Explicit state tracking — knowing which MVU slots are filled, which are empty, and what the current discovery phase is — is the mechanism that prevents premature convergence. The "minimum viable understanding" completion conditions (e.g., for `/debug`: symptom, reproduction path, at least one falsifiable hypothesis) must be tracked explicitly, not inferred from conversational feel. [Source: [Learning to clarify: Multi-turn conversations with Action-Based Contrastive Self-Training — Google Research](https://research.google/blog/learning-to-clarify-multi-turn-conversations-with-action-based-contrastive-self-training/)] [Connects to: WKFL-02 through WKFL-05 — each lens has explicit MVU completion conditions]

- **UC-3: The goal formulation stage prevents trial-and-error prompting.** Research confirms: "the goal formulation stage can stimulate reflection before engaging in trial-and-error prompting." Without an explicit discovery phase, users prompt iteratively, each iteration refining the direction — but each failed iteration costs tokens and erodes context quality. The framing discovery step is the mechanism that front-loads reflection before the expensive pipeline starts. This is not about slowing users down — it's about preventing the compounding cost of misdirection that gets discovered only after execution. [Source: [Agentic Workflows for Conversational Human-AI Interaction Design — arxiv 2501.18002](https://arxiv.org/html/2501.18002v1)] [Connects to: WKFL-02 through WKFL-05 — each lens discovery phase replaces trial-and-error]

- **UC-4: Read operations parallelizable; write operations must be single-threaded.** This is a hard architectural constraint for the initialization flows. The existing-project `/init` scan phase (automated scan of structure, tech stack, entry points, etc.) involves parallel read operations — safe to parallelize. Writing PROJECT.md, updating STATE.md, writing capability files — these must be sequential to prevent merge conflicts and broken state. The three-phase `/init` existing-project flow (scan → validate → gap fill) already respects this constraint, but the planning of each phase must enforce it explicitly. [Source: [Slash Commands vs Subagents: How to Keep AI Tools Focused — Jason Liu](https://jxnl.co/writing/2025/08/29/context-engineering-slash-commands-subagents/)] [Connects to: INIT-01, INIT-02 — initialization modes must respect read/write ordering]

- **UC-5: Framing misclassification has non-zero probability and must be handled by design, not by hope.** No upfront classification mechanism achieves 100% accuracy. A user typing `/enhance` when the right framing is `/refactor` is a real scenario. If the workflow has no misclassification detection and no pivot mechanism, the entire pipeline runs under the wrong frame and produces a plan with wrong risk posture, wrong decomposition strategy, and wrong review targets. The mid-discovery cross-framing detection requirement is not a nice-to-have — it is a quality gate that prevents the cost of a full pipeline run under the wrong frame. The cost of detecting misclassification during discovery (one conversation turn) is trivially lower than the cost of detecting it at review (after execution). [First principles: Any classification system with less than 100% accuracy will occasionally misclassify. The cost of misclassification compounds at each subsequent pipeline stage. Detection must happen as early as possible — during discovery, before the pipeline commits to a frame.] [Connects to: 06-CONTEXT "lens misclassification" decisions]

- **UC-6: Auto-detection of project state (new vs. existing) requires a concrete decision boundary.** The `/init` command must make a binary decision based on filesystem evidence. Ambiguous cases — some files exist but not `.planning/` — require exactly one clarifying question, not a free-form conversation. This maps to established UX patterns for onboarding wizards: "when you can't determine how to route based on initial input, ask a targeted clarification." The decision boundary must be defined in terms of observable filesystem state (`.planning/` exists, code exists, both, neither) so the workflow can branch deterministically. [Source: [Conversation Builder — Routing AI Agents Conversationally — LivePerson](https://developers.liveperson.com/conversation-builder-generative-ai-routing-ai-agents-route-consumers-conversationally.html)] [Connects to: INIT-01, INIT-02 — single command, auto-detects mode]

### Validated Assumptions

- **VA-1: The "always summarize before transitioning" principle prevents misunderstanding compounding.** The Decision 06-CONTEXT to always summarize before transitioning from discovery to pipeline is validated by established UX patterns: playback prompts "wait, I forgot to mention..." and catches misunderstandings before they get encoded into requirements. In conversational AI, this is particularly important because the AI may have inferred things the user didn't intend — the summary externalizes the AI's internal model and gives the user a chance to correct it. Skipping this step is the primary cause of "built exactly what was specified but doesn't solve the problem" outcomes. Evidence: this pattern maps directly to the "discovery brief" structure where the one-sentence problem statement must be validated before proceeding. [Source: [AI UX Patterns — aiuxpatterns.com](https://www.aiuxpatterns.com/)] [First principles: the summary-before-transition is an evidence gate. The pipeline cannot start until the AI's understanding has been externalized and validated.] [Connects to: 06-CONTEXT decisions — always summarize before transitioning]

- **VA-2: Different work types require structurally different discovery flows, not just different question wording.** The four lenses have directional metaphors that reflect genuinely different reasoning patterns — debug works backward (symptom → cause), new works forward (problem → shape), enhance works outward (existing → extension), refactor works underneath (surface → structure). These are not cosmetic differences. A debug discovery that explores "what should it look like?" is wasting turns on the wrong question. A new feature discovery that jumps to "what's breaking?" is solving the wrong problem. The lens-specific MVU completion conditions (different slots, different thresholds) reflect the structural difference in what "understood enough to proceed" means per framing. Evidence: industry practice in AI coding workflows increasingly distinguishes task types: "extended thinking for refactors, visual debugging for UI bugs, specification-first for new features." [Source: [My LLM coding workflow going into 2026 — Addy Osmani](https://addyosmani.com/blog/ai-coding-workflow/)] [Source: [AI Coding Workflow — MachineLearningMastery.com](https://machinelearningmastery.com/the-roadmap-for-mastering-ai-assisted-coding-in-2025/)] [Connects to: WKFL-02, WKFL-03, WKFL-04, WKFL-05]

- **VA-3: Progressive onboarding — triggering context introduction based on user behavior rather than upfront dump — is validated as the correct init pattern.** The `/init` existing-project flow (automated scan → user validation → gap fill) implements progressive onboarding: the system first generates its understanding, then presents for validation, then fills specific gaps. This is documented best practice: "progressive onboarding introduces features and information gradually, timed to when they are most relevant, triggered by user behavior." The alternative — asking all questions upfront before scanning — produces worse outcomes because users can't accurately answer questions about their codebase without seeing the system's automated understanding first. [Source: [Advanced Guide to Progressive Onboarding in UX — UserGuiding](https://userguiding.com/blog/progressive-onboarding)] [Connects to: INIT-02 — existing-project three-phase flow]

- **VA-4: Independent validation sections in init accelerate confirmation.** The 06-CONTEXT decision that "init validation sections are independent — confirming stack doesn't block confirming architecture" is validated by a simple principle: dependencies between validation items create bottlenecks and can cascade failures. If confirming the tech stack requires the same user session as confirming the architecture, any uncertainty about the stack delays architecture confirmation. Designing sections as independent units allows users to confirm what they're certain about immediately, flag uncertainty where it exists, and complete the validation at their own pace without artificial sequencing. [First principles: serial dependencies in a human-in-the-loop process create bottlenecks. Independent units parallelize confirmation without parallelizing the conversation.] [Connects to: INIT-02 — parallel validation sections]

### Domain Risks

- **DR-1: The discovery summarize-and-confirm step will be skipped under time pressure, which is exactly when it's most needed.** The strongest temptation in any workflow is to shortcut the reflection step when the user is eager to start execution. But users who provide rich upfront context are precisely the ones most likely to have made assumptions that need surfacing — they've thought about it so much that their mental model has drifted from the problem's actual constraints. Mitigation: the summarize-before-transition step must not be optional or user-controllable in the workflow. It is a hard gate, not a soft recommendation. The only permissible fast-path is the explicit skip/fast-track where the user self-certifies their context is complete, in which case the brief is still generated (just auto-populated rather than conversation-generated). [First principles: the value of a gate is eliminated when it can be bypassed by the parties it governs. The discovery summary gate must be system-enforced, not user-enforced.] [Connects to: WKFL-02 through WKFL-05 — "three exit signals" for discovery completion]

- **DR-2: Fuzzy resolution of capability/feature references introduces ambiguity at the command entry point.** All 11 commands accept natural language references (`/debug the drill timing thing`). This is correct UX. But fuzzy resolution carries a failure mode: multiple-match and no-match cases must be handled explicitly. If fuzzy resolution silently selects the wrong target, the entire pipeline runs on the wrong context. Mitigation: the 06-CONTEXT decisions already specify: multiple matches → present top 3, user picks; no match → ask to clarify or create new. This must be implemented as a hard check in every framing command, before discovery begins. A wrong target discovered at plan time (after research, discovery, and requirements generation) is catastrophically expensive to correct. [First principles: the cost of a routing error compounds at each pipeline stage. Fuzzy resolution errors must be detected before any pipeline computation begins.] [Connects to: WKFL-01 — fuzzy resolution for all framing commands]

- **DR-3: The compound work case (primary + secondary lens) is significantly harder to implement than single-lens workflows.** The "this is primarily `/enhance` with a `/refactor` component" use case requires the workflow to simultaneously maintain two framing contexts: one for the primary lens (which drives the pipeline) and one for the secondary (which informs it). This creates several implementation risks: (a) which lens governs plan risk posture when they conflict? (b) does the review receive both sets of framing metadata? (c) how does the Discovery Brief represent compound context? The 06-CONTEXT decision to have a single run with layered context is correct architecturally, but the implementation must define these ambiguities explicitly before building. Risk: compound work that appears to work during testing but silently loses the secondary lens context mid-pipeline. [First principles: two simultaneous frames create potential contradictions. The system must have an explicit conflict resolution rule for every point where they can diverge.] [Connects to: 06-CONTEXT compound work decisions]

- **DR-4: Context window budget for the discovery conversation must account for the Discovery Brief and what follows it.** The discovery conversation itself consumes tokens. The Discovery Brief — a structured artifact summarizing discovery — consumes tokens. After the brief, the workflow must still orchestrate the research phase (spawning agents, reading their results), generate requirements, and so on. If the discovery conversation is allowed to run long (deeply exploratory topics, compound work, multiple misclassification pivots), the remaining context budget for pipeline orchestration shrinks. Mitigation: the "diminishing returns" exit signal (circling detected) is not just a quality signal — it is a budget management mechanism. The workflow must monitor discovery length and trigger the diminishing-returns exit before the orchestration context becomes critically constrained. [Source: Phase 2 Domain Truth, Truth 2 — context windows are finite and competitive] [Connects to: WKFL-02 through WKFL-05 — three exit signals including diminishing returns]

- **DR-5: The escalation/backward-flow protocol creates a pipeline that can loop without explicit safeguards.** The universal escalation protocol allows any stage to push back upstream: Minor → continue, Moderate → pause, Major → recommend returning to a specific stage. The risk is a pipeline that loops: discovery → requirements → plan → plan detects major issue → returns to discovery → requirements (again) → plan (again) → finds major issue again. Without a loop detection mechanism, this can consume unbounded resources. The 06-CONTEXT decision addresses this for the planning phase (hard stop at 2 rounds) but does not explicitly address cross-stage backward flow loops. Mitigation: Major escalations must count against a global pipeline-run escalation budget. A pipeline that has returned to discovery once should surface this to the user with explicit framing ("we've already reset to discovery once — here is what changed"). [First principles: any system with backward flow edges is a potential loop. Loops in a token-consuming system require explicit termination conditions.] [Connects to: 06-CONTEXT universal escalation protocol]
