## Tech Constraints Findings

### Hard Constraints

- **`scripts/build-hooks.js` does not exist** — `/Users/philliphall/get-shit-done-pe/package.json:42` references `"build:hooks": "node scripts/build-hooks.js"`. The file is absent from the repo (`Glob` confirms: `scripts/` contains only `run-tests.cjs` and `validate-install.js`). `prepublishOnly` runs this script, so `npm publish` will fail as-is. The hooks build pipeline is broken at the source. — [Source: `/Users/philliphall/get-shit-done-pe/package.json:42`, Glob of `/Users/philliphall/get-shit-done-pe/scripts/`]

- **`hooks/dist/` is the published artifact, `hooks/` is the source — and they are inconsistent** — `package.json` `files` array includes `"hooks/dist"` but not `"hooks/"`. `install.js` copies from `hooks/` (source). When the package is published to npm and a user runs `npm install -g get-shit-done-pe`, `hooks/` will not exist in the installed package tree — `install.js` line 563–579 (`const hookFiles = [...]`) will find nothing to copy. Result: silent hook install failure. — [Source: `/Users/philliphall/get-shit-done-pe/package.json:8–15`, `/Users/philliphall/get-shit-done-pe/bin/install.js:563–579`]

- **`validate-install.js` calls `process.exit()` at top level — cannot be `require()`'d** — Lines 358–368 call `process.exit(0)` or `process.exit(1)` unconditionally. Any `require('./validate-install')` from within `install.js` terminates the parent process. The `install-feedback` FN-02 requirement ("run validation programmatically") cannot be satisfied by direct `require`. — [Source: `/Users/philliphall/get-shit-done-pe/scripts/validate-install.js:358–368`]

- **npx caches packages and does NOT always fetch `@latest` from the registry** — Multiple confirmed npm CLI bugs (issues #4108, #6179, #7838) document that `npx get-shit-done-pe@latest` may execute a cached older version rather than re-fetching from npm. The `auto-latest` design depends on `npx get-shit-done-pe@latest --global` fetching the new version — this is unreliable without `--prefer-online` or equivalent. — [Source: [npm/cli #4108](https://github.com/npm/cli/issues/4108), [npm/cli #6179](https://github.com/npm/cli/issues/6179)]

- **The `--global` flag is NOT recognized by npx as a npx-level flag** — `npx` recognizes its own flags (`--package`, `--call`, `--yes`, etc.) before the package name; `--global` is not one of them. When a user runs `npx get-shit-done-pe --global`, the `--global` argument is passed through to the bin script (`bin/install.js`) as `process.argv[2]`, which is the correct and intended behavior for this package. This is not a blocker, but it means the BRIEF.md's `npx get-shit-done-pe --global` syntax works by convention of the bin script, not npx semantics. — [Source: [npm npx docs](https://docs.npmjs.com/cli/v11/commands/npx/), `/Users/philliphall/get-shit-done-pe/bin/install.js:20`]

- **Claude Code hook snapshot semantics: hooks do not take effect mid-session** — Claude Code captures hook configuration at session startup. Changes to `settings.json` hooks mid-session require a new session to activate. This is a hard constraint for auto-latest: a background install during SessionStart will not affect the current session — new hooks and files are only active in the next session. — [Source: [Claude Code hooks reference](https://code.claude.com/docs/en/hooks) — "Claude Code captures a snapshot of hooks at startup and uses it throughout the session."]

- **Node.js version gate: `>=16.7.0` per `package.json`** — `fs.rmSync` (used in `install.js` at line 169) was added in Node 14.14.0; `fs.mkdirSync` with `recursive` option in Node 10.12.0. No constraint from the stdlib usage — the stated `>=16.7.0` engine requirement covers all APIs in use. — [Source: `/Users/philliphall/get-shit-done-pe/package.json:34`, Node.js docs]

- **`settings.json` `hooks.SessionStart` array is initialized but currently empty** — `install.js` lines 630–631 initialize the array and write it to `settings.json`, but push nothing to it. There is no idempotency guard for SessionStart (unlike `PostToolUse` which has `hasContextMonitorHook` check). The `auto-latest` hook registration must add its own idempotency guard to avoid duplicate entries on re-install. — [Source: `/Users/philliphall/get-shit-done-pe/bin/install.js:630–631`]

- **Zero runtime dependencies constraint** — `PROJECT.md` states "zero runtime deps — Node.js stdlib only." Any new code in `install.js` or the auto-latest hook must use only Node stdlib (`fs`, `path`, `os`, `https`, `child_process`). No `node-fetch`, `axios`, `semver`, or other packages. — [Source: `/Users/philliphall/get-shit-done-pe/.planning/PROJECT.md:81`]

### Dependency Capabilities

- **Node.js `child_process.spawn` with `detached: true` + `child.unref()`** — Enables fully non-blocking background process that survives parent exit. The existing `gsd-check-update.js` uses this exact pattern: `spawn(process.execPath, ['-e', ...], { stdio: 'ignore', detached: true })` then `child.unref()`. This is the correct mechanism for auto-latest's background npm install. `execSync` is not suitable (blocks the event loop). — [Source: `/Users/philliphall/.claude/hooks/gsd-check-update.js:25–62`, [Node.js child_process docs](https://nodejs.org/api/child_process.html)]

- **npm registry API: `GET https://registry.npmjs.org/<package>?fields=dist-tags`** — Returns a compact response (~100 bytes) with just `{"dist-tags":{"latest":"x.x.x"}}`. Preferred over `npm view get-shit-done-pe version` (spawns a child process) or the full packument (large). Can be called via Node.js `https.get()` with a timeout. No auth required for public packages. — [Source: [npm registry API docs](https://www.edoardoscibona.com/exploring-the-npm-registry-api), [tutorialpedia.org](https://www.tutorialpedia.org/blog/get-versions-from-npm-registry-api/)]

- **`npm view <package> version` via `execSync`** — Alternative to direct registry HTTP. Works within a spawned child process. The existing `gsd-check-update.js` uses this pattern with `timeout: 10000`. Risk: spawns another child process inside an already-background process; adds latency. Viable but `https.get()` is lighter. — [Source: `/Users/philliphall/.claude/hooks/gsd-check-update.js:45`]

- **Claude Code `SessionStart` hook** — Fires on session start, resume, `/clear`, and compaction. `source` field indicates which trigger. No matcher filtering beyond `startup`/`resume`/`clear`/`compact`. Exit code 2 shows stderr to user only (non-blocking). Stdout on exit 0 is added as context to Claude. `async: true` field available (Claude Code 2.1+) to run hook in background without blocking session startup. — [Source: [Claude Code hooks reference](https://code.claude.com/docs/en/hooks)]

- **Claude Code `async: true` hook field** — Available since Claude Code 2.1 (released January 25, 2026). When set, hook runs in background without blocking. Critical limitation: `async: true` hooks cannot use `decision` or `permissionDecision` fields — they are fire-and-forget. This matches the auto-latest requirement exactly (silent background update, no blocking). — [Source: [Claude Code async hooks post](https://blog.devgenius.io/claude-code-async-hooks-what-they-are-and-when-to-use-them-61b21cd71aad), [Claude Code hooks reference](https://code.claude.com/docs/en/hooks)]

- **Claude Code `PostToolUse` hook with `matcher`** — The `gsd-askuserquestion-guard.js` hook uses `matcher: 'AskUserQuestion'` to filter PostToolUse events. Exit code 2 on PostToolUse shows stderr to Claude (tool already ran — cannot block). This is confirmed working in the existing codebase. — [Source: `/Users/philliphall/get-shit-done-pe/bin/install.js:660–671`, [Claude Code hooks reference](https://code.claude.com/docs/en/hooks)]

- **`fs.writeFileSync` / `fs.readFileSync` on `~/.claude/`** — Standard Node.js stdlib. No special permissions required on macOS for files owned by the current user. Failures surface as `EACCES` (permissions) or `ENOENT` (parent dir missing). `install.js` uses `fs.mkdirSync(..., { recursive: true })` to handle missing parents — this pattern is already established. — [Source: `/Users/philliphall/get-shit-done-pe/bin/install.js:171`, [Node.js fs docs](https://nodejs.org/api/fs.html)]

- **`npm list -g --depth=0 --json`** — Detects globally installed packages programmatically via `execSync`. JSON output parseable to check for `get-shit-done-cc` presence. Alternative to spawning `npx get-shit-done-cc --uninstall` for cc detection. — [Source: [GeeksForGeeks npm list guide](https://www.geeksforgeeks.org/node-js/how-to-get-a-list-of-globally-installed-npm-packages-in-npm/)]

- **esbuild `^0.24.0` (devDependency)** — Present in `package.json`. Was used by the now-deleted `scripts/build-hooks.js` to compile hook source files to `hooks/dist/`. Cannot be used in production code (devDep only). The hooks source files (`gsd-context-monitor.js`, `gsd-statusline.js`, `gsd-askuserquestion-guard.js`) are plain CommonJS with no transpilation needs — esbuild was likely used for bundling/minification, not for syntax transformation. — [Source: `/Users/philliphall/get-shit-done-pe/package.json:38`]

### Compatibility Issues

- **`package.json` `files` array ships `hooks/dist/`, but `install.js` reads from `hooks/` (source directory)** — When published to npm, the installed package will contain `hooks/dist/` but not `hooks/`. `install.js` at line 563 hardcodes source paths: `const hooksSrc = path.join(src, 'hooks')`. This will silently find no files post-npm-publish. Resolution: either change `files` to include `hooks/` directly and remove the `hooks/dist` entry, OR restore the build step so hooks are compiled to `hooks/dist/` and update `install.js` to read from `hooks/dist/`. — [Source: `/Users/philliphall/get-shit-done-pe/package.json:8–15`, `/Users/philliphall/get-shit-done-pe/bin/install.js:563`]

- **Live `~/.claude/settings.json` has `gsd-check-update.js` as a SessionStart hook** — This file exists in the live installed `~/.claude/hooks/` but has been removed from the repo source. The `cc-replacement` feature's cleanup logic must strip `gsd-check-update` from `SessionStart` entries in settings.json. Current `uninstall()` only strips hooks matching `gsd-statusline` from SessionStart (lines 368–385) — it will miss `gsd-check-update`. — [Source: `/Users/philliphall/.claude/settings.json:100–109`, `/Users/philliphall/get-shit-done-pe/bin/install.js:368–385`]

- **`uninstall()` strips only `gsd-statusline` from PostToolUse, missing `gsd-context-monitor` and `gsd-askuserquestion-guard`** — Lines 388–409 filter PostToolUse by `gsd-context-monitor` and `gsd-askuserquestion-guard`, which is correct. But lines 368–385 (SessionStart cleanup) only check for `gsd-statusline` in SessionStart. The `gsd-check-update` pattern in SessionStart is missed. — [Source: `/Users/philliphall/get-shit-done-pe/bin/install.js:368–409`]

- **`validate-install.js` expected hooks list misses `gsd-askuserquestion-guard.js`** — Lines 178–186 check for only `gsd-context-monitor.js` and `gsd-statusline.js`. The guard hook (added in the AskUserQuestion fix) is installed but not validated. Any programmatic invocation of the validation step will report pass even if the guard hook is absent. — [Source: `/Users/philliphall/get-shit-done-pe/scripts/validate-install.js:178–186`]

- **`package.json` `bin` key currently `"get-shit-done-cc"` — npm registers this as the global binary name** — On `npm install -g get-shit-done-cc`, npm creates a symlink named `get-shit-done-cc` in the global bin directory. Changing the `bin` key to `"get-shit-done-pe"` after publish means the new binary name will be `get-shit-done-pe`. The old `get-shit-done-cc` binary will remain until the old package is uninstalled. — [Source: `/Users/philliphall/get-shit-done-pe/package.json:5–7`, [npm bin field docs](https://docs.npmjs.com/cli/v11/configuring-npm/package-json#bin)]

- **npx `@latest` cache stale bug** — When auto-latest runs `npx get-shit-done-pe@latest --global` inside a background spawn, it may execute an older cached version. This makes the `@latest` self-update mechanism unreliable. The more reliable path is `npm install -g get-shit-done-pe@latest` (explicit registry fetch via npm install). — [Source: [npm/cli #4108](https://github.com/npm/cli/issues/4108), [npm/cli #6179](https://github.com/npm/cli/issues/6179)]

### Alternatives

- **Blocked: `require()` validate-install.js** → **Alternative: `execSync('node scripts/validate-install.js', { stdio: 'pipe' })`** catches exit code and stdout without terminating the parent, OR refactor validate-install.js to export a function with `if (require.main === module)` guard. — [First principles: CommonJS module pattern; `execSync` stdio capture is established in the same file at line 262]

- **Constrained: `npx get-shit-done-pe@latest --global` for auto-update** → **Alternative: `npm install -g get-shit-done-pe@latest`** explicitly forces registry fetch, bypasses npx cache issues, is the canonical way to update global packages. — [Source: [npm install docs](https://docs.npmjs.com/cli/install), [npm/cli #4108](https://github.com/npm/cli/issues/4108)]

- **Constrained: running upstream uninstall from `$HOME`** → **Alternative: `npm uninstall -g get-shit-done-cc`** directly removes the package without invoking upstream's buggy detection logic. Then the post-uninstall remnant scan handles any lingering `~/.claude/` files. — [Source: `cc-replacement/FEATURE.md:FN-01` known bug note, [npm uninstall docs](https://docs.npmjs.com/cli/v11/commands/npm-uninstall)]

- **Blocked: `hooks/dist/` build pipeline (scripts/build-hooks.js deleted)** → **Alternative: publish `hooks/` source directly** by updating `package.json` `files` array from `"hooks/dist"` to `"hooks"`. No esbuild, no build step, no missing script. Hook source files are already plain CommonJS. — [First principles: YAGNI — build step adds complexity without benefit for plain JS files]

- **Gap: stale `gsd-check-update.js` in live `settings.json` SessionStart** → **Alternative: extend `cleanupOrphanedHooks()` pattern** with `gsd-check-update` added to `orphanedHookPatterns` array. The function already handles general orphaned hook cleanup — this is one more entry. — [Source: `/Users/philliphall/get-shit-done-pe/bin/install.js:214–261`]
