### First Principles

- **`npx <package>` does NOT guarantee fetching the latest version.** Since npm v7 rewrote npx, cached packages are reused on subsequent invocations even when newer versions are published. Only `npx <package>@latest` forces a registry check and fetch. This is a documented, long-standing bug with multiple open GitHub issues and no resolution. The design implication: if the target experience is "always get newest version," the install command must explicitly use `@latest` in user-facing documentation and the auto-update hook must call `npx get-shit-done-pe@latest`, not bare `npx get-shit-done-pe`. — [First principles: npm resolves packages through a cache-first lookup chain: local node_modules → global install → npx cache → registry fetch. Each layer short-circuits the next. Without `@latest`, npm has no obligation to check the registry.] + [https://github.com/npm/cli/issues/2329]

- **Idempotency requires explicit state comparison, not just "don't error on re-run."** A truly idempotent install must produce the same filesystem state whether run once or ten times. This means: (1) file copy operations must overwrite rather than skip if the destination already exists, (2) hook registration must deduplicate entries in settings.json before writing rather than appending, (3) token replacement must be safe to apply to already-replaced content (i.e., the replacement must be a no-op if `{GSD_ROOT}` is not present). Idempotency that only avoids crashes — but leaves stale state or duplicated entries — violates the invariant. — [First principles: Idempotency = f(f(x)) == f(x). For install scripts, this means the target state after N runs is identical to the target state after 1 run. The only way to guarantee this is to always write the desired final state, not to conditionally add to existing state.]

- **`postinstall` scripts carry a systemic trust problem that makes them inappropriate for user-facing install logic.** Security literature documents that `postinstall` runs automatically on `npm install` for any dependent in the tree, making it a well-known attack vector. As a result, security-conscious users and organizations configure `--ignore-scripts` globally, which silently breaks postinstall-based install logic with no error surfaced to the user. GSD's model — using the `bin` field and running install.js explicitly via `npx` — is architecturally correct precisely because it requires explicit user invocation. This constraint validates the current approach and rules out moving install logic into lifecycle hooks. — [https://www.nodejs-security.com/blog/npm-ignore-scripts-best-practices-as-security-mitigation-for-malicious-packages] + [https://cheatsheetseries.owasp.org/cheatsheets/NPM_Security_Cheat_Sheet.html]

### Universal Constraints

- **The `bin` field key in package.json defines the exact CLI command name that npm exposes globally.** When `npm install -g` runs, npm symlinks the file at `bin["get-shit-done-pe"]` into the global bin directory (e.g., `/usr/local/bin/get-shit-done-pe`). If the key is `get-shit-done-cc` and the file is `bin/install.js`, the command exposed is `get-shit-done-cc`. Renaming the package identity requires renaming this key — not just the `name` field. The `name` field controls npm registry identity; the `bin` key controls the shell command name. These are independent fields and both must change. — [https://2ality.com/2022/08/installing-nodejs-bin-scripts.html] + `/Users/philliphall/get-shit-done-pe/package.json` (current: `"get-shit-done-cc": "bin/install.js"`)

- **A Claude Code `SessionStart` hook exists and supports matchers for `startup`, `resume`, `clear`, `compact`.** This is confirmed by the official hook reference documentation. The hook fires when a session begins or resumes, making it a valid trigger for a version-check-and-update behavior. However, the hook is synchronous with the session start sequence — a blocking operation in `SessionStart` delays the session. The auto-update design (non-blocking background subprocess) must account for this: the hook script must spawn its version-check process in the background and exit immediately, not wait for npm to finish. — [https://code.claude.com/docs/en/hooks] (hook lifecycle table, SessionStart row)

- **The npm registry exposes a lightweight JSON API for fetching dist-tags without downloading the full package.** `https://registry.npmjs.org/<package>?fields=dist-tags` returns only the tag-to-version mapping (e.g., `{"latest": "2.1.0"}`), avoiding the multi-MB full metadata response. This is the correct implementation pattern for version-check logic in the auto-update hook — no npm subprocess required, just a single `https.get()` call with a short timeout. — [https://www.tutorialpedia.org/blog/get-versions-from-npm-registry-api/] + [https://github.com/npm/registry/blob/main/docs/REGISTRY-API.md]

- **npm does not enforce mutual exclusion between separately-named packages that manage the same files.** Two packages (`get-shit-done-cc` and `get-shit-done-pe`) can coexist in npm's global package list simultaneously, each owning different copies of files, potentially with conflicting hook registrations. npm has no concept of "this package supersedes that one." Mutual exclusion must be enforced by the installer itself — detecting the presence of the competing package, uninstalling it, and verifying the result. This is an application-level responsibility, not a platform guarantee. — [First principles: npm's global install namespace tracks packages by name. Two differently-named packages are orthogonal entries. No dependency or conflict relationship exists unless declared explicitly in package.json `peerDependencies`/`conflicts`, which npm does not enforce for global installs.]

- **`settings.json` in `~/.claude/` is shared global state across all hook registrations.** Any installer that overwrites the entire file destroys registrations made by other tools or the user's own configuration. The correct operation is always an additive merge: read the existing file, merge the new hook entries, write back. This constraint is universal regardless of which package owns the installer. — [First principles: A file with multiple writers requires read-modify-write semantics to avoid clobbering. Overwrite semantics are only safe when the writer is the sole owner of the file, which is not the case for `~/.claude/settings.json`.]

### Validated Assumptions

- **The BRIEF.md assumption that `npx get-shit-done-pe` "always gets newest version" is architecturally incorrect without `@latest`.** The feature spec (BRIEF.md: "npx get-shit-done-pe = always gets newest version from npm, deploys to ~/.claude") and auto-latest FEATURE.md both describe this as the desired user experience, but npx's actual behavior caches after first use. The implementation must either (a) document the install command as `npx get-shit-done-pe@latest`, or (b) accept that `npx get-shit-done-pe` only fetches fresh on first use and subsequent freshness is handled by the auto-update hook. Both are workable — the assumption just needs to be made explicit. — [https://github.com/npm/cli/issues/2329] + [https://github.com/npm/rfcs/issues/700]

- **The upstream uninstall approach (running `npx get-shit-done-cc@latest --claude --global --uninstall`) is unreliable by known bug.** The cc-replacement FEATURE.md explicitly calls this out: "Known bug: upstream misdetects global/local when run from $HOME." This is a validated real constraint — the upstream installer's cwd-detection logic is broken when invoked from a home directory context. The design correctly compensates with a post-uninstall remnant verification pass against safe-to-remove patterns (`gsd:*`, `gsd-*`). The remnant-scan fallback is the reliable path; the upstream uninstall is best-effort. — `/Users/philliphall/get-shit-done-pe/.planning/capabilities/install-and-deploy/features/cc-replacement/FEATURE.md` (FN-01 decision section)

- **`validate-install.js` currently produces per-check verbose output (PASS/FAIL per line) that conflicts with the install-feedback design goal of "silent install, clear pass/fail at end."** The existing script at `/Users/philliphall/get-shit-done-pe/scripts/validate-install.js` uses `console.log` calls on every check (lines 41, 48-50). The install-feedback feature requires that validation result be folded into the overall pass/fail, not printed independently. The validate-install.js script must be callable programmatically — returning a structured result rather than printing to stdout — or its output must be captured and suppressed by the caller. TC-01 identifies this correctly: "validate-install.js must be callable programmatically." — `/Users/philliphall/get-shit-done-pe/scripts/validate-install.js` (lines 38-51, 354-368)

### Domain Risks

- **The `@latest` resolution gap creates a false sense of "always fresh" for users who already have the package cached.** If a user runs `npx get-shit-done-pe` without `@latest`, they get whatever is in the npx cache — potentially an old version — with no indication that a newer version exists. Since the auto-update hook handles ongoing freshness, the initial install command documented in the README is the critical gap. If the README shows `npx get-shit-done-pe` (no `@latest`), first-time users who installed the cached version will never see the update mechanism kick in for stale initial installs. Mitigation: document the install command as `npx get-shit-done-pe@latest` and use the same in the auto-update hook. — [https://github.com/npm/cli/issues/5262] + [https://github.com/npm/cli/issues/4108]

- **A blocking SessionStart hook for auto-update will visibly delay every Claude Code session start.** The auto-latest feature calls for a hook that fires on session start to check for updates. If the hook waits for the npm registry response (even with a 5-second timeout), every session start has a minimum latency of 0–5 seconds. This is acceptable only if the hook exits immediately after spawning a background process. The correct Node.js pattern is `child_process.spawn(..., { detached: true, stdio: 'ignore' }).unref()` — which spawns the process and immediately releases the event loop. Any implementation that uses `execSync` or `await` on the version check will block the session. — [First principles: Claude Code hooks are synchronous in the session lifecycle. A hook that runs a blocking network call adds that call's latency to every session start. Detached child processes are the only way to get "fire and forget" behavior in Node.js CLI contexts.] + [https://code.claude.com/docs/en/hooks] (default timeout: 600s for command hooks)

- **CLAUDE.md delimiter management creates an irreversible state if the user has existing undelimited content from a previous install.** The cc-replacement feature introduces `<!-- GSD-PE:START -->` / `<!-- GSD-PE:END -->` delimiters, but CLAUDE.md content written by the upstream `get-shit-done-cc` installer may have no delimiters. The design correctly calls for "warn user, don't touch" when delimiters are absent. However, the practical consequence is: the pe install appends new delimited content while old undelimited cc content remains, producing duplicate GSD instructions. Users must manually remove the old cc block. This is documented in the feature as an accepted tradeoff (only one current user), but is a real risk if user count grows. — `/Users/philliphall/get-shit-done-pe/.planning/capabilities/install-and-deploy/features/cc-replacement/FEATURE.md` (FN-03)

- **The `prepublishOnly: npm run build:hooks` step is a mandatory pre-publish gate that must not be skipped.** The `hooks/dist/` directory (containing esbuild-bundled hook files) is the published artifact — not the source files in `hooks/`. If `build:hooks` does not run before `npm publish`, the published package contains either stale compiled output or no `hooks/dist/` at all. This is a procedural risk for the package-identity feature, which involves a publish action. The build step is already wired in `package.json:scripts.prepublishOnly` — confirming it runs automatically on `npm publish` — but the author must not use `npm publish --ignore-scripts` or bypass the hook. — `/Users/philliphall/get-shit-done-pe/package.json` (line 43: `"prepublishOnly": "npm run build:hooks"`)
