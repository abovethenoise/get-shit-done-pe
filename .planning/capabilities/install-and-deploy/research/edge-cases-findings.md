### Boundary Conditions

- **`settings.json` hooks array at exactly 0 entries after filter**: install.js deletes the `SessionStart` key when the filtered array is empty (line 384: `delete settings.hooks.SessionStart`) ‚Äî correct. But it does NOT delete the `hooks` object itself when SessionStart and PostToolUse are both empty, unless both are empty simultaneously. If only one is emptied, the `hooks` key persists with a partial structure. ‚Äî `/Users/philliphall/get-shit-done-pe/bin/install.js:412-415` (checks `Object.keys(settings.hooks).length === 0` only in uninstall path, not in install path)

- **`~/.claude/` is a symlink**: `fs.existsSync` follows symlinks and returns true, but `fs.rmSync(destDir, {recursive: true})` on a symlink target would delete the link destination ‚Äî potentially catastrophic if the user has `~/.claude` symlinked to a shared or versioned directory ‚Äî [First principles: Node.js fs.rmSync behavior with symlinks]

- **Token replacement regex `/\{GSD_ROOT\}\//g` ‚Äî extra trailing slash**: the regex matches `{GSD_ROOT}/` (with slash). Any reference written as `{GSD_ROOT}` without the trailing slash (e.g., in a JSON value) would be missed and remain unresolved. `validateNoUnresolvedTokens` checks for `{GSD_ROOT}` (without slash) so it would catch this ‚Äî but the replacement would silently not apply. ‚Äî `/Users/philliphall/get-shit-done-pe/bin/install.js:184-185` vs `:469`

- **Auto-update cache timestamp: exactly 24 hours boundary**: `< 24` vs `<= 24` off-by-one ‚Äî if `lastCheck` is exactly 24 hours old (difference = 86400 seconds), the feature spec says "if < 24 hours: skip" ‚Äî so exactly 24h triggers a check. Acceptable, but worth confirming the implementation uses `>=` not `>`. ‚Äî `auto-latest/FEATURE.md:FN-03`; [First principles: boundary ambiguity on 24h comparison]

- **`process.stdin.isTTY` is undefined (not `false`)**: when stdin is a pipe (e.g., `npx get-shit-done-pe | tee log.txt`), `isTTY` is `undefined`, which is falsy ‚Äî falls through to the non-interactive global default path. Correct behavior, but the condition `!process.stdin.isTTY` would also trigger if Node.js has a bug where isTTY is undefined for a real TTY. ‚Äî `/Users/philliphall/get-shit-done-pe/bin/install.js:821` (`!process.stdin.isTTY`)

- **`hooks/dist` directory listed in `package.json` files but absent from repo**: `npm publish` includes only files matching the `files` array. If `hooks/dist` does not exist at publish time (no build step run), npm will silently exclude it without error ‚Äî resulting in a published package where the hooks never deploy correctly. ‚Äî `/Users/philliphall/get-shit-done-pe/package.json:13`; `ls hooks/dist` ‚Üí "does not exist"

- **CLAUDE.md file exceeding typical size**: if `~/.claude/CLAUDE.md` is very large (e.g., 500KB from extensive user notes), reading it entirely into memory for string manipulation is acceptable on modern systems ‚Äî not a practical concern, but worth noting the implementation uses `fs.readFileSync` (synchronous, full-file). ‚Äî [First principles: Node.js sync file read memory behavior]

- **cc-replacement: remnant scan uses glob patterns `gsd:*` and `gsd-*`**: if Claude Code ever adds a built-in command or agent with a `gsd-` prefix (unrelated to this package), the scan would incorrectly delete it. The safe-to-remove list in `FN-02` uses path-prefixed patterns ‚Äî safe only if those paths are exclusive to this package. ‚Äî `cc-replacement/FEATURE.md:FN-02`

### Integration Failure Scenarios

- **npm registry is unreachable** ‚Üí `auto-latest` FN-01 must silently skip (specified); if the http request hangs without a timeout, the Claude Code session start hook will block indefinitely ‚Äî session start blocks for minutes ‚Äî `auto-latest/FEATURE.md:TC-01` (timeout required but implementation detail unspecified)

- **`npx get-shit-done-cc@latest --uninstall` subprocess fails** ‚Üí `cc-replacement` FN-01 specifies "warn user, continue with pe's own cleanup." However, install.js currently has no subprocess call for upstream uninstall ‚Äî this logic does not yet exist. pe's remnant scan (FN-02) is the only cleanup path today. ‚Äî `cc-replacement/FEATURE.md` decisions block; `/Users/philliphall/get-shit-done-pe/bin/install.js` (no execSync calls for cc uninstall)

- **`validate-install.js` exits non-zero** ‚Üí currently exits the process with code 1; the install-feedback feature requires this be folded into the silent pass/fail model. If validation fails mid-silent-install, the current implementation prints directly to stdout (breaking "no intermediate noise" requirement). ‚Äî `install-feedback/FEATURE.md:FN-02`; `/Users/philliphall/get-shit-done-pe/scripts/validate-install.js:359-368`

- **Claude Code is running during install** ‚Üí hook files in `~/.claude/hooks/` are actively executed by Claude Code while install.js deletes and rewrites them ‚Äî a Claude Code tool execution during the window between `fs.unlinkSync` (delete old hook) and `fs.copyFileSync` (copy new hook) would find the hook missing and error or skip silently ‚Äî `/Users/philliphall/get-shit-done-pe/bin/install.js:562-579`; [First principles: non-atomic hook replacement]

- **`esbuild` devDependency absent at publish time** ‚Üí `prepublishOnly: npm run build:hooks` fails; npm publish is blocked ‚Äî but `scripts/build-hooks.js` does not exist, so this fails even earlier. The hooks in this repo are plain JS that do not require compilation. The `hooks/dist` entry in `files` is a vestige from upstream. ‚Äî `/Users/philliphall/get-shit-done-pe/package.json`; `ls /Users/philliphall/get-shit-done-pe/scripts/`

- **`auto-latest` update background process runs `npx get-shit-done-pe@latest --global`** ‚Üí this spawns install.js which is the same file that registered the hook ‚Äî a circular: each session start potentially triggers a full re-install of all hooks, which modifies `settings.json`, which is read by the next session. If the version check has a bug and always returns "newer version available," this results in install running every session start. ‚Äî `auto-latest/FEATURE.md:FN-01`; [First principles: version comparison logic not yet implemented]

### Known Issues in Ecosystem

- **npm/cli `execSync` stdout pipe crash**: calling `npx` via `child_process.execSync` with `{stdio: 'pipe'}` can cause npm/npx to hang or crash ‚Äî [GH npm/cli #2114](https://github.com/npm/cli/issues/2114); use `spawnSync` with individual stdio arrays instead

- **settings.json race condition in Claude Code**: concurrent Claude Code sessions produce truncated/invalid JSON in `settings.json` (or `.claude.json`) due to non-atomic writes ‚Äî reported 8+ times, fix shipped in Claude Code v2.1.61 but only for Claude Code's own writes, not for third-party installers ‚Äî [GH #28847](https://github.com/anthropics/claude-code/issues/28847), [GH #29217](https://github.com/anthropics/claude-code/issues/29217)

- **`prepublishOnly` does not run on `npm install -g`**: only runs on `npm publish` ‚Äî so if hooks needed compilation, installing from npm would get uncompiled source; mitigated here because hooks are plain JS. The `hooks/dist` entry in `package.json` `files` array is vestigial and will silently be excluded from the published package since the directory does not exist ‚Äî [npm scripts docs](https://docs.npmjs.com/cli/v11/using-npm/scripts/); [openclaw GH #11348](https://github.com/openclaw/openclaw/issues/11348)

- **`gsd-build/get-shit-done` GH #721**: upstream cc installer misdetects global vs local install when run from `$HOME` ‚Äî path collision between `./.claude/` and `~/.claude/` corrupts 193 file references ‚Äî confirmed active bug, must be mitigated in cc-replacement FN-01 by running subprocess from a non-home cwd ‚Äî [GH #721](https://github.com/gsd-build/get-shit-done/issues/721)

- **npm global EACCES on macOS/Linux**: `~/.claude/` created with root ownership after `sudo npm install` leaves subsequent user-level writes failing with EACCES ‚Äî common on systems where Node was installed via package manager ‚Äî [npm EACCES docs](https://docs.npmjs.com/resolving-eacces-permissions-errors-when-installing-packages-globally/)

- **`npm list -g` exit code 1 for absent packages**: when a package is not installed, `npm list -g <name>` returns exit code 1, causing `execSync` to throw ‚Äî must use try/catch and parse stderr/stdout rather than relying on exit code 0 ‚Äî [nodejs/help #4188](https://github.com/nodejs/help/issues/4188)
