## Domain Truth Findings

### First Principles

- **A pipeline stage's precondition is artifact presence, not prior-stage execution.** If stage N produces artifact A, and stage N+1 requires artifact A, then stage N+1 can run whenever A exists -- regardless of whether stage N produced it in this session, a prior session, or it was manually created. This is the foundation for mid-pipeline entry (EU-01): the contract is artifact satisfaction, not execution history. Current framing-pipeline enforces sequential stage numbering that masks this truth. — [First principles: a function's validity depends on its inputs being satisfied, not on the call stack that preceded it. If FEATURE.md has requirements and PLAN.md exists, execute can run. The pipeline is a DAG of artifact dependencies, not a linear call chain.]

- **Execution scope and review scope are independent concerns that must not be conflated.** Execution is decomposition: break work into parallel-safe per-feature units (each feature gets its own PLAN.md, SUMMARY.md). Review/doc is synthesis: detect emergent issues across those units. The optimal granularity for decomposition (small, independent) is the opposite of the optimal granularity for synthesis (broad, holistic). Per-feature review misses cross-cutting issues (state conflicts, interface violations). Monolithic execution loses parallelism. FN-01 and FN-02 correctly separate these -- executing per-feature, reviewing per-scope. — [First principles: decomposition and synthesis are dual operations with opposing optimal granularities. Integration testing literature validates this: components that pass individually often fail when combined due to interface mismatches and assumption divergence.]

- **In LLM-agent pipelines, context window exhaustion is the binding physical constraint on stage boundaries.** Unlike CI/CD where stages are separated by deployment environments, LLM pipeline stages are separated by context exhaustion. Each stage must be independently resumable because context loss is not exceptional -- it is expected. This means: (1) all inter-stage state must be persisted to files, not held in memory; (2) auto-chaining (FN-08) can only work within a single context window; (3) stage boundaries should align with natural context-flush points. — [First principles: LLM context windows are finite and non-recoverable once exhausted. Any multi-stage LLM workflow relying on in-context state will fail under real workloads. Source also: https://factory.ai/news/context-window-problem]

- **The executor/judge model-tier separation is a cost-quality optimization, not merely organizational.** Workers (researchers, reviewers, doc explorers) produce artifacts from instructions -- they need breadth and speed (Sonnet tier). Synthesizers and planners integrate multiple inputs under judgment -- they need reasoning depth (Opus tier). Misalignment wastes money (Opus doing rote work) or quality (Sonnet making synthesis decisions). TC-08 corrects 5 mismatched role_type declarations in the current codebase. — [https://www.emergentmind.com/topics/planner-executor-agentic-framework + First principles: cost-quality optimization requires matching model capability to task complexity]

### Universal Constraints

- **Idempotency is required for any pipeline supporting mid-entry or retry.** If a user can enter at /execute or /review, or if a stage can be re-run after failure, every stage must produce the same result regardless of how many times it runs with the same inputs. This means: overwrite (not append) artifacts, no side effects that accumulate across runs, and clear "already done" detection. The current SUMMARY.md-presence check for scope detection is a correct idempotency signal. — [https://www.prefect.io/blog/the-importance-of-idempotent-data-pipelines-for-resilience]

- **Fan-out/fan-in requires a failure threshold, not all-or-nothing semantics.** When N parallel agents run, some will fail (LLM non-determinism, context issues, tool errors). The system must define how many failures are tolerable before aborting and how the synthesizer handles partial inputs. GSD already does this correctly (>50% failure = abort for research, >=2/4 failure = abort for review). This threshold is a universal constraint for any parallel-agent architecture. — [https://dev3lop.com/fan-out-fan-in-parallel-processing-without-chaos/]

- **Human gates belong at judgment points, not at handoff points.** CI/CD best practice distinguishes automated gates (objective checks) from human gates (subjective decisions). In GSD's pipeline, judgment points are: review findings triage (accept/defer/dismiss) and doc approval. Stage transitions (plan->execute, execute->review) are mechanical handoffs that should auto-chain. Placing gates at handoffs creates friction; omitting them at judgment points creates runaway automation. FN-08 correctly places human gates only at review Q&A and doc approval. — [https://codefresh.io/learn/ci-cd-pipelines/ci-cd-process-flow-stages-and-critical-best-practices/]

- **Auto-chain semantics require unambiguous completion signals.** For execute->review->doc to auto-chain, each stage must emit a machine-readable completion signal, not prose. If execute.md ends with "The user decides next steps" (which it currently does), no downstream stage can reliably trigger. This is the fundamental contract of pipeline orchestration: stages communicate via structured signals. The current execute.md wiring bug is exactly this problem -- the auto-chain is specified in framing-pipeline.md but never implemented because execute.md lacks a completion signal. — [First principles: pipeline stages are connected by contracts. If stage N's output contract doesn't match stage N+1's input trigger, the chain breaks.]

### Validated Assumptions

- **Absorbing research into plan is valid because they share the same input contract.** Both research and planning take: FEATURE.md requirements + brief context + capability context. Research produces RESEARCH.md which feeds planning. Making this a single stage eliminates one inter-stage handoff and one context window flush. The gather-synthesize pattern (6 parallel gatherers + synthesizer) is preserved intact inside plan.md -- only the orchestration wrapper changes. Evidence: framing-pipeline.md Stage 1 and Stage 3 receive identical context payloads. — [/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/framing-pipeline.md lines 89-103 vs 239-257]

- **Requirements generation belongs upstream in discussion because it requires iterative human judgment.** Pipeline stages should be automatable or have bounded human interaction (approve/reject). Requirements generation is inherently iterative and exploratory. The current Stage 2 is the only pipeline stage that is fundamentally interactive rather than checkpoint-based. Moving it to discuss-feature aligns with the principle that pipelines process decided work, they don't decide work. FN-05 codifies this correctly. — [First principles: a pipeline optimizes throughput of known work; discovery optimizes understanding of unknown work. Mixing these in one flow creates impedance mismatch.]

- **SUMMARY.md presence is a sufficient and necessary scope-detection signal.** To determine which features have been executed, checking for SUMMARY.md files is both necessary and sufficient. No additional manifest is needed because SUMMARY.md is the execution output artifact (created by execute, consumed by review), it already exists per-feature, and its absence cleanly indicates "not yet executed." This avoids new artifact types per TC-06. — [First principles: the simplest correct signal is the output artifact of the stage you're checking for. A separate tracking artifact creates a synchronization problem between tracker and actual state.]

### Domain Risks

- **Context window exhaustion during auto-chain is the highest-probability failure mode.** The plan->execute->review->doc chain runs 4 stages in sequence within one context window. For any non-trivial feature, accumulated context from plan artifacts, execution summaries, review traces, and doc exploration will exceed available context. Research shows LLMs produce confident but incorrect results near context limits without explicit detection. FN-08 includes "present next command if context exhausted" but detecting exhaustion before quality degrades is the hard problem. — [https://blog.jetbrains.com/research/2025/12/efficient-context-management/]

- **Merging two orchestrators creates a single point of failure with higher cognitive load.** When capability-orchestrator merges into framing-pipeline (FN-06), the resulting file must handle both feature-scope (linear: plan->execute->review->doc) and capability-scope (DAG: wave-order features, then scope-level review+doc). This is a known risk in workflow consolidation: the merged orchestrator becomes harder to reason about. Mitigation requires clean conditional branching (not interleaved logic) and preserving DAG wave-ordering intact. — [https://kislayverma.com/software-architecture/architecture-pattern-orchestration-via-workflows/]

- **Backward compatibility (EU-04, FN-09) requires exhaustive reference-graph tracing.** When deleting capability-orchestrator.md and research-workflow.md, every file that references them must be updated. Missing a reference creates a silent failure: the workflow invokes a deleted file, and the LLM either hallucinates the content or errors. Unlike compiled languages, there is no linker to catch dangling references -- they surface only at runtime. TC-07 (CLI smoke test) catches some of these, but workflow-to-workflow references require manual grep verification. — [First principles: in systems where components reference each other by file path, deleting a file without updating all references creates dangling pointers that are undetectable until runtime.]
