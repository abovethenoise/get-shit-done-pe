### Boundary Conditions

- **Zero features in capability**: CAPABILITY.md Features table has no rows. DAG builder produces empty graph. Topological sort yields zero waves. Completion step would mark capability as "complete" despite no work. Must guard with early exit + informative message. -- `capability-orchestrator.md:31-35`

- **Single feature in capability**: DAG has one node, one wave. Capability-scope review degenerates to feature-scope review. Cross-feature analysis mandates (FN-01: "cross-scope state conflicts, interface contract violations") have nothing to cross-reference. Not an error, but the review quality claims are vacuous. -- [First principles: N=1 makes scope-fluid review equivalent to single-feature review]

- **SUMMARY.md exists but is 0 bytes**: Created by mkdir/touch during a crashed executor. Passes `test -f` but fails `test -s`. framing-pipeline research checks both (`framing-pipeline.md:153`), but FN-01 scope detection specifies presence only. Must use `-f && -s` consistently. -- `framing-pipeline.md:153` vs FEATURE.md FN-01

- **Feature with requirements but no PLAN.md (partial plan failure)**: Planner crashes after creating file but before writing content. Progress sees "has plans" and routes to Execute (Route A). Execute fails on empty plan content. Must check plan file size, not just existence. -- `progress.md:112`: "Feature has PLANs without SUMMARYs -> Execute"

- **RESEARCH.md lens mismatch during reuse**: plan.md Step 5 checks frontmatter lens match. Boundary: `lens` field is null or missing -- treated as stale, re-runs research. Correct behavior per `plan.md:74`. This check must survive the research-absorption into plan. -- `plan.md:72-74`

- **Concurrent pipeline runs on same feature**: Two Claude Code sessions run `/gsd:execute` on same feature. Both discover same incomplete plans, both write SUMMARY.md. No file-level locking. Git commits may conflict or one overwrites the other. -- [First principles: no shared lock mechanism in gsd-tools; each session has independent process state]

- **Maximum execution scope**: Capability with 10+ features, each with 3-5 plans. Context assembly for review = Layer 1-3 context (~15k tokens) + 10 FEATURE.md (~30k) + 10 SUMMARY.md (~20k) + artifact file contents. Approaches or exceeds useful context even before reviewer agents read code. -- [First principles: 200k context window, but recall degrades beyond ~80k tokens empirically]

- **Re-review cycle count at capability scope**: max_re_review_cycles=2 per review invocation. Capability with 5 features: cycle 1 finds issues in feature A, cycle 2 finds issues in feature B (introduced by fix to A). No cycle 3. Remaining issues surface "for manual resolution" even though caused by the fix loop. -- `review.md:138-140`

- **Escalation backward-reset budget exhausted early**: Budget is 1 reset per pipeline run (`framing-pipeline.md:70-71`). If consumed during research phase, no resets remain for execute/review. For capability scope (multiple features), escalation probability increases linearly with feature count. -- `framing-pipeline.md:70-71`

- **Compound lens (primary + secondary)**: Requirements now come from discussion (FN-05), not pipeline Stage 2. If discussion used primary lens weighting but pipeline executes with a different secondary lens, the lens mismatch between requirements and execution is invisible. No validation compares FEATURE.md lens context with pipeline lens. -- [First principles: requirements are upstream of pipeline; lens context may diverge]

- **Overlapping focus groups**: Feature X in both focus group A and B (both active). Progress routing reads `active_focus` from STATE.md frontmatter -- a singular field, not an array. Cannot represent parallel focus group membership. -- `progress.md:92-93`; STATE.md has singular `active_focus` field

### Integration Failure Scenarios

- **gsd-tools CLI crashes** -> Every workflow init step fails. All workflows start with `gsd-tools init ...` and parse the output as JSON. No workflow has error handling around the parse. Malformed config.json, corrupt STATE.md, or missing .planning/ directory all trigger this. -- `review.md:21`, `doc.md:21`, `execute.md:19`, `plan.md:23-26`

- **Claude Code Task() API changes** -> Task() is the sole mechanism for subagent spawning. All gather-synthesize patterns depend on parallel Task() calls. If Claude Code introduces sequential-only mode, timeout limits, or max-concurrent-task caps, 6 research gatherers + 4 reviewers + 5 doc explorers serialize across the pipeline. Context budgets and wall-clock time multiply. -- [First principles: Task() is an undocumented Claude Code SDK API with no stability guarantee]

- **FEATURE.md format/structure change** -> Reviewers parse EU/FN/TC sections using pattern matching on `EU-xx`, `FN-xx`, `TC-xx` IDs. If discuss-feature produces requirements with different numbering (e.g., `REQ-01` instead of `EU-01`) or different section headers, requirement tracing breaks silently. No schema validation exists. -- `review.md:44`: "Extract requirement IDs from FEATURE.md (EU-xx, FN-xx, TC-xx)"

- **plan-validate receives wrong REQ_SOURCE** -> After TC-03 removes Stage 2, requirements live in FEATURE.md only. plan.md Step 8.2 calls `plan-validate "${REQ_SOURCE}" ${PLAN_FILES}`. If REQ_SOURCE still references REQUIREMENTS.md (legacy path) instead of FEATURE.md, validation passes vacuously (no reqs to check). -- `plan.md:233`; init returns `requirements_path` which may point to legacy location

- **STATE.md exceeds size limit** -> Workflows read STATE.md for context. If it grows beyond the 100-line constraint, it occupies excessive context in the orchestrator and every subagent. Over multiple pipeline runs, STATE.md accumulates decisions, blockers, and session data without pruning. -- `templates/state.md` specifies 100-line constraint

- **Auto-chain interrupted by session timeout** -> Execute completes, orchestrator starts chaining to review, session times out. No persistent marker records that auto-chain was in-progress. Next session: user runs `/gsd:progress`, sees "complete, not reviewed" (Route B), must manually invoke `/gsd:review`. The auto-chain intent is lost. -- `framing-pipeline.md:294-298` (auto-chain is in-process, not persisted to disk)

- **CAPABILITY.md Features table malformed** -> Missing Depends-On column or malformed markdown table syntax. DAG builder cannot parse dependencies. If it treats missing column as "no dependencies," all features land in Wave 1 (concurrent). If it errors, entire orchestration halts. -- `capability-orchestrator.md:step 2`

### Existing Error Handling (gaps)

- **`review.md`**: handles reviewer agent failure (retry once, abort if >=2 fail, Step 5) but NOT capability-scope input assembly. Steps 2-3 reference `FEATURE_SLUG` singular throughout. No path exists to assemble multi-feature artifact lists. -- `review.md:23-44`

- **`doc.md`**: handles explorer failure (retry once, abort if >=3 fail) but NOT capability-scope invocation. The `gsd:doc` command already has capability-level iteration (Step 6, lines 101-118), but this runs doc.md per-feature -- the old N-times pattern the refactor aims to eliminate per TC-04. -- `doc.md:116-117`; `commands/gsd/doc.md:114-118`

- **`execute.md`**: handles plan failures and classifyHandoffIfNeeded bug (lines 108-109) but does NOT persist per-feature pass/fail status to any artifact. After execution, the only evidence is SUMMARY.md file presence. No structured execution-outcome record for downstream stages to consume. -- `execute.md:107-115`

- **`capability-orchestrator.md`**: handles per-feature failure in waves (Step 4b.3 -- "Continue" or "Abort") but does NOT record which features completed vs failed. Completion (Step 5) only updates CAPABILITY.md and ROADMAP.md with aggregate status. -- `capability-orchestrator.md:105-113`

- **`progress.md`**: handles missing PROJECT.md/STATE.md (Step initialize) but does NOT handle empty focus groups. Dependency readiness check (lines 90-106) reads "depends:" entries from ROADMAP.md focus group section -- if no focus groups exist, this code path reads nothing and produces undefined behavior in the comparison. -- `progress.md:90-106`

- **`core.cjs` resolveSlugInternal**: handles empty input (line 396), cap/feat format (line 402), single slug (line 413), fuzzy match (lines 431-468), and no-match (line 472). Does NOT handle missing `.planning/capabilities` directory distinctly -- the try/catch at line 445 silently returns empty candidates, yielding "no_match" without explaining the root cause. -- `core.cjs:445`

- **`init.cjs` cmdInitFeatureOp**: validates capSlug and featSlug non-empty (line 587) but does NOT validate that the `op` parameter is a recognized value. Any typo passes through as `operation: "typo"` in the result, and the workflow never checks it. -- `init.cjs:586-601`

### Known Issues in Ecosystem

- **Claude Code Task() false failure (classifyHandoffIfNeeded)**: Known runtime bug where Task() reports failure despite successful subagent completion. Already documented in `execute.md:109`. Affects all parallel Task() spawns. Mitigation: spot-check output files before trusting failure signal. -- `execute.md:109`

- **Claude Code `model="inherit"` resolution uncertainty**: The `inherit` parameter means "same model as parent session." If user is on a plan without Opus access, `inherit` resolves to Sonnet, breaking the executor/judge tier separation. No documentation guarantees resolution behavior. -- `core.cjs:20`: comment says "gets Opus from user's session" but this is an assumption, not a contract

- **Claude Code context degradation with orchestrator accumulation**: Subagents get fresh 200k context windows. But the orchestrator (the main session running framing-pipeline.md) accumulates all stage outputs, banners, Q&A history. By the time it reaches Stage 5 (review), the orchestrator's effective recall is degraded. This affects auto-chain quality -- later stages receive worse orchestration instructions. -- [First principles: LLM recall degrades non-linearly with context length; orchestrator is the bottleneck]

- **YAML frontmatter parsing fragility**: `extractFrontmatter()` in frontmatter.cjs uses regex-based parsing. Multi-line YAML values, comments, or leading whitespace cause silent parse failures. At capability scope, if ANY feature's FEATURE.md has malformed frontmatter, that feature may be silently dropped from scope detection. -- `init.cjs` (regex patterns like `content.match(/^status:\s*(.+)$/m)`)
