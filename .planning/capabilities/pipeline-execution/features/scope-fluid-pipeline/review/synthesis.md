#### Finding 1: Doc command capability-scope path inconsistency

**Severity:** major
**Source:** end-user, quality (implicit via scope-fluid coverage)
**Requirement:** EU-04 / FN-02
**Verdict:** design gap (does not break existing behavior but creates inconsistent scope semantics)

**Evidence (from reviewer):**
- `commands/gsd/review.md:82` -- `Pass: CAPABILITY_SLUG, FEATURE_SLUG=null (capability scope -- review.md handles multi-feature artifact collection)` -- Review command correctly passes null FEATURE_SLUG for capability scope.
- `commands/gsd/doc.md:114-118` -- `For each feature in run list (sequentially): ... Pass: CAPABILITY_SLUG, FEATURE_SLUG={current feature}, LENS` -- Doc command iterates features individually, passing each feature's slug.
- Reasoning (end-user): When `/gsd:doc` is invoked at capability scope via the standalone command, each feature's doc explorers only see that single feature's artifacts. When invoked via framing-pipeline.md, doc.md receives FEATURE_SLUG=null and its Step 3 scope-fluid path collects all features. Two different behaviors for the same capability-scope intent.

**Spot-check:** verified -- `commands/gsd/doc.md:118` confirms `FEATURE_SLUG={current feature}` vs `commands/gsd/review.md:82` confirms `FEATURE_SLUG=null`.

### Conflicts

#### Disagreements

- **FN-09 orphaned references scope:** Functional and technical reviewers verdict FN-09 as "met" (scoping to `commands/`, `get-shit-done/`, `agents/`). Quality reviewer verdicts it "not met" (including README.md and `.documentation/`).
  - Resolution: The FN-09 requirement text says "No orphaned references to deleted files" without directory scoping. README.md is an authored file with an active stale reference. Quality reviewer's broader scope is more correct per the requirement text. However, the implementation plan explicitly scoped cleanup to specific directories, and `.documentation/` files are generated content. Splitting the difference: README.md is a real miss (major); `.documentation/` is expected to be caught by the doc stage.
  - Tiebreaker applied: no -- judgment sufficient.

- **Remediation loop interpretation:** Functional reviewer verdicts FN-03 as "met" based on framing-pipeline.md's loop alone. Quality reviewer flags the dual-loop risk as "not met (suspected)."
  - Resolution: FN-03 is met in framing-pipeline.md. The quality concern is valid but is about the interaction between two independently-correct mechanisms, not about either one failing its spec. Classifying as major quality concern rather than FN-03 failure.
  - Tiebreaker applied: no -- judgment sufficient.

#### Tensions

- **Scope-fluid collection: DRY vs separation of concerns:** Quality reviewer (Finding 8) notes that scope-fluid artifact collection logic appears in both framing-pipeline.md and the individual workflows, but accepts it as justified separation of concerns (pipeline = intent, workflow = implementation). No action needed but worth monitoring if the pattern drifts.

- **Doc command capability path: fix now vs defer:** End-user reviewer flags the doc command's per-feature iteration as inconsistent with review command's null-FEATURE_SLUG approach. This could be fixed by making doc command match review command's pattern, but the current approach works (each feature gets documented). The inconsistency matters most for the doc aggregator's cross-feature detection capabilities (terminology, orphaned docs) which would only fire with capability-scope awareness.
  - Assessment: Fix should align doc command with review command pattern (pass FEATURE_SLUG=null for capability scope) to enable cross-feature detection. This is the higher-value approach.
