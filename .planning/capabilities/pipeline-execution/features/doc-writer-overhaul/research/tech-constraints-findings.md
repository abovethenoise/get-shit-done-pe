## Tech Constraints Findings

### Hard Constraints

- **Task() model parameter accepts only "sonnet", "haiku", or "inherit" — "opus" is not valid.** Judges must use `model="inherit"` to run on Opus, which requires the user's parent session to be running Opus. — [/Users/philliphall/get-shit-done-pe/get-shit-done/references/model-profiles.md, lines 36-38]

- **Skill command files live at `~/.claude/commands/gsd/`.** That is the installation target, not the repo source. The repo source is `/Users/philliphall/get-shit-done-pe/get-shit-done/`, and install.js deploys to `~/.claude/`. A new `/gsd:doc` skill file must be added in both the repo source and re-installed to `~/.claude/commands/gsd/doc.md`. — [/Users/philliphall/.claude/commands/gsd/ (directory listing), install.js install pattern]

- **`init feature-op` is the CLI init route for review and doc.** It returns `feature_found`, `feature_dir`, `capability_dir`, `state_path`, `roadmap_path`, `has_research`, `has_plans`, etc. It does NOT return a LENS — LENS is always caller-supplied, never CLI-derived. The `/gsd:doc` skill must supply or infer LENS itself before calling `doc.md`. — [/Users/philliphall/get-shit-done-pe/get-shit-done/bin/lib/init.cjs, `cmdInitFeatureOp()` lines 586-655]

- **STATE.md frontmatter tracks `active_capability` and `active_feature` but NOT `pipeline_position` lens or doc-specific state.** The `pipeline_position` field exists as a text label (`extractField('Pipeline position')`), but it is not set by any pipeline workflow — it is cosmetic. A `/gsd:doc` no-arg invocation cannot read LENS from STATE.md via CLI; it must infer from FEATURE.md frontmatter or RESEARCH.md frontmatter. — [/Users/philliphall/get-shit-done-pe/get-shit-done/bin/lib/state.cjs, `cmdStateSnapshot()` lines 400-511; STATE.md session continuity section]

- **Each Task() spawns a fresh context window.** The orchestrating agent's in-memory context (including LENS) does not transfer to a Task() subagent. The orchestrator must explicitly embed LENS in each explorer prompt string. — [/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/gather-synthesize.md key_constraints: "The orchestrator passes PATHS not content. Each gatherer reads its own agent definition and context files in its fresh context window."]

- **`gsd-doc-writer` agent currently has `role_type: executor` which maps to Sonnet.** If the synthesizer must run on Opus/inherit, it cannot reuse the same `subagent_type="gsd-doc-writer"` with `model="inherit"` without also accepting that the agent file declares executor. The `model` parameter in Task() overrides the role_type resolution — so passing `model="inherit"` on a synthesizer Task() will override the declared role_type and run on Opus regardless. This means a single agent file CAN serve both explorer (sonnet) and synthesizer (inherit) roles via the Task() `model` parameter override, without requiring a new agent file. — [/Users/philliphall/get-shit-done-pe/agents/gsd-doc-writer.md frontmatter; /Users/philliphall/get-shit-done-pe/get-shit-done/references/model-profiles.md lines 7-13, 36-38]

- **Explorer findings output directory `{feature_dir}/doc/` does not exist in the current doc.md workflow.** Current doc.md only creates `.documentation/modules .documentation/flows .documentation/capabilities`. The gather phase will need a `mkdir -p "${FEATURE_DIR}/doc"` before spawning explorers. — [/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/doc.md Step 1]

- **The `gsd-doc-writer` agent's defined tool set is `[Read, Write, Bash, Grep, Glob]`.** This is sufficient for explorers that scan source files, review artifacts, and `.documentation/`. No new tools are needed. — [/Users/philliphall/get-shit-done-pe/agents/gsd-doc-writer.md frontmatter]

- **Context window budget: 5 parallel explorers + 1 synthesizer = 6 Task() calls.** The research gather-synthesize pattern has 6 gatherers; review has 4. 5 doc explorers is within the established pattern. No known hard ceiling on parallel Task() count in Claude Code is documented, but the existing 6-gatherer research phase validates this parallelism is feasible. — [/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/gather-synthesize.md; /Users/philliphall/get-shit-done-pe/get-shit-done/workflows/research-workflow.md; First principles: pattern is already working at N=6]

### Compatibility Issues

- **Single `gsd-doc-writer` agent handling both explorer and synthesizer roles via prompt differentiation**: The agent currently has a single defined goal ("generate module and flow reference docs"). If used as an explorer for the "friction reduction" or "standards & decisions" focus areas, it must ignore its default documentation output format (module/flow headings, 3-pass validation) and instead produce findings files. This is a role mismatch that must be resolved via explicit prompt override in the Task() call — the agent's own `## Role` and `## Success Criteria` sections would conflict with explorer behavior. The review pattern resolves this with separate agent files per reviewer; the research pattern uses separate agent files per gatherer. The FEATURE.md decision to reuse a single agent file is a design risk. — [/Users/philliphall/get-shit-done-pe/agents/gsd-doc-writer.md Role + Success Criteria sections; /Users/philliphall/get-shit-done-pe/get-shit-done/workflows/review.md (4 separate reviewer agent files); /Users/philliphall/get-shit-done-pe/agents/ (6 separate research gatherer agents)]

- **Explorer "code comments" focus area reads actual source files**: The `gsd-doc-writer` agent already has `Read, Grep, Glob` tools, so this is supported. However, the agent currently receives source file paths from the orchestrator. For the explorer pattern, each explorer gets a file list from the orchestrator (same as today). No compatibility issue — same tool set, same input pattern. — [/Users/philliphall/get-shit-done-pe/agents/gsd-doc-writer.md Tool Guidance; /Users/philliphall/get-shit-done-pe/get-shit-done/workflows/doc.md Step 3-4]

- **`/gsd:doc` no-arg invocation — LENS inference from git log**: FEATURE.md (TC-02) specifies "reads STATE.md session continuity or recent git commits to infer target." Git log parsing would require Bash inside the skill command. Existing skills (`/gsd:review`, `/gsd:execute`) do not use git log for inference — they require explicit args or AskUserQuestion. STATE.md "Stopped at" field (free text, e.g. `pipeline-execution/plan-presentation — Execution complete`) contains enough to extract the feature slug but not the LENS. — [/Users/philliphall/get-shit-done-pe/.planning/STATE.md Session Continuity section; /Users/philliphall/.claude/commands/gsd/review.md process]

- **Capability-level `/gsd:doc {cap}` routing**: FEATURE.md (TC-02) specifies "iterates reviewed features via capability-orchestrator pattern." The capability-orchestrator.md currently dispatches `framing-pipeline.md` per feature (full 6-stage pipeline). For `/gsd:doc {cap}`, only the doc stage should run per feature. Reusing `capability-orchestrator.md` directly would trigger full pipelines — incompatible. A lighter iteration loop is needed (iterate features with review artifacts, call `doc.md` per feature). — [/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/capability-orchestrator.md; FEATURE.md TC-02]

- **`doc.md` Step 4 currently uses `subagent_type="general-purpose"` not `subagent_type="gsd-doc-writer"`**: The current doc.md spawns with `subagent_type="general-purpose"`. The FEATURE.md specifies explorers should use `subagent_type="gsd-doc-writer"`. This is a spec alignment issue — moving to named subagent_type is compatible but requires verifying the named type resolves correctly in the Claude Code Task() API. The review pattern uses named types (`gsd-review-enduser`, etc.) successfully. — [/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/doc.md Step 4; /Users/philliphall/get-shit-done-pe/get-shit-done/workflows/review.md Step 4]

### Alternatives

- **Single agent role conflict** → **Two lightweight explorer-specific agent files** (e.g., `gsd-doc-explorer.md` + `gsd-doc-synthesizer.md`): Clean separation of concerns, consistent with research (6 agent files) and review (5 agent files) patterns. Adds 2 files but eliminates prompt-override maintenance risk. — [First principles: research and review both use per-role agent files; shared-agent-with-prompt-override is the exception, not the rule in GSD]

- **Blocked: `capability-orchestrator.md` for capability-level doc** → **Inline feature iteration loop in `/gsd:doc` skill**: Iterate features in CAPABILITY.md with existing review artifacts (check for `{feature_dir}/review/synthesis.md`), call `doc.md` per feature sequentially. Same pattern used inline in `capability-orchestrator.md` Step 4b, just scoped to doc only. No new workflow file needed. — [/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/capability-orchestrator.md Step 4b]

- **Blocked: LENS inference from STATE.md `pipeline_position`** → **LENS inference from RESEARCH.md frontmatter `lens:` field**: Reliable, machine-written, available whenever research phase ran. Fallback to "enhance" if absent. Already proposed in FEATURE.md FN-04 as the inference path. — [/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/framing-pipeline.md Stage 1 synthesizer prompt]

- **Blocked: No-arg LENS inference without artifacts** → **AskUserQuestion with lens selection** as final fallback if RESEARCH.md absent and STATE.md insufficient: Present 4-option lens picker before proceeding. Consistent with existing AskUserQuestion patterns across skills. — [First principles: user input is the safe fallback when inference is ambiguous; AskUserQuestion is the established GSD pattern for human checkpoints]
