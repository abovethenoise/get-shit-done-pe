## Domain Truth Findings

### First Principles

- **Documentation staleness is structurally caused, not behaviorally.** Documentation drifts not because developers are careless but because the feedback loop between code change and documentation update is broken. The moment documentation exits the change-coupled flow, decay is inevitable. The correct fix is to close the loop — not ask developers to be more disciplined. — [https://www.tabnine.com/blog/ai-code-documentation-why-documentation-hurts-and-how-ai-helps/]

- **Parallel exploration beats serial investigation when work is decomposable and concerns are orthogonal.** Google Research found parallel agents improved performance by 80.9% on parallelizable tasks but *degraded* it by 39–70% on tasks requiring strict sequential reasoning. The documentation surface (code comments, module docs, standards, config, friction) maps to distinct artifact types with no logical dependency between them — each can be investigated independently. This is the textbook decomposable case. — [https://research.google/blog/towards-a-science-of-scaling-agent-systems-when-and-why-agent-systems-work/]

- **The synthesizer role is not cosmetic — it is where value is produced.** Parallel gathering without synthesis produces an undifferentiated pile of findings. The synthesizer's job — deduplication, conflict resolution, priority ordering — is where recommendations become actionable rather than noise. A synthesizer that only concatenates gatherer outputs fails the user. — [First principles: gathering creates breadth; synthesis creates signal. Deduplication and priority are irreducible steps between raw findings and actionable output.]

- **Lens/context at documentation time determines which changes matter, not which files were touched.** A refactor that moved 500 lines without changing behavior warrants migration notes and before/after mappings, not new API docs. A bug fix warrants root-cause documentation, not architecture docs. The same set of changed files can require completely different documentation depending on why the change happened. Lens must be a first-class input, not an afterthought. — [First principles: the purpose of documentation is to answer future questions; which questions arise depends entirely on the nature of the change, not just its extent.]

### Universal Constraints

- **Focus area boundaries must be enforced at spawn time, not at synthesis time.** Anthropic's multi-agent research system found that vague subagent instructions caused parallel agents to perform identical work rather than divide labor — "researching the semiconductor shortage" caused multiple agents to independently investigate the same 2025 supply chain events. Clear, exclusive focus area assignment is the only reliable partition mechanism. — [https://www.anthropic.com/engineering/multi-agent-research-system]

- **Partial failure in parallel exploration is recoverable; total failure is not.** In any N-agent parallel system, individual agent failures must be non-fatal to the overall workflow — the synthesizer must be designed to work with whatever subset of findings arrives. AWS prescriptive guidance identifies timeout and partial result handling as primary design requirements for scatter-gather patterns. GSD's research workflow already encodes this: ">50% failure threshold" triggers abort, not individual failures. This constraint applies equally to doc explorers. — [https://docs.aws.amazon.com/prescriptive-guidance/latest/agentic-ai-patterns/parallelization-and-scatter-gather-patterns.html]

- **Documentation recommendations without a target file and change description are noise, not signal.** Automated documentation tooling fails when it generates suggestions that require human interpretation to apply. The minimal useful recommendation unit is: (target artifact, current state, proposed change, rationale). Without all four, the user must do reconstruction work that defeats the purpose of automation. — [https://semaphore.io/blog/ai-tools-software-documentation — "AI may not fully grasp the nuances of user needs and the contextual use of the software, which can lead to less user-friendly documentation."]

- **The full documentation impact surface after a feature lands extends beyond the files directly modified.** Research on documentation drift establishes that downstream documentation (flow docs referencing modified modules, standards that assumed prior behavior, CLAUDE.md instructions that no longer apply) silently becomes stale without any signal. A documentation system that only covers directly modified artifacts misses the majority of the impact surface. — [https://www.augmentcode.com/learn/auto-document-your-code-tools-and-best-practices — "As your codebase is constantly evolving...your documentation starts getting stale."]

### Validated Assumptions

- **Gather->synthesize is a proven documentation pattern, not speculative.** The pattern "divide the document into sections, summarize each section in parallel, then combine the summaries" is described as a "common pattern" in documented multi-agent system design. AWS prescriptive guidance lists "document summarization across parallel subtasks" as an ideal scatter-gather use case. GSD already validates this with the 6-gatherer research workflow. The assumption that the same pattern transfers to documentation exploration is grounded in structural similarity (both involve decomposable investigation followed by synthesis). — [https://docs.aws.amazon.com/prescriptive-guidance/latest/agentic-ai-patterns/parallelization-and-scatter-gather-patterns.html]

- **A single agent handling both explorer and synthesizer roles (via prompt differentiation) is architecturally viable.** The key requirement is role clarity at spawn time — the agent must know whether it is in explorer mode (focused investigation, write findings file) or synthesizer mode (read all findings, produce unified recommendations). Anthropic's system uses "detailed task decomposition with clear boundaries" as the primary coordination mechanism, not agent type differentiation. Prompt differentiation is sufficient when boundaries are explicit. — [https://www.anthropic.com/engineering/multi-agent-research-system]

- **Defaulting lens to "enhance" for standalone doc invocations is correct.** The most common post-feature-implementation state is incremental change to existing capabilities. "Enhance" documentation emphasis (what changed, updated modules/flows, decision rationale) is the appropriate default because it captures the delta without overgenerating new architecture docs or root-cause analysis for changes that aren't bugs. This is validated by the GSD STATE.md session continuity pattern — doc is almost always run after review, which follows execution of an enhancement. — [/Users/philliphall/get-shit-done-pe/.planning/STATE.md — "Stopped at: pipeline-execution/plan-presentation — Execution complete (3/3 plans), ready for Review stage"]

### Domain Risks

- **Overlapping explorer scope produces conflicting recommendations that the synthesizer cannot resolve without authoritative partition rules.** If the "standards & decisions" explorer and the "module & flow docs" explorer both notice the same architectural pattern, they may generate contradictory recommendations for how to document it. This is the primary failure mode of multi-agent systems identified in literature: "sub-agents cannot see sibling work — if one assumes fiscal year ends in December and another assumes March, the parent receives conflicting outputs." The fix is enforcing non-overlapping scope at the explorer definition level, not at synthesis time. — [https://arxiv.org/html/2501.06322v1 — "Multi-Agent Collaboration Mechanisms: A Survey of LLMs"]

- **Documentation recommendations produced without lens context degrade into generic coverage rather than targeted insight.** A doc system that generates the same recommendations regardless of whether the change was a bug fix or a refactor forces users to manually filter irrelevant suggestions. This is analogous to the "SEO content farm" failure in Anthropic's research system — volume without quality. The risk for the doc-writer overhaul is that lens propagation to explorers is treated as optional or cosmetic rather than structurally required. — [https://www.anthropic.com/engineering/multi-agent-research-system — "Agents chose SEO-optimized content farms over authoritative...sources"]

- **Coordination overhead grows with explorer count; 5 explorers is near the upper bound for manageable synthesis.** Literature on multi-agent systems consistently identifies "every additional agent means another LLM call" and "if agents waited on each other, response times grew quickly — in some cases, three-agent chains tripled both cost and delay." At 5 parallel explorers, the coordination overhead (sequential synthesizer read of 5 files) is bounded and manageable. Expanding beyond 5 focus areas in future should trigger a review of whether synthesis quality degrades. — [https://www.deepchecks.com/how-multi-agent-llms-differ-from-traditional-llms/]

- **The Q&A loop's value is contingent on recommendations being specific enough to approve or reject without ambiguity.** If synthesized recommendations are vague (e.g., "update module documentation"), the Q&A loop becomes a burden rather than a filter — users must interpret what "approve" means. The current gsd-doc-writer agent already encodes the correct constraint: every finding must include target file, current state, recommended change, rationale, and REQ IDs. This structure must be preserved end-to-end from explorer output through synthesizer to Q&A presentation. Relaxing it at any stage collapses the loop. — [/Users/philliphall/get-shit-done-pe/.planning/capabilities/pipeline-execution/features/doc-writer-overhaul/FEATURE.md — TC-03: "Each finding: target file, current state, recommended change, rationale, REQ IDs if applicable"]
