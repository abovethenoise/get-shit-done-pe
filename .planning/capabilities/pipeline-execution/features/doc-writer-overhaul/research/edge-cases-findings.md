### Boundary Conditions

- **Standalone invocation with FEATURE.md but no SUMMARY.md:** The artifact location step (doc.md Step 3) reads SUMMARY.md for key files. If the feature was specified (FEATURE.md exists) but never executed (no SUMMARY.md), the key files list is empty. The code-comments explorer has nothing to scan. Must either error ("no execution artifacts") or produce a recommendations-only report with a clear warning. `doc.md` Step 3 says "Supplement with git diff" — if git diff also shows nothing, all code-related explorers operate blind. — `doc.md` Step 3; `FEATURE.md` EU-02 Out of Scope: "Running doc without any prior execution/review artifacts existing"

- **Capability-level with all features already documented (status=complete):** The capability-orchestrator skips features marked `complete`. If every feature is already complete, the doc run processes zero features and produces no output. The `/gsd:doc {cap}` invocation should detect this and present a summary rather than silently exiting. — `capability-orchestrator.md` Step 3: "Features already marked complete in the status are skipped"

- **Explorer count at abort threshold: exactly 3/5 fail:** The research workflow aborts at >50% failure (more than 3/6). With 5 explorers, 3/5 = 60% > 50% — abort triggered. But 2/5 = 40% — proceeds. The exact threshold for 5 explorers must be stated explicitly in the doc.md restructure. Using the existing formula (>50%): 3+ failures abort, 2 or fewer proceed. — `gather-synthesize.md`: "if failed_count / total_gatherers > 0.50"

- **FEATURE.md `lens` frontmatter field absent:** FEATURE.md frontmatter schema is `type`, `capability`, `status`, `created` — no `lens` field. FN-04 says standalone invocation infers LENS from "FEATURE.md or RESEARCH.md frontmatter." FEATURE.md will never carry this field unless the schema is extended. RESEARCH.md does carry it (from framing-pipeline synthesis). This is the reliable inference source. — `agents/gsd-doc-writer.md` YAML header; `framing-pipeline.md` Stage 1 synthesizer prompt

- **Synthesizer receives all 5 explorer findings files but they conflict on every recommendation:** The synthesizer is instructed to "resolve conflicts" (FN-03) but no priority ordering is defined for doc focus areas. With 5 areas all recommending the same target file for different reasons, the synthesizer must use judgment. This is acceptable (synthesizer is judge-role/Opus) but should be documented as an expected behavior, not an error. — `FEATURE.md` FN-03; [First principles: without explicit conflict priority, LLM synthesizer will use contextual judgment — acceptable for doc recommendations but may produce inconsistent results]

- **Q&A loop receives recommendation entries with no "content preview" (recommendations are instructions, not content):** Current Step 7 displays "content preview" of generated docs. A recommendation entry like "Add docstring to gsd-tools.cjs:parseFeature() explaining the 3-tier slug resolution logic" has no content to preview. The loop must present the recommendation's what/why fields instead, and the Approve/Edit/Reject semantics must shift to Approve/Revise/Skip. FN-06 says "same AskUserQuestion loop" — the options must remain structurally compatible but their display labels and input handling will need updating. — `doc.md` Step 7; `FEATURE.md` FN-06

- **`/gsd:doc` with no arguments invoked immediately after a capability-level operation (STATE.md shows capability, not feature):** The session continuity entry may be "pipeline-execution — research complete" (capability-scoped). FN-04 says "infers target from most recent review activity" — but STATE.md does not track per-feature review completion timestamps, only the "Stopped at" free-form field. The git log fallback must look for commits containing `docs({cap}/{feat})` or `review({cap}/{feat})` patterns to identify the most recent reviewed feature. — `STATE.md` Session Continuity structure; `doc.md` Step 10: `git commit -m "docs(${CAPABILITY_SLUG}/${FEATURE_SLUG}): add generated documentation"`

### Integration Failure Scenarios

- **gsd-tools CLI fails during `/gsd:doc` skill init** → `feature_found` cannot be determined → skill cannot resolve CAPABILITY_SLUG/FEATURE_SLUG → entire invocation fails before any explorers spawn. Same failure mode as all other pipeline stages. No special mitigation needed beyond existing error handling pattern. — `doc.md` Step 1: `node "$HOME/.claude/get-shit-done/bin/gsd-tools.cjs" init feature-op`; `review.md` Step 1: identical pattern

- **DISCOVERY-BRIEF.md missing when `/gsd:doc` tries to infer LENS from it** → LENS inference falls through to "enhance" default. This is the specified behavior (TC-02). No failure — graceful degradation. — `FEATURE.md` TC-02: "infers LENS from FEATURE.md or RESEARCH.md frontmatter if available, defaults to 'enhance' if not determinable"

- **review/synthesis.md absent (review was never run or was deleted)** → doc.md Step 3 finds no review artifacts. Code Comments and Module & Flow Docs explorers can still operate on SUMMARY.md + git diff. Standards & Decisions, Project Config, and Friction Reduction explorers can operate without review artifacts (they analyze config files and workflows). Impact: WHY blocks in generated docs will be empty (WHY blocks require cited review findings per `gsd-doc-writer.md`). — `agents/gsd-doc-writer.md`: "Review findings (synthesis.md) --> 'why is it this way' (WHY blocks -- only from cited findings)"

- **framing-pipeline.md invokes doc.md but omits LENS from framing_context** → doc.md receives no LENS value. Current workflow signature requires LENS as an input (`doc.md` inputs section). If nil, Step 2 Layer 4 "lens-specific doc focus" is undefined. Explorers receive no lens emphasis. Behavior degrades to "enhance" defaults without explicit fallback logic in doc.md. — `doc.md` inputs: "LENS: Primary lens (debug | new | enhance | refactor)" — no documented fallback if absent

- **Task() parallel spawn limit exceeded: 5 explorers + upstream tasks in same context** → The Claude agent SDK does not publish a hard concurrent Task() limit. At 5 explorers plus the synthesizer (6 total doc Tasks), plus any surrounding orchestration, this is within observed behavior for research (6 gatherers). Risk is theoretical but present if the pipeline stage context is already running multiple concurrent Tasks from a parent orchestrator. — [First principles: Claude SDK Task() concurrency is bounded by the runtime; 5 parallel spawns mirrors the established 6-gatherer research pattern and is likely safe]

- **capability-orchestrator.md dispatches doc.md per feature sequentially but framing-pipeline.md auto-chains doc after review** → If `/gsd:doc {cap}` is used AFTER features already completed the full pipeline (review auto-chained to doc), the doc stage runs again for already-documented features. This produces duplicate doc-report.md files and duplicate `.documentation/` entries. The capability-level doc skill must check whether `{feature_dir}/doc-report.md` already exists and prompt before overwriting. — `review.md` Step 12 auto-chain: "Auto-invoke doc workflow"; `FEATURE.md` FN-04 capability-level behavior

### Known Issues in Ecosystem

- **LLM-as-parallel-agent pattern (gather-synthesize):** Known issue is quality variance — parallel agents with the same base model produce inconsistent depth across dimensions. The research workflow mitigates this with dimension-specific agent definitions (6 distinct agent files). The doc overhaul proposes a single `gsd-doc-writer` agent for all 5 explorer roles via prompt differentiation. This is explicitly weaker than the research pattern and risks shallower findings from focus areas whose prompts are less precisely scoped. — `FEATURE.md` TC-01: "single gsd-doc-writer agent handles both explorer and synthesizer roles via prompt differentiation (same as research gatherers use dimension-specific prompts with shared agent type)" — note: research gatherers do NOT share agent type, they use 6 distinct agent files

- **Claude Task() output size:** Each explorer produces a findings file. If a large codebase produces very large findings (many code comment gaps across hundreds of files), the synthesizer's context window fills before it can process all 5 findings files. No mitigation is defined in the FEATURE.md. The research pattern avoids this by scoping each gatherer to a specific dimension — the doc pattern should similarly scope each explorer to a bounded file set (already implicit in TC-03 but not enforced). — [First principles: LLM context windows are finite; 5 findings files + synthesizer context + core context can exceed limits on large projects]

- **STATE.md free-form text parsing for `/gsd:doc` no-arg target inference:** STATE.md "Stopped at" is authored text with no schema. Parsing "pipeline-execution/plan-presentation — Execution complete" to extract feature slug requires either regex matching on the `cap/feat` pattern or a structured field the skill can read. The git log approach is more reliable: `git log --oneline --grep="review(" -n 5` extracts the most recent review commit's cap/feat slug from the commit message format. — `STATE.md` line 169: "Stopped at: pipeline-execution/plan-presentation..."; `doc.md` Step 10 commit message format: `docs(${CAPABILITY_SLUG}/${FEATURE_SLUG}): add generated documentation`
