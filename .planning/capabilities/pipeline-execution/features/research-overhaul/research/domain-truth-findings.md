### First Principles

- **Ambiguity in instructions produces ambiguous model behavior, not error states.** When an LLM receives an instruction it can interpret multiple ways, it does not halt or flag the ambiguity — it picks one interpretation silently, typically the lowest-effort one that satisfies the surface form of the request. "Invoke @workflow.md" is a reference, not a command; the model interprets it as "read this file" or "delegate responsibility to this file" rather than "spawn Task calls per the instructions in this file." This is the root mechanism of the GSD shortcut failure. — [First principles: LLMs are trained to complete requests plausibly, not pedantically. Underspecified instructions cause the model to fill the gap with its own inference, and that inference defaults to the most direct path available. Confirmed by: arxiv 2505.13360v2: "accuracy drops by an average of 22.6% (and up to 93.1%) compared to when the requirement is explicitly stated"; Anthropic best practices: "If you say 'can you suggest some changes,' Claude will sometimes provide suggestions rather than implementing them"]

- **Explicit operational syntax overrides implicit semantic meaning.** The Task() pseudo-code pattern in plan.md Step 7 works because it specifies the *operation* to perform (spawn an agent) and the *parameters* to pass (prompt, model, description). The "Invoke @" pattern in Step 5 specifies only a *reference* — it leaves the spawning action unspecified. The fundamental principle: instructions that specify what to *do* (an operation) are followed more reliably than instructions that specify what to *read* or *consult* (a reference). — [First principles: LLMs execute operations more reliably than they dereference and execute implied operations from references. A reference points to something; an operation directs something. The gap between "read this and use it" and "do this explicit thing" is where interpretation variance enters. Confirmed by Anthropic docs: "To make Claude take action, be more explicit... Claude is trained for precise instruction following and benefits from explicit direction"]

- **A pipeline step that can be skipped will be skipped under resource pressure.** LLMs operating agentic tasks make micro-decisions about whether each step is "really necessary" given the cost (tokens, turns) of executing it. Skip gates (--skip-research, research_enabled, has_research) communicate to the model that the step is optional — that there exist conditions under which skipping is valid. Once the model has learned that skipping is a valid branch, it will tend toward that branch when the check is close-call. This is not a safety/trust issue — it is a structural issue. The presence of a skip path trains the model's inference about the step's optionality. — [First principles: Models learn from the structure of their instructions, not just their content. A conditional gate around a step communicates that the step is optional. Mandatory steps must be structurally non-skippable, not merely labeled "mandatory." Confirmed by metaswarm framework: "mandatory quality gate intercepts at every handoff point — agents cannot skip" via structural enforcement, not prompting]

### Universal Constraints

- **Subagents cannot spawn other subagents in Claude Code.** This is a hard platform constraint. The orchestrating workflow (plan.md, framing-pipeline.md) is running in the main conversation context — it is the only level that can invoke Task() calls. If plan.md's Step 5 fails to spawn the research gatherers and instead delegates to research-workflow.md as a subagent, that subagent cannot then spawn the 6 gatherers itself. The gather-synthesize pattern only works when the main context is doing the spawning. — [source: https://code.claude.com/docs/en/sub-agents — "Subagents cannot spawn other subagents. If your workflow requires nested delegation, use Skills or chain subagents from the main conversation."]

- **Each subagent receives only its own system prompt plus basic environment details — not the parent conversation context.** If research-workflow.md is invoked as a subagent reference rather than executed in the main context, the spawned subagent cannot see the framing_context, feature_path, lens metadata, or any of the assembled context payload that plan.md built. It runs blind. This constraint means that lens-aware research requires the research orchestration to happen in the main context, where the assembled context is available to pass to each gatherer via Task() prompts. — [source: https://code.claude.com/docs/en/sub-agents — "Subagents receive only this system prompt (plus basic environment details like working directory), not the full Claude Code system prompt"]

- **Underspecified prompt requirements are 2x more likely to regress across model updates.** Requirements that are implicit rather than explicit have a standard deviation 2x higher in compliance, and 5.9% of unspecified requirements regress more than 20% when the model is updated — nearly 2x the rate of specified requirements. For a workflow used every time a feature is planned, this variance compounds. Any requirement that must hold across all executions must be explicitly stated. — [source: https://arxiv.org/html/2505.13360v2 — "accuracy drops by an average of 22.6% (and up to 93.1%)... standard deviation of 8.9%, a more than 2x increase compared to when they are specified... 5.9% requirements regress more than 20% over model updates when they are unspecified"]

- **Parallel tool calling requires explicit instruction to reach ~100% execution rate, but can be nudged to that rate through prompting.** Claude 4.x models will parallelize independently in many cases, but this behavior is "easily steerable" and can be boosted to near-100% with explicit prompt instruction. Conversely, the absence of explicit parallelization instruction means the model may execute sequentially by default in ambiguous cases. For the 6-gatherer parallel spawn to be reliable, the instruction to spawn all gatherers simultaneously must be unambiguous. — [source: https://platform.claude.com/docs/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices — "While the model has a high success rate in parallel tool calling without prompting, you can boost this to ~100%... use_parallel_tool_calls prompt block"]

### Validated Assumptions

- **The Task() pseudo-code pattern (explicit params, explicit model, explicit description) is the established unambiguous delegation pattern for Claude Code.** Plan.md Step 7's use of `Task(prompt=..., subagent_type=..., model=..., description=...)` matches the documented Claude Code delegation syntax precisely. This is not a GSD-specific invention — it mirrors the official subagent spawning pattern. Step 5's "Invoke @workflow.md" is neither the official pattern nor a documented delegation mechanism — it has no analog in Claude Code's architecture. — [source: https://code.claude.com/docs/en/sub-agents; confirmed by gather-synthesize.md Step 2 template which uses the same Task() pseudo-code pattern for gatherer spawning]

- **The model self-diagnosed the indirection failure correctly.** The user transcript documents the model identifying that "Invoke @workflow.md" created ambiguity: it could mean "delegate to this file" or "read and execute yourself." The brief at `/Users/philliphall/get-shit-done-pe/.planning/capabilities/pipeline-execution/BRIEF.md` confirms: "The model admitted the indirection caused it to take shortcuts — delegating to primary-collaborator or spawning only one agent instead of orchestrating the 6 parallel gatherers." This self-diagnosis is valid: the model is pattern-matching "Invoke @X" to its known behavior for "@-references" (read the file) rather than to an agentic spawn operation. — [source: BRIEF.md line 23; confirmed by Anthropic prompt engineering docs: "Think of Claude as a brilliant but new employee who lacks context on your norms and workflows"]

- **Lens-aware research reuse is an unsolved design problem in the current codebase — the binary `has_research` check is structurally insufficient.** The current check is a file-existence test: if RESEARCH.md exists, skip research. This ignores lens changes (research from /new lens does not cover /enhance needs), time elapsed, or scope changes. No existing system in the pipeline addresses this. The framing-pipeline.md and plan.md both have the research step but neither has logic to evaluate whether *existing* research is adequate for the *current* lens. — [source: plan.md Step 5: "If has_research AND no --research flag: Use existing, skip to step 6"; framing-pipeline.md has no has_research check at all — it always invokes research-workflow.md]

- **framing-pipeline.md Stage 1 uses the same ambiguous "Invoke @" pattern as plan.md Step 5.** The BRIEF.md identifies this as an assumption needing verification — it is confirmed. framing-pipeline.md line 86: `@{GSD_ROOT}/get-shit-done/workflows/research-workflow.md` with a Pass: block. This is structurally identical to the pattern in plan.md Step 5 that caused the failure. Both callers have the same root cause. — [source: /Users/philliphall/get-shit-done-pe/get-shit-done/workflows/framing-pipeline.md lines 83-106]

### Domain Risks

- **The Task() pseudo-code pattern alone does not guarantee the model understands it as a hard spawn command rather than pseudo-code to "follow loosely."** The Task() block is not executable syntax — it is natural language in code-block formatting. Without context establishing "this is the exact invocation pattern, not an illustration," the model may treat it as guidance rather than instruction. Plan.md Step 7 works reliably because it has a clear preceding header ("Spawn gsd-planner Agent") and the Task() block appears as the primary mechanism, not one of many options. The research step must achieve the same structural clarity. — [First principles: Code-block formatting creates an expectation of precise execution, but LLMs are not interpreters — they execute based on surrounding context. If the block is framed as an example, it will be treated as one. If it is framed as the instruction, it will be followed as one.]

- **Instructing models to parallelize research without specifying the exact agents to spawn produces inconsistent gatherer selection.** If plan.md says "spawn 6 gatherers in parallel" without naming them, the model may spawn different agent types or fewer agents than intended. The Task() pattern must include the specific agent paths and output paths for each gatherer to be deterministic. The current gather-synthesize.md Step 2 template already shows the correct pattern — callers need to instantiate it with the exact gatherer definitions. — [source: /Users/philliphall/get-shit-done-pe/get-shit-done/workflows/gather-synthesize.md Step 2 — template requires agent_path, dimension_name, output_path per gatherer; confirmed by research-workflow.md Section 3 which defines all 6 gatherer paths explicitly]

- **Over-instruction creates competing constraints that reduce compliance for the most critical requirements.** Research shows "simply adding more requirements to a prompt does not reliably improve performance" and "accuracy drops 19% when cramming too many requirements into a single prompt." Plan.md Step 5 currently has multiple competing gates (--skip-research, research_enabled, has_research, partial failure handling) — each gate is an additional constraint the model must track while also executing the spawn logic. Removing skip gates (as the Brief specifies) is not just a design preference: it reduces constraint load on the instruction that matters. — [source: https://arxiv.org/html/2505.13360v2; confirmed by Anthropic best practices: "Claude Opus 4.5 and Claude Opus 4.6 are also more responsive to the system prompt than previous models. If your prompts were designed to reduce undertriggering, these models may now overtrigger. The fix is to dial back any aggressive language."]

- **Research skippability is not just a UX issue — it is an epistemological risk to the entire pipeline.** The pipeline's downstream quality (plan, execute, review) depends on research findings grounding the planner in reality. A plan produced without research uses only the model's training knowledge about the domain, which may be stale, imprecise, or wrong for the specific codebase. For GSD's use case (planning code changes to an actual, specific codebase), unstated requirements about the codebase are exactly the class of requirements LLMs fail on most severely (up to 93.1% accuracy drop). Research is not an enrichment step — it is the mechanism that grounds planning in the specific, not the general. — [source: https://arxiv.org/html/2505.13360v2 — accuracy drop up to 93.1% for unspecified requirements; First principles: LLMs generalize from training; codebase-specific facts are outside training and must be supplied via research or RAG]
