# Research Synthesis

**Synthesized:** 2026-03-04
**Subject:** pipeline-execution/plan-presentation
**Gatherer Results:** 6/6 succeeded

## Consensus

Findings agreed upon by multiple gatherers. Higher confidence -- multiple independent analyses reached the same conclusion.

### The current flat-table + "approve?" checkpoint is a documented anti-pattern

The existing plan presentation (step 8.6 flat summary + step 8.3 per-finding loop that skips when empty) produces rubber-stamp approvals. Domain research shows "plausible plans generated by LLMs can mislead users into trusting the LLM agents when they are in fact wrong" (CHI 2025). Prior art confirms every reviewed tool (Cursor, Devin, Copilot Workspace, Claude Code Plan Mode) moved away from flat approval without justification. The existing system analysis confirms step 8.6 shows only "Feature, plan count, task count, waves / Validation status / Key decisions" with no ordering rationale or approach justification.

[Sources: Domain Truth (FP-1, FP-4, UC-3), Prior Art (Anti-Patterns), Existing System (Step 8.6 analysis), User Intent (AC-7)]

### Interactive Q&A must be always-offered, never conditional on findings

Current step 8.3 loops over findings via AskUserQuestion -- when findings list is empty, no Q&A fires. This means well-formed plans get the least scrutiny. Domain research establishes that "a skippable Q&A is functionally equivalent to no Q&A" and that human involvement in planning improves performance only when the interaction is meaningful. Tech constraints confirm the "always offer" model is viable by restructuring 8.3 from finding-reactive to plan-proactive. Prior art shows all reviewed tools offer unconditional plan review before finalization.

[Sources: Domain Truth (FP-3, UC-5), User Intent (AC-4), Tech Constraints (Feasibility: "Restructure 8.3"), Existing System (Step 8.3 constraint), Prior Art (Anti-Pattern: gated Q&A), Edge Cases (zero-findings boundary)]

### The deep-dive Q&A should offer named plan areas, not open-ended questions

Prior art from Claude Code Plan Mode, Devin, and Copilot Workspace all structure review around named areas. Domain research confirms that under cognitive load, structured choices outperform unstructured prompts. User intent implicitly requires named choices (IR-1). The canonical list converges across sources:

1. Wave ordering and task sequence rationale
2. Approach chosen vs alternatives rejected
3. Requirement coverage
4. Assumptions the planner made
5. Self-critique fix details

[Sources: Prior Art (Canonical Patterns, Ambiguity 2 resolution), Domain Truth (UC-2), User Intent (IR-1)]

### Justification narrative must be grounded and verifiable, not boilerplate

Domain research warns that fluent justification for a bad plan is more dangerous than no justification (the plausibility trap). The narrative must reference specific requirement IDs, dependency edges, and file references. User intent reinforces this (IR-2: "a narrative that could apply to any plan does not pass the 'why this specific plan?' test"). Prior art's ADR pattern provides the structural answer: Context / Decision / Consequences format forces grounded claims.

[Sources: Domain Truth (DR-1, UC-3), User Intent (IR-2), Prior Art (ADR pattern)]

### Round 1 fix surfacing requires a planner return format change

Round 1 fixes are applied silently and never written to disk or returned. The planner's context is discarded after Task() completes. Multiple gatherers independently conclude this data cannot be reconstructed post-hoc -- it must be captured during planning. Prior art resolves the invariant tension: adding a return field is a data format change, not a logic change, compatible with Invariant #3.

[Sources: Tech Constraints (Hard Constraint: "Round 1 silent fixes blocked"), Domain Truth (UC-4), Existing System (planner return format analysis), User Intent (AC-2, IR-3, ambiguity resolution), Prior Art (Ambiguity 1 resolution)]

### Visual flow diagrams should be conditional on plan complexity

A single-wave, 1-2 task plan has no meaningful architecture to diagram. Forcing a diagram violates KISS. Multiple sources converge: Edge Cases identifies this as a common boundary condition; Prior Art notes no reviewed tool emits diagrams for trivially simple plans; User Intent AC-5 must be interpreted against complexity. Domain research confirms visual structure aids topology comprehension but adds no value when topology is trivial.

[Sources: Edge Cases (simple plan boundary), Prior Art (Anti-Pattern: forcing diagrams), Domain Truth (VA-1), User Intent (Risk-3)]

### Progressive disclosure should be 2 levels, not 3 nested levels

NN/g research (cited by Prior Art) documents that 3+ disclosure levels produce low usability. The 3-layer justification maps to 2 disclosure levels: Layers 1+2 always shown (narrative + surfaced decisions), Layer 3 offered on demand (drill-down). This is not 3 nested navigational levels.

[Sources: Prior Art (Anti-Pattern: >2 progressive disclosure levels, Recommended Starting Point), Domain Truth (UC-2)]

### The planner (not orchestrator) should generate justification content

The planner has full reasoning context at decision time in a fresh 200k context window. The orchestrator is in the user's session with accumulated context. Tech Constraints identifies orchestrator-derived justification as "viable" but notes context growth risk. Domain Truth establishes that "the 'why not' dimension cannot be surfaced post-hoc" (UC-4). Prior art confirms: Cursor and Copilot Workspace generate rationale as part of the agent's output, not re-derived by the orchestrator.

[Sources: Tech Constraints (Feasibility table, Alternatives), Domain Truth (UC-4), Prior Art (Ambiguity 1 resolution)]

## Conflicts

Disagreements between gatherers. Each conflict includes both positions and a resolution.

### Whether step 8.3 findings loop is replaced or restructured

**User Intent says:** Step 8.3 per-finding Q&A is replaced by a single proactive AskUserQuestion offering named plan areas. The deep-dive replaces the finding-reactive loop.
**Existing System says:** Step 8.3 loop must not be eliminated, only restructured -- steps 8.4 and 8.5 depend on 8.3 having gathered user responses per finding. The output contract (accepted/edited/dismissed findings) must be preserved.

**Resolution:** Both can coexist. The per-finding loop for Round 2 findings remains (it feeds the re-spawn mechanism), but the deep-dive Q&A is a new step that runs regardless of finding count. Prior Art supports this separation: "two Q&A rounds with unclear relationship recreates the opacity problem" -- they must be clearly distinguished. The findings loop handles specific issues; the deep-dive handles plan-level interrogation. The findings loop runs first (if findings exist), then the deep-dive always runs, then finalize.

### Whether PLAN.md format needs any change

**User Intent says:** PLAN.md artifact format is unchanged (AC-8, hard invariant).
**Tech Constraints says:** The planner writes only PLAN.md files; to persist justification on disk requires either a new file (PLAN-RATIONALE.md) or extending the planner return format.

**Resolution:** No conflict. PLAN.md format stays unchanged. Justification is either (a) added to the planner's structured return message (a `### Justification` section in the "PLANNING COMPLETE" block) or (b) written as a separate file. Option (a) is simpler and avoids a new file artifact. The planner return format is a workflow convention in planner-reference.md, not the PLAN.md schema.

### Scope of planner-reference.md changes vs "planner logic unchanged" invariant

**User Intent says:** Invariant #3: "Planner agent planning logic unchanged."
**Tech Constraints says:** Surfacing Round 1 fixes and adding justification narrative both require planner-reference.md changes.
**Prior Art says:** Adding return fields is a data format change, not a logic change.

**Resolution:** Prior Art's distinction is the correct frame. The planner's planning algorithm (how it generates tasks, waves, dependencies, self-critique rounds) is unchanged. What changes is the return payload structure -- the planner writes additional sections in its completion message. This is analogous to adding a column to a database view: the query logic is the same, the output shape is different.

## Gaps

### Missing Dimensions

None -- all 6 gatherers succeeded.

### Low-Confidence Findings

- **AskUserQuestion character/content limits:** No gatherer found a documented limit for the `question` field. Edge Cases estimates ~3200 chars visible in terminal, but this is first-principles reasoning, not measured. The actual rendering behavior under very long question content is unknown. [Source: Edge Cases, single-source]

- **Instruction drift in multi-turn re-spawns:** Edge Cases notes that re-spawned planners may blend prior justification with new guidance rather than cleanly replacing contradicted claims. This is a known LLM failure mode but no mitigation was validated. [Source: Edge Cases, single-source]

- **Research ambiguity surfacing from legacy RESEARCH.md format:** Prior Art proposes checking for structured `<open_questions>` sections and falling back gracefully, but no gatherer verified whether current RESEARCH.md files actually contain parseable ambiguity sections. [Source: Prior Art + Edge Cases, low evidence]

### Unanswered Questions

- What is the actual terminal rendering behavior when AskUserQuestion `question` field exceeds ~100 lines?
- How does the 3-iteration max interact with justification-only revision requests? Should justification critique consume an iteration?
- When plan-checker findings contradict planner justification, what reconciliation UX should the user see?

## Constraints Discovered

Hard limits the planner MUST respect. These are non-negotiable -- violating them leads to incorrect, insecure, or broken implementations.

| Constraint | Source | Impact |
|-----------|--------|--------|
| PLAN.md artifact format is invariant -- no schema or frontmatter changes | User Intent (AC-8), Tech Constraints | Justification content lives in return messages or new files, never in PLAN.md |
| AskUserQuestion is the only interaction primitive | Existing System, Tech Constraints | All user-facing content goes through header + question + options; no free-form dialogue |
| AskUserQuestion options are short-form labels only | Tech Constraints | Rich content goes in `question` field; options remain brief strings |
| Step 8.3 output contract (per-finding accept/edit/dismiss) must feed steps 8.4-8.5 | Existing System | Cannot eliminate the findings loop; can restructure presentation around it |
| Max 3 iterations of the 8.1-8.5 loop is a hard cap | Existing System, Edge Cases | New presentation format must not extend loop mechanics |
| Round 1 fix data is ephemeral -- only accessible during planner execution | Tech Constraints, Domain Truth (UC-4) | Must be captured in planner return; cannot be reconstructed post-hoc |
| plan-checker is conditionally enabled (`plan_checker_enabled`) | Existing System | Step 8.8 format changes must handle both enabled and disabled checker paths |
| Planner returns via structured completion message, not in-context data | Tech Constraints | Orchestrator parses text return; any new data must be in the return format |
| plan.md is a markdown instruction file -- changes take immediate effect, no tests | Tech Constraints | Only verification is running the workflow; instruction ambiguity risks degraded behavior |
| Context budget pressure at step 8 | Tech Constraints, Domain Truth (DR-4) | Justification must earn its token cost through information density; bounded, not exhaustive |
| ui-brand.md has no ASCII flow diagram template | Existing System, Tech Constraints | New flow notation must be simple enough to work without a formal template |

## Recommended Scope

Actionable guidance for the planner: what to build, what to skip, what needs more investigation.

### Build (In Scope)

- **Extend planner-reference.md return format** with `### Justification` section (ordering rationale, approach rationale, KISS rationale) and `### Round 1 Fixes` summary -- planner generates these during planning when context is richest [Sources: Tech Constraints feasibility, Domain Truth UC-4, Prior Art Ambiguity 1]

- **Restructure plan.md step 8.3** to present justification narrative + surfaced decisions (Round 2 findings grouped by category + Round 1 fix summary) before the per-finding Q&A loop. Narrative always renders; findings loop runs if findings exist [Sources: User Intent AC-1/AC-2, Existing System integration points]

- **Add unconditional deep-dive AskUserQuestion** as a new step between findings resolution and finalize. Named plan area choices: wave ordering, approach vs alternatives, requirement coverage, assumptions, self-critique details. Runs regardless of finding count [Sources: User Intent AC-4, Prior Art canonical pattern, Domain Truth FP-3]

- **Replace step 8.6 flat summary** with the full 3-layer presentation: narrative + surfaced decisions + visual flow (when applicable) + finalize prompt. Approval comes after all layers [Sources: User Intent AC-6/AC-7, Prior Art plan-then-approve pattern]

- **Add conditional ASCII wave flow diagram** for plans with 2+ waves or 3+ plans. Derive from PLAN.md `wave` and `depends_on` frontmatter. Use simple `[Plan: objective] --> [Plan: objective]` notation. Omit for trivially simple plans [Sources: User Intent AC-5, Edge Cases complexity gate, Prior Art anti-pattern]

- **Add simple flow diagram convention to ui-brand.md** to standardize the ASCII flow notation for reuse [Source: Existing System gap, Tech Constraints compatibility]

- **Update step 8.8** to explicitly inherit the new presentation format for checker findings (grouped by severity, with justification cross-reference) rather than implicit "same as 8.3" [Source: Existing System integration point]

### Skip (Out of Scope)

- **PLAN-RATIONALE.md as a new file artifact** -- adding a `### Justification` section to the planner return message achieves the same goal without a new file. Simpler [Sources: Tech Constraints feasibility, KISS principle]

- **Changing plan-checker validation logic** -- hard invariant per BRIEF.md [Source: User Intent scope boundaries]

- **Changing planner planning algorithm** (task generation, wave assignment, dependency analysis) -- invariant. Only the return format changes [Source: User Intent IR-3/IR-6]

- **Separating justification-only revisions from the 3-iteration counter** -- Edge Cases raised this but it adds complexity to loop mechanics with unclear benefit. The 3-iteration limit is rarely hit [Source: Edge Cases, single-source, low likelihood]

- **Review/execute stage presentation changes** -- flagged as follow-up in BRIEF.md, not this feature [Source: User Intent scope boundaries]

### Investigate Further

- **AskUserQuestion rendering limits** -- before building long narrative content, test how the tool handles 50+ line question bodies in practice. May constrain narrative length [Source: Edge Cases, Tech Constraints]

- **Re-spawn justification freshness** -- when user requests changes (8.6 -> 8.5 -> re-spawn), the new planner must regenerate justification. The re-spawn prompt in plan.md revision mode needs to explicitly request justification regeneration. Planner-reference.md Revision Mode section currently has no mention of this [Source: Edge Cases, Existing System planner revision mode gap]

- **Checker-vs-planner contradiction UX** -- when checker findings contradict planner justification, the user needs reconciliation context. No gatherer proposed a concrete UX pattern for this. Low frequency but high confusion when it occurs [Source: Edge Cases integration failure scenario]

- **Research ambiguity sourcing** -- determine whether RESEARCH.md reliably contains parseable open questions (AC-3). If format varies, the feature should conditionally surface this layer rather than always attempting it [Source: User Intent ambiguity, Prior Art Ambiguity 3, Edge Cases boundary condition]
