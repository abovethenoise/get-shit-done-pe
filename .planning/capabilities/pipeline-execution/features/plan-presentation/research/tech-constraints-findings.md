### Hard Constraints

- **Planner agent returns results via files and a structured completion message — NOT in-context data** — The orchestrator does not receive the planner's reasoning, decisions, or self-critique content in its own context window. After `Task()` completes, the orchestrator checks for PLAN.md files on disk and reads the planner's structured return message (the "PLANNING COMPLETE" block). All justification narrative, self-critique findings, and decision rationale exist only in the planner's isolated 200k context window unless explicitly written to disk. — [Source: `/Users/philliphall/get-shit-done-pe/.planning/phases/02-agent-framework/research/TECH-CONSTRAINTS.md` section "How Agents Return Results"; pipeline-invariants.md Invariant 5 "Context Loading Via Paths Not Content"]

- **Planner self-critique findings (Round 2) are returned as a structured `findings` list in the "PLANNING COMPLETE" block** — The planner-reference.md mandates a specific return format: `## PLANNING COMPLETE` with a `### Findings` section containing findings or "None -- all issues resolved in Round 1". This is prose in the planner's final response. The orchestrator receives this text as the Task return value, making it parseable. Round 1 fixes are explicitly silent ("Do not surface Round 1 fixes to the user"). — [Source: `/Users/philliphall/.claude/get-shit-done/references/planner-reference.md` "Structured Return Formats" section, lines 377-400; "Self-Critique Protocol" section, lines 252-276]

- **The planner does NOT write justification reasoning or decision rationale to any file** — The planner writes only PLAN.md files. There is no "DECISIONS.md" or "RATIONALE.md" output in the planner's output contract. Any justification narrative must be either (a) synthesized by the orchestrator from the PLAN.md structure + self-critique findings returned in the completion message, or (b) produced by the planner as an additional output file if the output contract is expanded. — [Source: `/Users/philliphall/.claude/agents/gsd-planner.md` `writes: [PLAN.md]`; `output_format` section]

- **AskUserQuestion options are short-form labels, not rich content** — The existing plan.md Q&A in step 8.3 uses `AskUserQuestion` with options structured as brief labels: "Accept suggestion", "Edit", "Provide guidance", "Dismiss". The options field does not support multi-paragraph explanations or nested content. The `question` field can carry rich multi-line text (as shown in framing-pipeline.md step 2 which passes full requirements content in the `question` field), but option labels remain short strings. — [Source: `/Users/philliphall/.claude/get-shit-done/workflows/plan.md` lines 138-147; `/Users/philliphall/.claude/get-shit-done/workflows/framing-pipeline.md` lines 145-157]

- **AskUserQuestion is called once per finding in the current plan.md model** — Step 8.3 loops over each finding and issues one AskUserQuestion per finding. For an interactive "deep-dive" covering multiple plan areas simultaneously, this means either (a) multiple sequential AskUserQuestion calls (current pattern), or (b) a single AskUserQuestion where options represent plan areas to discuss (requires restructuring the Q&A model). The current loop-per-finding approach cannot offer a multi-area selection in one prompt. — [Source: `/Users/philliphall/.claude/get-shit-done/workflows/plan.md` lines 133-155: "For each finding... Use AskUserQuestion"]

- **ASCII/markdown renders as plain text in Claude Code terminal output** — Claude Code's response output is plain terminal text. Box-drawing characters (╔═╗║╚╝), horizontal rules (━━━, ───), and progress bars (████) from ui-brand.md render correctly as they are pure ASCII/Unicode characters. Markdown table syntax (`|---|---`) renders as text columns without actual cell borders in raw terminal output, but Claude Code's markdown renderer (if active in the UI) may render them as tables. GSD already uses both patterns (checkpoint boxes in checkpoints.md, markdown tables in final status blocks). — [Source: `/Users/philliphall/.claude/get-shit-done/references/ui-brand.md` all sections showing both box-drawing characters and markdown tables; `/Users/philliphall/.claude/get-shit-done/references/checkpoints.md` execution_protocol section]

- **plan.md step 8.6 ("User Approval") always exists and cannot be skipped** — The current workflow has an unconditional `AskUserQuestion` at 8.6 for final plan approval: "Finalize this plan?" This is the baseline interactive gate. The problem stated in the brief is that Q&A in 8.3 is skipped when there are no findings. The 8.6 gate is always present. — [Source: `/Users/philliphall/.claude/get-shit-done/workflows/plan.md` lines 159-169: "### 8.6. User Approval"]

- **plan-checker (step 8.7) is a `Task()` subagent — its output is also file-free prose** — The plan-checker's structured return formats ("VERIFICATION PASSED" / "ISSUES FOUND") are returned as text in the Task response. The checker does NOT write a file; its output lives in the orchestrator's received return value. Issues are structured YAML within the prose output. This is parseable by the orchestrator if it reads the return value carefully. — [Source: `/Users/philliphall/.claude/get-shit-done/references/checker-reference.md` "Structured Return Formats" section, lines 279-324; plan.md lines 173-188 "Handle Checker Findings"]

- **plan.md is a markdown instruction file read by Claude, not compiled code** — Changing plan.md directly changes orchestrator behavior at next invocation. There are no compilation steps, no tests to run against it, no schema validation. The only "test" is running a planning workflow and observing behavior. Additions to the checkpoint presentation section (8.3-8.6) take effect immediately on next plan.md read. The risk is instruction ambiguity or increased prompt size degrading model behavior. — [Source: `/Users/philliphall/get-shit-done-pe/.planning/capabilities/pipeline-execution/features/research-overhaul/research/tech-constraints-findings.md` "Hard Constraints" section; first-principles: GSD workflow files are AI instruction documents, not executable code]

- **Context window budget for the orchestrating model during plan checkpoint** — The plan.md orchestrator runs in the user's session context (not a fresh 200k window). By step 8, it has already read multiple files (FEATURE.md, CAPABILITY.md, RESEARCH.md, BRIEF.md, STATE.md, ROADMAP.md), processed init JSON, run CLI validation, and received the planner's completion message. Adding a justification narrative generation step AND visual flow generation before AskUserQuestion increases the orchestrator's context usage at checkpoint time. The planner-reference.md "Quality Degradation Curve" shows degradation starts at 50% context. — [Source: `/Users/philliphall/.claude/get-shit-done/workflows/plan.md` steps 1-8 (all context loaded before checkpoint); `/Users/philliphall/.claude/get-shit-done/references/planner-reference.md` "Quality Degradation Curve" table, lines 293-300]

- **PLAN.md format is invariant** — Per BRIEF.md invariants: "PLAN.md artifact format unchanged". The plan presentation overhaul must not alter the PLAN.md YAML frontmatter schema or XML task structure. The visual flows and justification narrative are orchestrator-level output (presented to user before approval), not additions to PLAN.md content. — [Source: `/Users/philliphall/get-shit-done-pe/.planning/capabilities/pipeline-execution/BRIEF.md` "Invariants" section, lines 86-90]

### Compatibility Issues

- **Justification narrative requires data the orchestrator must derive, not receive** — The orchestrator receives PLAN.md files (structured) + planner completion message (structured markdown). To generate a justification narrative ("why this order, why these tasks, why this approach"), the orchestrator must synthesize from: (a) PLAN.md wave structure / depends_on graph, (b) planner's findings list, (c) FEATURE.md requirements, (d) RESEARCH.md key findings. This derivation is feasible — the orchestrator already holds paths to all these files. But it is an additional reasoning step by the orchestrator, not a direct pass-through. If the narrative is wrong or incomplete, there is no correction mechanism before it is presented to the user. — [Source: `/Users/philliphall/.claude/get-shit-done/workflows/plan.md` step 8.1-8.2: what the orchestrator has at that point; `/Users/philliphall/.claude/get-shit-done/references/planner-reference.md` return format]

- **Round 1 silent fixes are not visible without planner protocol change** — The brief specifies "each self-critique fix shown with what/why rationale". Round 1 fixes are applied silently by the planner and never returned in its completion message. Surfacing Round 1 content requires adding it to the planner's return format (planner-reference.md change) and updating the planner agent to write it. A plan.md-only change cannot surface this data because it does not exist in any accessible artifact. — [Source: `/Users/philliphall/.claude/get-shit-done/references/planner-reference.md` lines 264-265: "Do not surface Round 1 fixes to the user"; plan.md step 8.1 only receives Round 2 findings]

- **"Always offer Q&A" conflicts with "no unresolved findings" skip logic** — Current plan.md step 8.3 loops over findings and presents Q&A per finding. If findings list is empty, the loop body never executes — no Q&A is presented (only 8.6 approval gate). The brief requires Q&A is always offered even with no findings. This requires restructuring 8.3: instead of "loop over findings", use "present key plan areas as Q&A options regardless of finding count". These are two different Q&A models — the current model is finding-reactive, the desired model is plan-proactive. — [Source: `/Users/philliphall/.claude/get-shit-done/workflows/plan.md` lines 133-155: conditional Q&A; `/Users/philliphall/get-shit-done-pe/.planning/capabilities/pipeline-execution/BRIEF.md` desired behavior #1]

- **ASCII flow diagram for plan architecture has no existing GSD template** — ui-brand.md defines checkpoint boxes, stage banners, tables, and status symbols but no flow diagram template. Any ASCII flow output for plan architecture would be ad-hoc generation by the orchestrator. The format works in terminal (all ASCII), but without a standardized template it may vary in quality and consistency across different plan shapes. — [Source: `/Users/philliphall/.claude/get-shit-done/references/ui-brand.md`: no flow diagram section present; BRIEF.md desired behavior #2 "ASCII/markdown flows to illustrate plan architecture"]

- **plan-checker output is not structured data** — The checker's "ISSUES FOUND" format is markdown prose with embedded YAML issues, not a machine-parseable JSON object. The orchestrator currently parses checker findings by reading the markdown text and extracting issue blocks. If plan presentation adds a "checker justification" layer (similar to planner justification), it faces the same synthesis-from-prose problem. The checker return format would need a structured `### Justification` section to make this reliable. — [Source: `/Users/philliphall/.claude/get-shit-done/references/checker-reference.md` "Structured Return Formats" lines 279-324: YAML issues embedded in markdown]

### Alternatives

- **Round 1 silent fixes blocked (not exposable without planner change)** → Alternative 1: expand the planner's Round 2 "Surface Issues" protocol to include a "what was fixed in Round 1" summary section in the return format. Planner already knows what it fixed — it just does not write it down. Add a `### Round 1 Fixes (summary)` field to the "PLANNING COMPLETE" return. This is a planner-reference.md addition, not a new file. Alternative 2: reframe the presentation to only surface Round 2 findings (already returned) plus an explanation of "Round 1 fixes were applied automatically — here is what changed" derived from comparing the planner's self-critique notes. — [Source: `/Users/philliphall/.claude/get-shit-done/references/planner-reference.md` "Self-Critique Protocol"; First principles: the planner has this data in context during Round 2 — it can summarize before completing]

- **"Always offer Q&A" vs finding-reactive loop** → Replace the per-finding loop (step 8.3) with a single proactive AskUserQuestion offering plan areas as selectable options: "Wave ordering + rationale", "Requirement coverage", "Approach decisions", "Findings ({N} noted)", "All good — proceed". This always shows even when findings list is empty, and consolidates the UX into one interaction instead of N sequential prompts. The result of the selection drives a focused follow-up conversation before the 8.6 approval gate. — [Source: `/Users/philliphall/.claude/get-shit-done/workflows/framing-pipeline.md` lines 145-157: existing multi-area AskUserQuestion pattern as precedent; First principles: one well-scoped prompt is more usable than N narrow prompts]

- **Orchestrator context growth from narrative generation** → If context budget at step 8 is a concern, offload justification generation to the planner (extend planner-reference.md return format). The planner runs in a fresh 200k context window and has full reasoning context at decision time — it can produce higher-quality justification than the orchestrator can derive post-hoc. The orchestrator then reads and presents the planner-generated text without re-deriving it, keeping orchestrator context growth minimal. — [Source: `/Users/philliphall/get-shit-done-pe/.planning/phases/02-agent-framework/research/TECH-CONSTRAINTS.md` "Per-Agent Window" section: "Each spawned agent gets a fresh 200k context window"; planner-reference.md "Quality Degradation Curve"]

- **No ASCII flow template in ui-brand.md** → Use the existing ` → ` arrow notation already established in GSD (CAPABILITY.md architecture spine). For plan architecture, a simple wave layout works without a new template:
  ```
  Wave 1: [Plan 01: objective] → [Plan 02: objective]
  Wave 2:                          [Plan 03: depends on 02]
  ```
  This format derives directly from PLAN.md `wave` and `depends_on` frontmatter fields and requires no new template standardization. If ui-brand.md needs a formal flow template, add it as part of this feature's deliverables. — [Source: `/Users/philliphall/get-shit-done-pe/.planning/capabilities/pipeline-execution/CAPABILITY.md` "Architecture Spine" flow notation; First principles: ASCII flows work at terminal width with simple indentation and arrows]
