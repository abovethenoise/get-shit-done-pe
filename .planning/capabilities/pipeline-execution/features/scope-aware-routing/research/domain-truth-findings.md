## Domain Truth Findings

Subject: pipeline-execution/scope-aware-routing — Make lens commands work at both capability and feature level via slug resolution routing

---

### First Principles

- **Scope must be resolved before work can be dispatched.** Any routing system that dispatches to specialized handlers must first determine the granularity of the target — is the input a container (capability) or a unit (feature)? Dispatching before scope is known produces either the wrong handler or an ambiguous handoff. In GSD's case, the slug resolver already returns `type: capability | feature`, making this determination explicit and machine-readable before any workflow runs. — [First principles: routing is a two-phase operation: classify the target, then dispatch. Classification failure contaminates every downstream stage. Analogy: HTTP routing must resolve whether `/blog/1` is a static page or a parameterized route before invoking a handler — routing before classification causes silent misroutes]

- **A two-tier hierarchy (capability → feature) requires two distinct routing behaviors, not one.** When a system organizes work into containers and items, an operation applied to a container means "apply to all items within, in dependency order." An operation applied to an item means "apply exactly here." These are structurally different dispatch paths and cannot share the same entry logic. Collapsing them into one path forces every caller to manually resolve the difference — which is what the current lens commands do, handling only the feature path. — [First principles: container-level operations are fan-out orchestration; item-level operations are direct dispatch. The distinction is not cosmetic — container fan-out requires DAG topology, dependency ordering, wave grouping, and inter-feature coordination that item dispatch does not]

- **Unknown input to a "new" entry point requires a create-or-classify decision, not a failure.** When a user provides a slug that matches nothing, two valid interpretations exist: "create a new container (capability)" or "create a new item under an existing container (feature)." Defaulting to either without asking produces silent misclassification. The system cannot infer which one is intended from the slug alone, because the slug doesn't exist yet. This is a domain invariant that holds regardless of implementation. — [First principles: slug non-existence is under-determined — it does not distinguish "new capability" from "new feature." Only user intent can resolve it. Any implementation that doesn't ask the user is guessing]

### Universal Constraints

- **Routing decisions must be idempotent with respect to existing state.** If a capability is already mid-pipeline (in-progress), routing to capability-orchestrator must not restart completed features. The orchestrator's wave logic already skips `complete` features — but the routing layer must pass through the slug accurately so the orchestrator can apply this check. Routing must not strip or fabricate state. — [First principles: any routing layer that modifies target identity (slug, type) before dispatch corrupts state management downstream; verified in capability-orchestrator.md Step 3: "Features already marked `complete` in the status are skipped"]

- **All four lens commands must share identical routing logic for capability-level dispatch.** The lens (new/enhance/debug/refactor) is a parameter passed through to the orchestrator; it does not change the routing decision itself. Routing is: resolve slug → check type → branch. The lens modifies orchestrator behavior downstream, not the routing branch. Any deviation between lens commands creates inconsistency and maintenance risk. — [First principles: routing topology is orthogonal to operation semantics. If enhance routes differently to capabilities than debug does, you have four routing systems instead of one. - **Stub creation must be idempotent (no-overwrite).** When a capability has features declared in CAPABILITY.md but not yet on disk, the system must create feature stubs before the orchestrator runs. But if stubs already exist (from a prior invocation), re-creating them would overwrite user data. Idempotence — check before create — is not optional. This constraint holds regardless of implementation language or storage medium. — [First principles: file creation as a prerequisite step must guard against clobbering existing work. Verified in FEATURE.md TC-02: "Only create if directory doesn't already exist (no overwrite)" — `/Users/philliphall/get-shit-done-pe/.planning/capabilities/pipeline-execution/features/scope-aware-routing/FEATURE.md` line 143]

- **Fuzzy slug resolution must produce exactly one confirmed match before dispatch.** The 3-tier resolver (`exact → fuzzy → fall-through`) can return `ambiguous` when multiple capabilities or features substring-match the user's input. Routing cannot proceed on an ambiguous result — the wrong target would receive a full pipeline run. The resolver's output contract enforces this: `resolved: false, reason: ambiguous` when multiple candidates exist. — [Source: `/Users/philliphall/get-shit-done-pe/get-shit-done/bin/lib/core.cjs` lines 462–469: `if (candidates.length > 1) { return { resolved: false, ..., reason: 'ambiguous' }; }`]

This means the routing branch — "is this a capability or feature?" — requires no new resolver logic, only consuming the type field that already exists. The assumption that slug-resolve is scope-aware is confirmed by code. — [Source: `/Users/philliphall/get-shit-done-pe/get-shit-done/bin/lib/core.cjs` lines 402–428: exact-match path returns `type: 'capability'` for capability matches and `type: 'feature'` for feature matches]

The assumption that "capability-level routing just needs to call the orchestrator" is validated — the orchestrator already contains the fan-out machinery. ### Domain Risks

- **Ambiguous slug at capability level silently routes to the wrong handler if the type check is bypassed.** If a developer implements routing as "if slug resolves, pass to orchestrator; if not, pass to pipeline" without checking the `type` field, a feature slug would incorrectly trigger capability-orchestrator, which would then fail to find a matching CAPABILITY.md. The resolver returns both resolution status AND type — both must be checked. Checking only `resolved: true` is insufficient. — [First principles: two orthogonal signals (resolved? and what type?) cannot be collapsed into one check. Source: core.cjs `resolveSlugInternal` returns both `resolved` and `type` as independent fields]

- **The /gsd:new "unknown slug" flow has a different decision tree than the other three lenses.** For enhance/debug/refactor, a non-matching slug is an error (the target should already exist). For /gsd:new, a non-matching slug is a valid entry point for creating something new. Treating all four lenses identically in the fall-through case would either suppress valid /gsd:new creation flows or incorrectly allow creation from enhance/debug/refactor. The lens identity must gate the fall-through behavior. — [Source: FEATURE.md FN-01: "If no match: for /gsd:new, ask 'New capability or new feature?'; for other lenses, error with suggestion to check /gsd:status" — `/Users/philliphall/get-shit-done-pe/.planning/capabilities/pipeline-execution/features/scope-aware-routing/FEATURE.md` lines 73–74]

- **Capability-level invocation of /gsd:new creates a two-step flow (discuss-capability → optional fan-out) that is unlike the three-step feature flow (discover → pipeline → done).** The /gsd:new capability path requires discuss-capability to establish the features table before any pipeline can run. If the routing layer passes control directly to capability-orchestrator without running discuss-capability first, the orchestrator will find an empty features table and error. The ordering constraint (discuss-capability precedes orchestrator for new capabilities) must be enforced at the routing layer. — [First principles: a fan-out orchestrator that reads a list to enumerate work cannot run before that list is created. Source: capability-orchestrator.md FN-02 error condition: "If features table is empty: error, suggest running discuss-capability first"]

- **Stub creation failure (e.g., filesystem permissions, malformed slug) during capability fan-out could silently skip features.** If stub creation fails for one feature but succeeds for others, the orchestrator will only dispatch to the successfully-stubbed features — silently omitting the failed one. The stub creation loop must log each result and surface failures before handing off to the orchestrator. — [First principles: fan-out systems with partial initialization produce partial results without surfacing the gap. The orchestrator relies on disk presence to enumerate features — a missing stub is an invisible gap. Source: FEATURE.md TC-02 constraint: "Log each creation: 'Created feature stub: {cap}/{feat}'" — the log exists to make creation visible but does not define failure handling]
