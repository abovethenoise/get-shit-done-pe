### Boundary Conditions

- **Empty features table in CAPABILITY.md** (zero rows after the header): capability-orchestrator reads the table, builds an empty DAG, produces zero waves, completes with "0 features processed" — no error raised; user receives a misleading "orchestration complete" with nothing done — exact trigger: `| Feature | Priority | Depends-On | Status |` header row followed immediately by no data rows — [source: `/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/capability-orchestrator.md:26-32`]

- **Slug input is empty string `""`**: `resolveSlugInternal` returns `{ resolved: false, reason: 'empty_input' }` at the first guard — [source: `/Users/philliphall/get-shit-done-pe/get-shit-done/bin/lib/core.cjs:395-397`]

- **Slug input containing `/` with exactly one segment (e.g., `"foo/"` or `"/foo"`)**: `split('/', 2)` produces `["foo", ""]` or `["", "foo"]` — empty string becomes empty slug via `generateSlugInternal`, `findFeatureInternal` returns `reason: empty_slug` — falls through to fuzzy — [source: `/Users/philliphall/get-shit-done-pe/get-shit-done/bin/lib/core.cjs:402-404`]

- **Capability with a single feature that has a `Depends-On` pointing to a non-existent feature slug**: The DAG build succeeds (the edge is added), but topological sort assigns the feature to a later wave that never fires because the dependency node doesn't exist — feature is never dispatched — exact trigger: `| my-feature | P1 | ghost-feature | exploring |` where `ghost-feature` has no directory — [First principles: DAG nodes are constructed from table rows only; phantom `Depends-On` references don't get auto-created as nodes]

- **Feature stub status field set to `exploring` in stub** (per TC-02) but feature-level pipeline expects `planning` or `specified` to proceed: template at `feature.md` line 5 uses `status: planning` as default, but TC-02 requires `exploring` for stubs — mismatch between template default and spec requirement — [source: `/Users/philliphall/get-shit-done-pe/get-shit-done/templates/feature.md:5`, `/Users/philliphall/get-shit-done-pe/.planning/capabilities/pipeline-execution/features/scope-aware-routing/FEATURE.md:147`]

### Known Issues in Ecosystem

- **LLM-as-markdown-table-parser pattern**: Using Claude to read and parse the CAPABILITY.md features table (pipe-delimited markdown) is an informal parsing strategy. Markdown tables with trailing/leading spaces in cells, optional alignment columns, or cells containing `|` (escaped or not) will silently produce wrong results. There is no structured schema validation for CAPABILITY.md. This is a systemic pattern in the GSD codebase — [First principles: no JSON schema or structured parser enforces CAPABILITY.md table format; the LLM reads it as text in capability-orchestrator Steps 1-2]

- **Claude Code Task() sequential vs parallel ambiguity**: capability-orchestrator Step 4b explicitly says "Sequential execution: process one feature at a time within each wave. Do NOT spawn parallel planners." This is a workflow-level constraint, not a runtime-enforced constraint. An LLM executing the orchestrator could misread this and spawn parallel `Task()` calls for wave features, causing concurrent writes to the same CAPABILITY.md status fields. — [source: `/Users/philliphall/get-shit-done-pe/get-shit-done/workflows/capability-orchestrator.md:80-83`]

- **State file contention during capability-level fan-out**: multiple pipeline instances per capability all read `.planning/STATE.md` as context — none write to it during execution — but if STATE.md is updated between feature runs (e.g., by an external commit), later features in the same wave get inconsistent project context with no re-read. — [First principles: each stage reads files at invocation time; no cache-busting mechanism exists]
