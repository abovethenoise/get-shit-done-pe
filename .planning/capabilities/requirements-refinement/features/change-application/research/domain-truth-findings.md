## Domain Truth Findings

### First Principles

- **Mutations against a dependency graph must follow topological order; no alternative ordering is correct.** Creates must precede operations that reference created entities; deletions must follow operations that depend on deleted entities. This is not a design choice -- it is a mathematical invariant of directed acyclic graphs. The FN-01 execution order (creates -> moves -> metadata -> reinstate -> defer -> kill features -> kill caps) is a valid topological ordering for the GSD entity graph, where capabilities contain features and features reference capabilities. Reversing any create/kill adjacency (e.g., killing a capability before killing its features) produces a corrupted state. The same principle governs database migration ordering (foreign key tables must exist before referencing tables) and package manager dependency resolution. — [First principles: topological sort is the only correct ordering for dependent operations in a DAG. This is proven in graph theory and universally applied in compilers (instruction scheduling), build systems (make), and package managers (apt/npm). See also https://en.wikipedia.org/wiki/Topological_sorting]

- **Each mutation must be atomic at the file level -- partial writes corrupt state more than failed writes.** A mutation that writes half a frontmatter block and crashes leaves an unparseable file, which is worse than a mutation that fails cleanly before writing. This is the core insight behind write-ahead logging (WAL) in databases: record intent before executing, so recovery knows what was attempted. For file-based systems without transactional guarantees, the practical equivalent is: read file, compute new content entirely in memory, write complete file in one operation. The TC-01 constraint ("each mutation is atomic at the file level") is the correct instantiation of this principle. — [https://martinfowler.com/articles/patterns-of-distributed-systems/write-ahead-log.html; https://dev.to/martinhaeusler/towards-atomic-file-modifications-2a9n]

- **An execution log written before/during execution (not only after) is a prerequisite for crash recovery and resumability.** DELTA.md serves as both an audit trail and a recovery artifact. If DELTA.md is only written after all mutations complete, a crash mid-execution leaves no record of what was applied. The WAL principle applies: log the intent (PENDING entries) before execution, update status (APPLIED/FAILED) after each mutation. This allows any interrupted run to be resumed by reading DELTA.md and skipping already-APPLIED entries. The current spec (FN-04) describes DELTA.md as output but does not specify incremental writing. This is a gap that should be addressed in planning. — [First principles: any system that executes a sequence of side-effecting operations and needs to recover from interruption must persist progress incrementally, not batch. This is the WAL pattern from database systems applied to file-based mutation sequences. See https://www.postgresql.org/docs/current/wal-intro.html]

- **Idempotency of individual mutations determines whether retry/resume is safe.** An idempotent mutation produces the same result whether applied once or multiple times. "Create capability X" is not naturally idempotent (second call fails if directory exists), but can be made idempotent with IF NOT EXISTS semantics. "Set status to exploring" is naturally idempotent. "Move feature from A to B" is idempotent only if the move checks current location. Without idempotency guarantees per mutation type, "fix and resume" (EU-02) risks double-applying a mutation that partially succeeded. Each of the 7 mutation types must be analyzed for idempotency and either made idempotent or guarded against double-application. — [First principles: idempotency is the mathematical property f(f(x)) = f(x). In distributed systems and IaC (Terraform, Ansible), idempotency is the foundational property that enables safe retry. See https://aws.plainenglish.io/what-is-idempotency-in-terraform-and-ansible-ebc2ef2e4234]

### Universal Constraints

- **CLI-first routing with direct-edit fallback creates a two-tier trust model that must be explicit in the audit trail.** Currently, only `capability-create` and `feature-create` exist as CLI routes. The remaining 5 mutation types (move, modify metadata, reinstate, defer, kill) will all use direct markdown edits flagged UNVALIDATED. This means the majority of mutations in early usage will be unvalidated. The UNVALIDATED flag is the correct architectural response -- it makes the trust boundary visible rather than hiding it. This pattern (validated path vs. best-effort fallback with explicit marking) appears in certificate validation (trusted vs. self-signed), API versioning (supported vs. deprecated), and content moderation (verified vs. unverified). The key invariant: the fallback path must never silently claim the same trust level as the validated path. — [CLI routes verified via `gsd-tools` command listing at `/Users/philliphall/get-shit-done-pe/get-shit-done/bin/gsd-tools.cjs`; pattern analysis from https://aws.amazon.com/builders-library/avoiding-fallback-in-distributed-systems/]

- **Rename/move operations are the most dangerous mutation type because tools commonly implement them as delete + create, risking data loss on partial failure.** A "move feature from capability A to capability B" that deletes from A before confirming creation in B can lose the feature entirely if creation fails. The safe pattern is: copy to destination, verify copy, then delete source (copy-verify-delete). Database migration tools face the same issue: a column rename implemented as DROP + ADD loses data. The FN-02 move mutation must use copy-then-delete semantics, never delete-then-create. — [https://stackrender.io/guides/schema-change-guide; First principles: any operation decomposed into delete + create has a window where the entity exists in neither location. Copy-verify-delete eliminates this window.]

- **Halt-on-failure with user choice is the only correct failure strategy for irreversible file mutations without automatic rollback.** Three failure strategies exist in mutation systems: (1) halt immediately (safe but loses progress), (2) skip and continue (risks dependent mutations failing), (3) rollback all changes (requires undo capability). GSD has no rollback mechanism (no snapshots, no undo log), so option 3 is unavailable. Pure halt (option 1) is safe but frustrating. The EU-02 design (halt + present options: fix/skip/abort) is the correct hybrid: it preserves safety while giving the user agency. The critical constraint: "skip and continue" is only safe if the skipped mutation has no dependents in the remaining queue. Skipping "create capability X" and then attempting "create feature X/foo" will fail. — [First principles: without rollback capability, the only safe options after failure are halt or skip-if-independent. Skip-with-dependents cascades failures. This is why Terraform's `-target` flag exists -- to scope operations when full apply is unsafe.]

- **Audit logs must capture the 5 Ws (who, what, when, where, why) to be useful for post-hoc analysis.** For DELTA.md: what = the mutation; when = timestamp; where = target file path; why = the originating CHANGESET entry ID; who = the execution context (automated vs. user-initiated). The current TC-02 spec captures what (change ID, result) and why (error message) but is thin on when (no per-entry timestamps) and where (no explicit target paths). For a mutation log consumed by refinement-artifact, the target file path is essential for generating a meaningful report. — [https://www.datadoghq.com/knowledge-center/audit-logging/; https://medium.com/@tony.infisical/guide-to-building-audit-logs-for-application-software-b0083bb58604]

### Validated Assumptions

- **The 7 mutation types cover the full CRUD surface for GSD's entity model.** GSD has two entity types (capability, feature) with lifecycle states (exploring, specified, building, shipped, deferred, killed). The 7 mutations map to: Create (create-cap, create-feature), Update (modify-metadata, reinstate, defer), Delete-equivalent (kill), and Relocate (move). There is no "merge" operation (combining two capabilities or features), which is intentional per the CAPABILITY.md invariants ("Manipulates the same artifacts as discuss-capability"). Merge would require conflict resolution logic that exceeds this feature's scope. The 7 types are sufficient for the change types that coherence-report can recommend. — [Feature spec FN-02 at `/Users/philliphall/get-shit-done-pe/.planning/capabilities/requirements-refinement/features/change-application/FEATURE.md`; entity model from CAPABILITY.md]

- **Reinstate is the most complex mutation type because it has cascading side effects beyond the target file.** Setting status to `exploring` is trivial, but clearing downstream artifacts (research, plans) that "may conflict with new requirements" (FN-05) requires knowing the artifact directory structure and making judgment calls about what conflicts. This is analogous to database cascade operations (ON DELETE CASCADE) -- the primary mutation is simple, but its cascading effects can be extensive and surprising. The spec wisely limits the cascade to research and plan artifacts while preserving EU/FN/TC sections, but the implementation must handle cases where these artifacts don't exist, are partially present, or have non-standard names. — [Feature spec FN-05; First principles: cascade operations in any system (databases, file systems, dependency graphs) amplify the blast radius of a single mutation. The more cascading side effects, the more failure modes.]

- **DELTA.md as an overwrite-per-run (not append-only) artifact is correct for this use case because refinement-artifact handles history.** Append-only logs are standard for audit trails, but DELTA.md is not a long-term audit log -- it is a per-run execution receipt consumed by refinement-artifact to build REFINEMENT-REPORT.md, which is the actual historical record. Overwriting DELTA.md on each run avoids the complexity of log rotation, run boundary detection, and stale entry filtering. The tradeoff: if refinement-artifact fails to consume DELTA.md before the next run, the previous run's results are lost. This is acceptable if the pipeline enforces sequential execution (refinement-artifact always runs after change-application). — [TC-02 spec: "Overwritten on each refinement run (not append-only -- refinement-artifact handles history)"; First principles: the durability requirement for a log depends on its consumption pattern. A log consumed exactly once by a known downstream consumer can safely be overwritten.]

### Domain Risks

- **5 of 7 mutation types will use the UNVALIDATED direct-edit path at launch, making the fallback path the de facto primary path.** The gsd-tools CLI currently has only `capability-create` and `feature-create`. Move, modify-metadata, reinstate, defer, and kill have no CLI routes. This means the "validated" path handles only 2/7 mutation types; the "fallback" handles 5/7. When the fallback is used more than the primary path, it is no longer a fallback -- it is the main implementation. The risk: UNVALIDATED direct edits bypass whatever validation the CLI routes provide (slug format, frontmatter schema, directory structure conventions). Planning must account for this reality and either (a) accept high UNVALIDATED rates initially, or (b) prioritize building CLI routes for the most dangerous mutations (move, kill) before shipping change-application. — [CLI route inventory from `gsd-tools.cjs`; FEATURE.md TC-01]

- **"Skip and continue" without dependency checking can produce cascading failures that are harder to diagnose than the original failure.** If mutation 3 (create capability X) fails and the user skips it, mutations 4 (create feature X/foo) and 5 (move feature bar to X) will also fail because capability X doesn't exist. The user then faces 3 failures instead of 1, with the root cause obscured by the cascade. The spec (EU-02) does not require dependency checking before skip. At minimum, the skip handler should warn the user about pending mutations that depend on the skipped one. — [First principles: in any ordered execution of dependent operations, skipping an operation invalidates all operations that transitively depend on it. Without dependency-aware skip, the user is making a decision without sufficient information.]

- **Direct markdown editing by an LLM agent is inherently non-deterministic -- two executions of the same mutation can produce different file content.** Unlike CLI routes (which are deterministic code), direct edits use the LLM to read, modify, and write markdown. The LLM may format frontmatter differently, adjust whitespace, reorder sections, or subtly alter content beyond the intended mutation. This is the fundamental difference between a programmatic transformation and an AI-assisted edit. The UNVALIDATED flag acknowledges this risk but does not mitigate it. Mitigation options: (a) constrain direct edits to frontmatter-only changes (parseable, verifiable), (b) diff the file before/after and reject changes outside the expected scope, or (c) use structured read-modify-write via the existing `frontmatter` CLI route for metadata-only mutations. — [First principles: LLM text generation is stochastic. Any file modification performed by an LLM rather than deterministic code introduces variance in output. This is not a bug but a fundamental property of the tool.]
