### Boundary Conditions

- **Zero actionable entries in CHANGESET.md**: After filtering out REJECT and RESEARCH_NEEDED, the actionable list is empty. Execution loop body never runs. DELTA.md must be a valid file with all-zero counts. Not an error -- just a no-op run. -- FEATURE.md FN-01: filter excludes REJECT, RESEARCH_NEEDED; feature-context edge case list

- **Exactly 1 actionable entry**: Minimal non-trivial execution. Failure handling (FN-03) on this single entry means "applied 0, failed 1, pending 0" -- the "pending entries" list shown to user is empty. User options (fix+resume, skip+continue, abort) collapse: skip+continue and abort are identical (no remaining entries). -- [First principles: N=1 simplifies but does not eliminate failure paths]

- **All 7 mutation types in a single CHANGESET.md**: Maximum complexity for the sorting algorithm. Each mutation type occupies a different position in the execution order (create caps -> create features -> moves -> metadata -> reinstate -> defer -> kill features -> kill caps). The sort must handle all 8 sort keys correctly. A single CHANGESET.md with all types is unlikely but represents the stress case. -- FEATURE.md FN-01: 8-level execution order

- **Feature move where source and target are the same capability**: A "Move feature X from cap-A to cap-A" entry is a no-op but may not be detected. `findFeatureInternal` would find the feature in cap-A, the "move" would copy it to cap-A (overwriting itself), and delete the original. Depending on implementation, this could succeed (no-op) or corrupt (delete after copy). -- [First principles: mv source == dest is undefined behavior without explicit check]

- **Capability slug with special characters after sanitization**: `generateSlugInternal` in `core.cjs:359-366` sanitizes to `[a-z0-9-]`. A CHANGESET.md entry referencing capability "My Awesome Cap!" becomes slug `my-awesome-cap`. If the CHANGESET.md action field contains the human-readable name (not the slug), the sanitization must be applied consistently. Mismatches between CHANGESET.md's reference and the actual slug cause "not found" errors. -- `core.cjs:359-366`: slug sanitization; `capability.cjs:16`: `generateSlugInternal(name)`

- **Feature with research/plan artifacts but no FEATURE.md**: Partially created feature directory (dir exists, research/ subdirectory exists, but FEATURE.md was never written). `findFeatureInternal` returns `{ found: false, reason: 'no_feature_file' }`. A reinstate targeting this feature fails because there is no FEATURE.md to update. A kill targeting it also fails. -- `core.cjs:543-545`: checks for FEATURE.md existence

- **CHANGESET.md with 50+ entries**: Each entry requires a read-validate-execute-write cycle. For 50 entries modifying different files, this is 50+ file reads and 50+ file writes. The orchestrator must track all results. Context pressure on the workflow agent is significant if it holds all results in memory. -- [First principles: linear scaling is fine computationally but context window is the real limit]

- **Reinstate clears downstream artifacts -- what counts as "downstream"?**: FN-05 says "Clear downstream artifacts: Research artifacts (if any), Plan artifacts (if any)." These are files in `research/` and `*-PLAN.md` within the feature directory. But a feature might also have `review/`, `RESEARCH.md`, `01-PLAN.md`, `02-PLAN.md` (multiple plan iterations). The spec does not enumerate which files/directories to clear. -- FEATURE.md FN-05: "Clear downstream artifacts that may conflict with new requirements"; existing features have `research/`, `RESEARCH.md`, `01-PLAN.md`, `02-PLAN.md` subdirectories per `.planning/capabilities/requirements-refinement/features/landscape-scan/`

- **CHANGESET.md entry with action field that does not map to any of the 7 mutation types**: If `changeset-parse` returns an entry with an unrecognized action (e.g., "merge capabilities" or "split feature"), FN-02's mutation type lookup fails. No default handler is specified. -- FEATURE.md FN-02: "7 mutation types supported" -- closed set, no "other" handler

- **Kill capability with active (non-killed) features**: The execution order puts kill-capabilities last (step 8), after kill-features (step 7). If the CHANGESET.md kills a capability but does not kill its features, the capability CAPABILITY.md gets status `killed` but its features remain in their current status. Features now belong to a killed capability -- an inconsistent state. -- [First principles: capability kill without feature kills creates orphaned-status features]

### Integration Failure Scenarios

- **`changeset-parse` CLI route does not exist** -> FN-01 specifies parsing via `gsd-tools changeset-parse`. This route is not yet implemented (`gsd-tools.cjs` command list has no `changeset-parse`). If change-application is built before this route, parsing must be done inline (direct markdown parsing). This is the expected state -- the route must be built as a dependency. -- `gsd-tools.cjs:123`: command list does not include `changeset-parse`; FEATURE.md FN-01: "Parse CHANGESET.md via `gsd-tools changeset-parse`"

- **`capability-create` CLI route fails (process.exit(1))** -> `error()` in `core.cjs:44-47` writes to stderr and calls `process.exit(1)`. When invoked via Bash tool from a workflow, this returns a non-zero exit code. The workflow agent sees the error message in stderr but must parse it to determine the failure reason. No structured error output (JSON with error code) is provided -- just a human-readable string. -- `core.cjs:44-47`: `error()` writes to stderr and exits

- **`feature-create` CLI route fails on partial creation** -> `feature.cjs:35-36` handles the case where the directory exists but FEATURE.md does not (partial creation). But if the directory exists AND FEATURE.md exists, it errors. A prior failed run that created the directory and file but failed on a subsequent entry leaves a "complete" feature that blocks re-creation on resume. -- `feature.cjs:27-29, 35-39`: partial creation handling

- **Orchestrator context exhaustion during long execution** -> For 20+ entries, the workflow agent accumulates execution results, file read contents, CLI outputs, and user interaction from FN-03 failure handling. Each failed entry triggers AskUserQuestion (3+ options), adding to context. After ~15 entries with 2-3 failures, the agent may hit context limits. -- [First principles: each mutation involves reading a file, executing a command, recording a result; this compounds in context]

- **CHANGESET.md written by a different version of refinement-qa** -> If the CHANGESET.md format evolves (new fields, changed structure), `changeset-parse` may produce malformed output. No version field in CHANGESET.md frontmatter to detect format mismatches. -- [First principles: schema evolution without versioning causes silent parse failures]

- **Upstream refinement-qa was aborted and CHANGESET.md has `status: partial`** -> change-application's FN-01 does not specify checking for partial status. If `changeset-parse` does not reject partial change sets, change-application would execute whatever entries exist, producing a run based on incomplete user decisions. Some entries may be missing user-initiated changes from the open phase. -- refinement-qa edge cases: incremental writes with `status: partial`

- **File system permissions prevent writes** -> `writeFileSync` throws `EACCES` if the process lacks write permissions on the target directory. No try/catch exists around file writes in the CLI routes. The process crashes with an unhandled exception rather than a structured error. -- [First principles: `writeFileSync` throws on permission errors; `capability.cjs:39` and `feature.cjs:44` use bare `writeFileSync`]

- **Git working tree has uncommitted changes to files being mutated** -> change-application modifies cap/feature files. If those files have uncommitted changes, the mutations overwrite the uncommitted work. No check for dirty working tree state before execution. -- [First principles: direct file writes do not check git status]

### Existing Error Handling (gaps)

- **`capability.cjs:19-22`**: Checks for duplicate capability and calls `error()` (hard exit) but does not return a structured error that change-application could catch and convert to FAILED status. The CLI route exits the process rather than returning an error code. -- `capability.cjs:21`

- **`feature.cjs:27-29`**: Same pattern -- duplicate feature causes hard exit, not a catchable error. Change-application calling this via Bash tool gets exit code 1 and a stderr message but no JSON error payload. -- `feature.cjs:29`

- **`frontmatter.cjs:27-29`**: YAML parse errors are caught and return `{}` with a stderr warning. But callers (like change-application doing a "modify metadata" mutation) cannot distinguish between "file has no frontmatter" and "file has corrupt frontmatter." Both return `{}`. A modify-metadata mutation on a corrupt file would silently create new frontmatter, losing whatever was there. -- `frontmatter.cjs:27-29`

- **FEATURE.md FN-02**: Specifies "determine the mutation type from the action field" but does not specify what happens when the action field does not match any of the 7 known types. No error handling for unknown mutation types. -- FEATURE.md FN-02 line 90

- **FEATURE.md FN-03**: "Report to user via AskUserQuestion" but does not specify what happens if AskUserQuestion returns empty (known Claude Code bug). A silent empty return on "fix and resume" vs. "skip and continue" vs. "abort" could cause the workflow to proceed with a default choice that the user did not make. -- FEATURE.md FN-03 lines 115-121; `hooks/gsd-askuserquestion-guard.js` catches empty but recovery path for change-application is unspecified

- **FEATURE.md FN-04**: DELTA.md is "overwritten on each refinement run" (TC-02). But if change-application is run twice without a full refinement cycle between runs, the second run overwrites the first DELTA.md. No append mode, no history. The first run's execution log is lost. -- FEATURE.md TC-02: "Overwritten on each refinement run"

- **FEATURE.md FN-05**: "Clear downstream artifacts" is vague on scope. Does not enumerate which file patterns to clear. Research artifacts could include `research/*.md`, `RESEARCH.md`. Plan artifacts could include `*-PLAN.md`, `01-PLAN.md`, `02-PLAN.md`. Review artifacts (`review/`) are not mentioned. The BRIEF.md file's status is unclear -- should it persist through reinstate? -- FEATURE.md FN-05 lines 148-154

- **No rollback mechanism**: If entry 5 of 10 fails, entries 1-4 are already APPLIED (files on disk are modified). FN-03 offers "abort" but does not undo entries 1-4. The user must manually revert if the partial application is undesirable. No git checkpoint is created before execution starts. -- FEATURE.md FN-03: abort "stop execution, mark all remaining entries as PENDING" -- no mention of reverting applied entries

### Known Issues in Ecosystem

- **Node.js `process.exit(1)` in CLI tools prevents error recovery**: The `error()` function in `core.cjs:44-47` calls `process.exit(1)`, which is the standard pattern for CLI tools. But when these CLI tools are invoked programmatically (via Bash tool from a workflow), the calling agent cannot distinguish between different failure reasons without parsing stderr text. Structured error codes or JSON error output would enable better programmatic handling. -- [First principles: CLI tools designed for human users produce human-readable errors; programmatic callers need machine-readable errors]

- **`fs.writeFileSync` is not atomic on any platform**: A crash (or `process.exit`) during `writeFileSync` can produce a truncated file. The standard mitigation (write to temp file, then `fs.renameSync`) is not used anywhere in the codebase. For change-application, a crash mid-write during a metadata modification leaves a corrupted FEATURE.md or CAPABILITY.md. -- [Node.js fs docs](https://nodejs.org/api/fs.html); refinement-artifact edge cases: "No write atomicity anywhere"

- **AskUserQuestion empty response bug**: Multiple GitHub issues (#9854, #10229, #29547, #29889). For change-application's FN-03 failure handling, an empty response to "fix and resume / skip and continue / abort" could silently choose a default action. The existing guard hook detects empty responses but the recovery path (user types directly in chat) breaks the structured option flow. -- [GH #29547](https://github.com/anthropics/claude-code/issues/29547)

- **Claude Code Bash tool stderr handling**: When a CLI command writes to stderr and exits non-zero, the Bash tool captures stderr output. But the format varies: some tools write "Error: message", others write stack traces. change-application must parse diverse error formats from `gsd-tools` CLI routes. Inconsistent error formatting across CLI routes makes programmatic error handling fragile. -- [First principles: `error()` in `core.cjs` produces "Error: {message}" but Node.js uncaught exceptions produce stack traces]

- **No `fs.cpSync` or directory copy in older Node.js**: The "move feature" mutation requires copying a feature directory from one capability to another, then deleting the original. Node.js added `fs.cpSync` in v16.7.0 with `recursive` support. If the project targets older Node versions, directory copy must be implemented manually (recursive read + write). The existing codebase does not use `cpSync` anywhere. -- [Node.js fs.cpSync docs](https://nodejs.org/api/fs.html#fscpsyncsrc-dest-options)

### Critical Design Decisions for Planner

1. **DELTA.md naming collision must be resolved before implementation.** Both change-application and refinement-artifact claim `.planning/refinement/DELTA.md`. One must be renamed. Recommendation: rename change-application's output to `EXECUTION-LOG.md`.

2. **Resume-after-failure requires incremental DELTA.md writes.** The spec implies DELTA.md is written at the end (FN-04), but fix+resume (FN-03) requires knowing what already succeeded. DELTA.md must be written incrementally, entry by entry, with a `status: partial | complete` frontmatter field.

3. **5 of 7 mutation types have no CLI route.** Only `capability-create` and `feature-create` exist. The other 5 (move, modify metadata, reinstate, defer, kill) must all use direct file edits. The planner must design the direct-edit implementation for each, using `spliceFrontmatter()` for metadata changes and filesystem operations for moves.

4. **CLI error handling is designed for humans, not programs.** `error()` calls `process.exit(1)` with a text message. change-application must treat every CLI invocation as potentially failing and parse stderr to determine the reason. Consider wrapping CLI calls in a helper that catches non-zero exits and returns structured results.
