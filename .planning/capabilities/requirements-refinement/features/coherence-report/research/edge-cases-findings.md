## Edge Cases Findings

### Failure Modes

| Failure | Likelihood | Severity | Mitigation | Source |
|---------|------------|----------|------------|--------|
| **Context window exhaustion on single-pass synthesis** -- TC-01 passes ALL scan artifacts + ALL project context + ALL capability files as content to a single agent invocation. For a 20-cap project with detailed specs, this easily exceeds the effective ~147K-152K token performance threshold (200K window minus system prompt overhead). matrix.md + dependency-graph.md + 50 finding cards + 20 CAPABILITY.md files + PROJECT.md/ROADMAP.md/STATE.md = potentially 100K+ tokens of input, leaving insufficient room for high-quality reasoning and output. | common | blocking | Budget input tokens before invocation: estimate total content size, and if it exceeds a threshold (e.g., 120K tokens), summarize lower-priority inputs (e.g., condense individual finding cards to summaries, include only executive sections of CAPABILITY.md files). Alternatively, split into two passes: root cause grouping first, then goal alignment + resolution sequence. | FEATURE.md TC-01 line 106: "Agent receives: all scan artifacts + project context files (contents, not paths)"; [Claude context limits](https://www.morphllm.com/claude-code-context-window): effective performance degrades at 147K-152K |
| **Violation of pipeline invariant #5 (paths not content)** -- The coherence-report spec explicitly requires passing contents to the synthesis agent (TC-01: "Agent receives: all scan artifacts + project context files (contents, not paths)"). This contradicts pipeline invariant #5 ("Context Loading Via Paths Not Content") which states orchestrators must pass file paths, not content. The landscape-scan feature has the same design choice, but that feature's per-pair agents are lightweight. The synthesis agent receives ALL artifacts at once -- the orchestrator must load all of them into its own context first before passing them, exhausting the orchestrator's context. | common | blocking | Either (a) break invariant #5 intentionally for this feature (document the exception and why), or (b) use a hybrid: orchestrator passes paths, but the synthesis agent reads them itself. Option (b) means the agent does file I/O, which contradicts TC-01's "No file I/O in agent" constraint. One of these constraints must yield. The least-bad option is letting the agent read files (it gets a fresh 200K window), and having the orchestrator pass only paths + a structured manifest. | `get-shit-done/references/pipeline-invariants.md:62-72` (invariant #5); FEATURE.md TC-01 line 109: "No file I/O in agent" |
| **Zero findings produces shallow output** -- FN-02 specifies that zero findings should produce a "clean bill of health" report. But the 6-section RECOMMENDATIONS.md structure (executive summary, root causes, systemic patterns, goal alignment, resolution sequence, contradictions) becomes degenerate: root causes = none, patterns = none, resolution = none, contradictions = none. The synthesis agent may hallucinate findings or produce boilerplate filler to satisfy the structure. | common | degraded | Define an explicit zero-findings template with shortened sections. Executive summary states "no findings." Root causes, patterns, resolution, contradictions sections each contain a single line: "None identified." Goal alignment still runs (can assess overall project health even without findings). Q&A agenda contains only a confirmation item. | FEATURE.md FN-02 line 76: "Zero findings: write clean bill of health with project coherence assessment" |
| **Many findings (50+) overflow agent output capacity** -- Claude Code agent output is capped at ~32K tokens. With 50+ findings, producing root cause groupings + systemic patterns + goal alignment + resolution sequence + contradictions + Q&A agenda may exceed this output limit, causing truncation. The RECOMMENDATIONS.md would be incomplete, and refinement-qa would consume a partial Q&A agenda. | rare | blocking | Pre-filter: if finding count exceeds a threshold (e.g., 30), have the orchestrator pre-group findings by type/severity before passing to the synthesis agent. This reduces the agent's work to inter-group synthesis rather than individual-finding analysis. Alternatively, paginate: synthesize top-20-severity findings in pass 1, remaining in pass 2 (but this breaks the "single-pass" constraint in FN-02). | FEATURE.md FN-02 line 68: "Single Claude invocation synthesizes the full report"; [Claude Code agent output limit](https://github.com/anthropics/claude-code/issues/10738) |
| **Missing scan artifacts (partial landscape-scan output)** -- FN-01 expects matrix.md, dependency-graph.md, and findings/ to all exist in `.planning/refinement/`. If landscape-scan was interrupted or partially failed, some artifacts may be present while others are missing. The spec does not define behavior when, e.g., matrix.md exists but findings/ is empty, or dependency-graph.md is missing entirely. | rare | degraded | FN-01 must validate presence of each expected artifact and categorize: (a) all present = proceed normally, (b) findings present but matrix/graph missing = synthesize from findings only with a warning, (c) no findings = zero-findings path, (d) only matrix/graph but no findings = error (scan incomplete, abort). | FEATURE.md FN-01 lines 52-57; landscape-scan edge cases: "coherence-report reads scan output before scan completes" |
| **Contradictory findings produce incoherent resolution sequence** -- FN-02 Section 6 explicitly surfaces contradictions. But the resolution sequence (Section 5) must still order all recommendations. If finding A says "add dependency X" and finding B says "remove dependency X," both appear in the resolution sequence. The synthesis agent must either exclude contradictions from the resolution sequence (deferring to Q&A) or include them with caveats. The spec does not clarify this. | common | degraded | Resolution sequence should exclude contradiction pairs and reference them as "deferred to Q&A." The contradictions section handles these. This keeps the resolution sequence internally consistent. | FEATURE.md FN-02 lines 75-76: contradictions section exists but relationship to resolution sequence is unspecified |
| **PROJECT.md with no validated requirements (TC-02 skip path) degrades prioritization** -- When no validated requirements exist (new project), TC-02 says "skip goal alignment and prioritize by severity + dependency impact only." But the resolution sequence in FN-02 uses goal alignment as a "ranking factor." Without it, severity + dependency impact may produce a flat ranking (many items at same severity, no dependency relationship). The resolution sequence becomes arbitrary. | rare | degraded | When goal alignment is skipped, explicitly state this in the executive summary. For tie-breaking in resolution sequence, use finding type as tertiary criterion (CONFLICT > GAP > OVERLAP > DEPENDS_ON > ASSUMPTION_MISMATCH > ALIGNMENT). | FEATURE.md TC-02 line 126: "skip goal alignment and prioritize by severity + dependency impact only" |
| **Root cause grouping when findings are ambiguous** -- FN-02 Section 2 groups findings by root cause. LLM-based root cause inference is subjective. Two findings about "missing validation" might share a root cause (systematic lack of validation contracts) or be completely unrelated (one is input validation, the other is schema validation). Over-grouping loses distinct issues; under-grouping produces a noisy report. | common | degraded | Require the synthesis agent to justify each root cause grouping with a specific causal chain. Include confidence level (high/medium/low) per grouping. Preserve individual finding IDs so refinement-qa can challenge any grouping. | [First principles: root cause analysis is inference, not deduction; LLMs lack domain-specific causal models] |
| **Q&A agenda items with no clear category** -- FN-03 categorizes agenda items as decision/informational/auto-resolve. Some items may not fit cleanly (e.g., a finding that is clearly correct but has complex implementation tradeoffs -- is it a decision or informational?). Miscategorization sends items to the wrong Q&A phase: auto-resolve items that needed discussion get applied without user input. | common | cosmetic | Default ambiguous items to "decision" (safest -- forces user confirmation). The auto-resolve category should only contain items where the synthesis agent has HIGH confidence and the recommended action has no tradeoffs. | FEATURE.md FN-03 lines 87-89: three categories with no default/fallback |

### Boundary Conditions

- **Exactly 0 findings, 0 capabilities**: Project has no capabilities at all. FN-01 loads empty capability list, empty findings. matrix.md is empty/absent. The synthesis agent receives essentially no input. Must produce a meaningful clean bill of health that says "no capabilities exist yet" rather than "no issues found." -- [First principles: empty input to synthesis = empty output or hallucination]

- **Exactly 1 finding**: Single finding cannot form a "systemic pattern" (Section 3 requires cross-cutting patterns across multiple root causes). Root cause section has exactly 1 entry. Resolution sequence has exactly 1 item. Contradictions section is necessarily empty. Q&A agenda has 1 item. The report structure is valid but sparse. -- FEATURE.md FN-02: "Systemic Patterns: Cross-cutting patterns that span multiple root causes"

- **All findings are same type and severity**: If all 15 findings are GAP/HIGH, root cause grouping and prioritization become homogeneous. Resolution sequence ordering is arbitrary since all items have identical severity and type. Goal alignment becomes the sole differentiator. -- [First principles: sorting by identical keys produces undefined order]

- **PROJECT.md missing entirely**: FN-01 reads PROJECT.md for goals/requirements. If PROJECT.md does not exist (project not initialized properly), the synthesis agent has no goal context. TC-02 skip path only covers "no validated requirements" -- missing PROJECT.md is a broader failure. Must distinguish "PROJECT.md exists but has no requirements" from "PROJECT.md does not exist." -- FEATURE.md FN-01 line 50: "Read project context: PROJECT.md (goals, requirements)"

- **ROADMAP.md or STATE.md missing**: FN-01 lists these as context inputs. Not every project has them (STATE.md is created during pipeline execution, ROADMAP.md is optional). Missing files should be treated as absent context, not errors. -- FEATURE.md FN-01 line 50

- **Finding card references nonexistent capability**: A FINDING-*.md card references a capability slug that no longer exists (capability was killed between scan and synthesis). The synthesis agent's goal alignment and resolution sequence reference a ghost capability. -- [First principles: landscape-scan and coherence-report run in sequence but artifacts are files that can be externally modified between steps]

- **Very large RECOMMENDATIONS.md**: With 50+ findings, the RECOMMENDATIONS.md output could be 10K+ lines. refinement-qa must parse the Q&A agenda section from this file. If the file is very large, the Q&A agent's context loading also faces pressure. -- [First principles: downstream consumer inherits upstream size problems]

- **Finding card IDs with gaps (non-sequential)**: After landscape-scan's dedup/consolidation, finding IDs may have gaps (FINDING-001, FINDING-003, FINDING-007). Root cause section references these IDs. The synthesis agent must not infer missing findings from gaps. -- landscape-scan FEATURE.md FN-04: consolidation can remove findings

- **Agent output contains malformed markdown**: The synthesis agent produces RECOMMENDATIONS.md content that the orchestrator writes to disk. If the agent produces malformed markdown (unclosed code blocks, broken tables, mangled frontmatter), the output file is technically present but unparseable by refinement-qa. -- [First principles: LLM outputs are non-deterministic; structural validation of markdown output is not specified]

### Integration Failure Scenarios

- **landscape-scan fails mid-scan** -> `.planning/refinement/` contains partial findings (FINDING-001 through FINDING-005 but scan was expected to produce more). matrix.md may be incomplete. coherence-report's FN-01 loads whatever exists and synthesizes from it. The resulting RECOMMENDATIONS.md is based on incomplete data but appears complete. The user has no signal that the input was partial. Mitigation: check for a scan-completion marker or verify expected pair count vs. checkpoint count before proceeding. -- landscape-scan edge cases finding: "coherence-report reads scan output before scan completes"

- **landscape-scan produces findings with invalid schema** -> FINDING-*.md cards missing required fields (type, severity, affected capabilities). FN-01 loads them as-is. The synthesis agent tries to group by root cause but cannot categorize findings without type/severity. Output has malformed root cause groupings. Mitigation: validate finding card schema in FN-01 before passing to synthesis; reject or flag malformed cards. -- [First principles: upstream outputs are LLM-generated; schema validation is required at every boundary]

- **Orchestrator context exhaustion** -> If the orchestrator loads all artifact contents into its own context (to pass to the synthesis agent per TC-01), the orchestrator's context is consumed. After synthesis, the orchestrator has insufficient context remaining to write RECOMMENDATIONS.md, trigger refinement-qa, or handle errors. This is the core tension with invariant #5. -- `get-shit-done/references/pipeline-invariants.md:66`: "Passing content bloats the orchestrator context"

- **Synthesis agent rate-limited or times out** -> Single-pass synthesis (FN-02) uses one Claude invocation. If that invocation is rate-limited or times out, the entire coherence-report feature fails with no partial output. Unlike landscape-scan (which checkpoints per pair), there is no intermediate state to resume from. -- [First principles: single invocation = single point of failure with no resume capability]

- **`.planning/refinement/` directory does not exist** -> If landscape-scan was never run (user runs coherence-report out of order), the input directory is missing. FN-01 must detect this and produce a clear error ("run landscape-scan first") rather than failing with a file-not-found error. -- FEATURE.md FN-01 line 53: reads from `.planning/refinement/`

- **refinement-qa cannot parse Q&A agenda** -> FN-03 embeds a Q&A agenda in RECOMMENDATIONS.md. If the format is ambiguous or the synthesis agent structures it differently than expected, refinement-qa's FN-01 ("Parse the Q&A agenda section") fails. There is no formal schema for the Q&A agenda format -- it depends on the synthesis agent producing a specific structure. -- refinement-qa FEATURE.md FN-01 line 71: "Parse the Q&A agenda section (final section of RECOMMENDATIONS.md)"

- **Capability files changed between landscape-scan and coherence-report** -> FN-01 loads capability files "for reference." If a capability was modified, killed, or created between when landscape-scan ran and when coherence-report runs, the findings reference stale state. The synthesis agent may produce recommendations against capabilities that no longer match the findings. -- [First principles: no locking or snapshot mechanism between pipeline stages]

### Existing Error Handling (gaps)

- **No validation of scan artifact presence in FN-01**: The spec says "Read all scan artifacts from `.planning/refinement/`" but does not specify behavior when artifacts are missing. `safeReadFile` (core.cjs:51) returns null, but FN-01 does not specify null-handling for each artifact type. -- FEATURE.md FN-01 lines 52-57

- **No schema validation for finding cards**: Finding cards are loaded by content (FN-01) but never validated against the expected schema (type, severity, affected capabilities, etc.). Malformed cards pass silently into the synthesis agent. -- FEATURE.md FN-01 line 56: "findings/ (all FINDING-{id}.md cards)" -- no validation step

- **No output validation for RECOMMENDATIONS.md**: The orchestrator writes whatever the synthesis agent returns. No check that the output contains the 6 required sections, has valid Q&A agenda items, or references real finding IDs. Malformed output propagates to refinement-qa. -- FEATURE.md FN-02 line 69: required structure listed but no validation specified

- **No input size guard**: No mechanism to estimate or limit total input tokens before invoking the synthesis agent. If content exceeds the agent's effective context window, the agent silently degrades rather than failing fast. -- FEATURE.md TC-01: no size constraints specified

- **No scan-completion verification**: FN-01 does not verify that landscape-scan completed successfully before loading artifacts. Partial scan output is consumed without warning. -- FEATURE.md FN-01 line 46: "Trigger from refinement orchestrator after landscape-scan completes" but no verification mechanism

### Known Issues in Ecosystem

- **Claude Code subagent 32K output token cap**: Agent subprocesses may hit output limits regardless of model configuration. A comprehensive RECOMMENDATIONS.md for a project with many findings could exceed this cap, producing truncated output. The orchestrator receives truncated content and writes an incomplete file. -- [GitHub issue #10738](https://github.com/anthropics/claude-code/issues/10738)

- **Claude Code context compaction during long operations**: If the orchestrator accumulates significant context while loading artifacts and managing the synthesis, auto-compaction may trigger (at 64-75% context usage), potentially losing important state mid-operation. -- [Claude Code context management](https://www.morphllm.com/claude-code-context-window): auto-compaction triggers at 64-75% capacity

- **LLM root cause inference is unreliable without domain-specific grounding**: Research on automated root cause analysis shows that LLMs produce plausible but often incorrect root cause groupings when operating without explicit domain constraints. The synthesis agent may conflate unrelated findings that share surface-level vocabulary. Without domain-specific prompting, accuracy degrades with finding count. -- [First principles: LLM clustering by semantic similarity != causal grouping; no RCA-specific prompting strategy is specified in TC-01]

- **Single-pass synthesis has no self-correction**: Unlike multi-pass architectures (e.g., chain-of-verification from Meta, 2023), a single Claude invocation cannot verify its own root cause groupings or goal alignment assessments. Errors in early sections (root causes) propagate to later sections (resolution sequence) within the same generation. -- [First principles: single-pass = no verification loop; referenced in `.planning/phases/05-documentation/05-RESEARCH.md:20` as relevant to verification accuracy]

- **Markdown table rendering at scale**: Resolution sequence and root cause sections may produce large markdown tables. Claude Code's rendering and the downstream Q&A agent's parsing of these tables can be fragile when tables exceed ~20 rows or contain cells with multi-line content. -- [First principles: zero runtime deps means hand-parsed markdown; no markdown AST library available]
