### Boundary Conditions

- **Exactly 0 agenda items**: Q&A agenda section exists in RECOMMENDATIONS.md but contains no items (clean bill of health). FN-02 structured loop iterates zero times. Open phase becomes the entire interaction. CHANGESET.md must handle having only USER_INITIATED entries (or no entries at all). -- coherence-report FEATURE.md EU-01: "Zero-findings case produces a 'clean bill of health' report"

- **Exactly 1 agenda item**: Minimal valid Q&A. Header format "QA 1/1" works. Contradiction pairs impossible (need >= 2 items). Open phase still runs. Exit loop ("Does this look good?") fires after 1 item -- feels abrupt. -- [First principles: N=1 is the simplest non-trivial case]

- **50+ agenda items**: At 50 items with ~3 AskUserQuestion calls each (presentation + resolution + possible follow-up for research/reject), that is 150+ AskUserQuestion invocations in a single session. User fatigue is severe. Context window pressure is extreme (see failure mode above). The header "QA 50/52" is 9 chars (safe). -- FEATURE.md FN-02: "every agenda item"; [First principles: 50+ sequential questions in one session exceeds human attention span]

- **Agenda items total exceeds 99**: Header "QA 100/150" is 11 chars (within 12-char limit). "QA 1000/1500" is 13 chars (EXCEEDS limit). Projects with 100+ coherence findings are theoretically possible but practically indicate a project that needs restructuring, not item-by-item Q&A. -- `get-shit-done/references/questioning.md:110`: 12-char header hard limit

- **User provides empty text for "Research needed" or "Reject/Modify"**: FN-02 requires user-provided text for these resolutions. If user selects the option but provides no text (empty string, whitespace only), the resolution is recorded without reasoning. change-application and refinement-artifact receive entries with empty reasoning fields. -- FEATURE.md FN-02: "user provides context/question text" and "user provides reasoning"

- **User revisits the same item multiple times during open phase**: FN-03 allows revisiting structured decisions. No limit is specified on how many times the same item can be revisited. User could flip an item between accept/reject indefinitely, producing a revision log with N entries for the same finding ID. CHANGESET.md dedup must keep only the final resolution. -- FEATURE.md FN-03: "Revisit a structured decision and change their resolution"

- **Contradiction pair where one item was already resolved and user wants to change it**: User accepts item A at position 5. At position 20, contradiction partner B is presented. User wants to reject B AND reverse their acceptance of A. This requires revisiting item A inline (during structured phase, not open phase). The spec says revisiting happens in open phase only. -- FEATURE.md FN-02: contradictions as paired items; FN-03: revisiting in open phase

- **CHANGESET.md with only RESEARCH_NEEDED entries**: User marks every item as "Research needed." CHANGESET.md has entries but zero actionable ones. change-application filters all out. DELTA.md shows 0 applied. Functionally identical to "user rejects everything" but semantically different (user wants investigation, not rejection). Next refinement run has no delta to compute against. -- FEATURE.md FN-04: 6 entry types; change-application FEATURE.md FN-01: RESEARCH_NEEDED excluded from actionable

- **RECOMMENDATIONS.md has Q&A agenda but no findings section**: Malformed output from coherence-report where the Q&A agenda references finding IDs that don't exist in the findings section. FN-01 loads the agenda but cannot load supporting context (finding cards). Presenting items without their full finding context produces degraded Q&A. -- FEATURE.md FN-01: "Load supporting context: finding cards from `findings/`"

- **Open phase with no structured phase preceding it (zero-item agenda)**: If the structured phase is skipped (zero items), the open phase is the first user interaction. The "exit loop" pattern ("Does this look good?") makes less sense when there's nothing to review. The question becomes "Anything to discuss?" not "Does this look good?" -- [First principles: exit confirmation assumes prior work to confirm]

### Integration Failure Scenarios

- **coherence-report produces malformed RECOMMENDATIONS.md** -> FN-01's "Parse the Q&A agenda section" fails. No agenda items are loaded. The structured phase has nothing to present. Depending on error handling, the workflow either crashes (unhandled parse error) or silently enters open phase with no context. User has no visibility into why structured items are missing. Mitigation: validate RECOMMENDATIONS.md structure in FN-01; if parse fails, present the raw RECOMMENDATIONS.md content to the user and ask them to proceed manually or re-run coherence-report. -- FEATURE.md FN-01: "Parse the Q&A agenda section (final section of RECOMMENDATIONS.md)"; coherence-report edge cases: "No output validation for RECOMMENDATIONS.md"

- **Supporting artifacts missing (findings/, matrix.md, dependency-graph.md)** -> FN-01 loads "supporting context" from these files. If landscape-scan was partial or files were deleted, FN-01 has agenda items that reference finding IDs with no corresponding finding cards. Each AskUserQuestion presents an item with "Finding summary" that cannot be populated. Mitigation: for each agenda item, check if referenced finding IDs have corresponding cards. If missing, present the item with "[Finding card not available -- recommendation based on synthesis only]." -- FEATURE.md FN-01: "Load supporting context: finding cards from `findings/`, matrix from `matrix.md`, dependency graph from `dependency-graph.md`"

- **`refinement-write` CLI route does not exist yet** -> TC-01 specifies "Change set writing uses `refinement-write` CLI route from refinement-artifact." This route is planned but not built. If refinement-qa is implemented before refinement-artifact's CLI routes, CHANGESET.md cannot be written via the prescribed path. Mitigation: fall back to direct file write with a TODO marker, or implement `refinement-write` as a prerequisite. -- FEATURE.md TC-01: "Change set writing uses `refinement-write` CLI route from refinement-artifact"

- **`changeset-parse` CLI route does not exist yet** -> TC-02 specifies a new CLI route "changeset-parse -- reads CHANGESET.md, returns JSON." This is consumed by change-application. If the format written by refinement-qa doesn't match what `changeset-parse` expects, downstream parsing fails silently (returns malformed JSON) or loudly (crashes). Mitigation: define CHANGESET.md schema first, implement `changeset-parse` and refinement-qa's writer against that schema simultaneously. -- FEATURE.md TC-02: "Parseable by gsd-tools.cjs (new CLI route: `changeset-parse`)"

- **Orchestrator context exhaustion before Q&A completes** -> The refinement orchestrator loads RECOMMENDATIONS.md + supporting artifacts into its context to present items. Per pipeline invariant #5, orchestrators should pass paths not content, but the Q&A orchestrator must read content to present it in AskUserQuestion questions. After loading a large RECOMMENDATIONS.md (potentially 5K-10K tokens) plus 30+ finding cards, the orchestrator's context is already 30-50% consumed before Q&A starts. Each Q&A round adds more. Mitigation: load finding cards lazily (one at a time, per item), not all upfront. Present only the relevant finding card for the current item. -- `pipeline-invariants.md:62-66`: "Passing content bloats the orchestrator context"

- **Session timeout or `/clear` during structured phase** -> No checkpoint mechanism exists for Q&A progress. All resolutions are lost. User must restart the entire Q&A from item 1. For a 30-item agenda, this means re-doing potentially 20 minutes of work. Mitigation: see "User aborts mid-Q&A" failure mode above. -- [First principles: Claude Code sessions can end at any time; `get-shit-done/references/continuation-format.md` describes next-step handoff but not mid-workflow resume]

- **DELTA.md naming collision with change-application** -> refinement-artifact's edge cases research identified that both refinement-artifact and change-application write to `.planning/refinement/DELTA.md`. If refinement-qa writes CHANGESET.md, and change-application writes DELTA.md, and refinement-artifact also writes DELTA.md, the naming collision affects the entire pipeline. refinement-qa's output (CHANGESET.md) is safe, but the downstream collision means refinement-qa's work can be partially lost in the overwrite. -- refinement-artifact edge cases: "DELTA.md Naming Collision (Critical Design Issue)" at `.planning/capabilities/requirements-refinement/features/refinement-artifact/research/edge-cases-findings.md:52-61`

### Existing Error Handling (gaps)

- **FEATURE.md FN-01**: Specifies "Read `.planning/refinement/RECOMMENDATIONS.md`" and "Parse the Q&A agenda section" but does not specify behavior when the file is missing, empty, or has no Q&A agenda section. No fallback path defined. -- FEATURE.md FN-01 lines 69-74

- **FEATURE.md FN-02**: Specifies "3 resolution options" but does not handle the case where AskUserQuestion returns an unexpected value (not one of the 3 options). User could type a freeform response instead of selecting an option. The guard hook catches empty responses but not unexpected option values. -- FEATURE.md FN-02 lines 86-89; `hooks/gsd-askuserquestion-guard.js:20-24` only checks for empty

- **FEATURE.md FN-03**: "Revisit a structured decision" requires finding the item by some identifier, but no item lookup mechanism is specified. How does the user reference "that item about the dependency gap"? By finding ID (user must remember)? By item number (fragile if order changed)? By description (fuzzy match, unreliable)? -- FEATURE.md FN-03 line 107

- **FEATURE.md FN-04**: "Entries sorted by type, then by finding severity" but finding severity comes from the finding card, which is a coherence-report artifact. User-initiated items (USER_INITIATED, ASSUMPTION_OVERRIDE) have no severity. How are they sorted relative to report-driven items? -- FEATURE.md FN-04 lines 127-128

- **No validation of CHANGESET.md output structure**: FN-04 specifies the format but no verification step. If the orchestrator writes malformed CHANGESET.md (e.g., missing required fields on an entry), change-application's `changeset-parse` fails downstream. No feedback loop catches this before the user advances. -- FEATURE.md FN-04; TC-02 line 160: "Parseable by gsd-tools.cjs"

- **No handling for the Q&A workflow file location ambiguity**: TC-01 says "Workflow file: `workflows/refinement-qa.md` (or embedded in the main refinement workflow)." This ambiguity means the planner must decide, but either choice has implications: separate file means a new Task spawn (fresh context -- good for long Q&A), embedded means Q&A runs in the orchestrator's context (bad for long sessions). -- FEATURE.md TC-01 line 142

### Known Issues in Ecosystem

- **AskUserQuestion empty response bug (multiple reports)**: This is the most frequently reported Claude Code bug, with at least 6 distinct GitHub issues (#9854, #10229, #10400, #12672, #29547, #29733, #29889). The bug manifests when AskUserQuestion is called from skills/plugins, when bypass permissions are enabled, or on the first call in a session. The existing guard hook mitigates but does not prevent. For a 50-item Q&A session, even a 5% failure rate means 2-3 items silently auto-resolved. -- [GH #29547](https://github.com/anthropics/claude-code/issues/29547); [GH #10229](https://github.com/anthropics/claude-code/issues/10229); [GH #29889](https://github.com/anthropics/claude-code/issues/29889)

- **Claude Code context compaction during long conversations**: Auto-compaction triggers at 64-75% context capacity and can hang indefinitely for large conversations (GH #19567). A 50-item Q&A session with full finding context per item will trigger compaction, potentially losing earlier resolutions that were stored only in context (not written to disk). -- [GH #19567](https://github.com/anthropics/claude-code/issues/19567); [Claude Code context management](https://www.morphllm.com/claude-code-context-window)

- **AskUserQuestion popup blocks reading Claude's response in plan mode**: When AskUserQuestion fires, the popup can obscure the context that the user needs to read before answering (GH #27998). For refinement-qa, the finding summary + recommendation must be readable BEFORE the options appear. If the popup covers the summary, users make uninformed decisions. -- [GH #27998](https://github.com/anthropics/claude-code/issues/27998)

- **No native free-text input in AskUserQuestion**: AskUserQuestion supports selectable options only. The "Research needed" and "Reject/Modify" paths require user-provided text (reasoning, questions). This requires a two-step interaction pattern (select option, then type text in chat) that is not natively supported. The user must know to type after selecting, which is not intuitive. -- `get-shit-done/references/questioning.md:99-128`: options-based interaction model

- **Claude Code agent output 32K token cap**: If refinement-qa runs as a subagent (Task), its output is capped at 32K tokens. A CHANGESET.md with 50+ entries, each with reasoning, could exceed this. If refinement-qa runs in the orchestrator, this cap doesn't apply but context pressure does. -- [GH #10738](https://github.com/anthropics/claude-code/issues/10738)

### Key Design Tension: Incremental Write vs. Single Output

The spec defines CHANGESET.md as a single output written after all phases complete (FN-04). But:

1. **Context exhaustion** requires incremental writes (don't hold 50+ items in context)
2. **Session interruption** requires incremental writes (don't lose progress)
3. **Revisiting decisions** requires either in-memory state (lost on interruption) or re-reading from disk (requires parsing own partial output)
4. **`changeset-parse`** expects a complete, well-formed file (not a partial one)

The planner must resolve this tension. The recommended approach: write CHANGESET.md incrementally with `status: partial` in frontmatter. On completion, update to `status: complete`. `changeset-parse` refuses to parse partial change sets. Resume path detects partial change sets and continues from last resolved item.
