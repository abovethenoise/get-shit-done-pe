## Domain Truth Findings

### First Principles

- **Pairwise comparison is inherently O(n^2) and cannot be fully avoided for complete coverage.** For n capabilities, full pairwise analysis requires n(n-1)/2 comparisons. Graph theory proves that to rate a set of n objects, at least n-1 independent comparisons (a spanning tree) are needed, but complete coverage requires the full n(n-1)/2. Any reduction below full pairwise trades coverage for speed. The tiered strategy (TC-03) is therefore not optional overhead -- it is a mathematical necessity for projects above ~20 capabilities. — [First principles: combinatorial mathematics; validated by graph theory literature on minimum spanning trees (https://en.wikipedia.org/wiki/Minimum_spanning_tree) and fast approximate hierarchical clustering (https://pmc.ncbi.nlm.nih.gov/articles/PMC2561018/)]

- **Sequential analysis with accumulated context produces higher-quality findings than parallel analysis, but creates an ordering dependency.** When later pairs can reference findings from earlier pairs ("this is another symptom of FINDING-003"), the analysis agent builds a richer model over time. However, this means pair ordering affects output quality -- pairs analyzed first have zero prior context. This is a fundamental tradeoff: sequential accumulation improves dedup quality at the cost of order sensitivity. — [First principles: information theory -- each additional observation constrains the hypothesis space. An analyzer with prior findings has strictly more information than one without. The tradeoff is that the first pair analyzed has the least context, creating an inherent asymmetry.]

- **LLMs are strong at detecting the existence of conflicts but weak at precisely localizing conflicting segments and communicating them.** ConInstruct benchmark shows Claude-4.5-Sonnet achieves 87.3% F1 on conflict detection, but explicitly alerts users to conflicts in only 45% of cases. When instructions contain 1-2 conflicts, GPT-4o silently generates a response 97.5% of the time without flagging the conflict. This means the scan-pair agent must be explicitly prompted to output structured findings -- LLMs will not spontaneously surface conflicts without strong prompting. — [https://arxiv.org/html/2511.14342]

- **Deduplication of N symptoms to M root causes is a clustering problem, not a filtering problem.** Igor (CCS 2021) demonstrates that naive deduplication yields bug counts at least one order of magnitude larger than root-cause clustering. The key insight: two findings may look different at the surface (different capability pairs, different wording) but trace to the same root cause. Effective dedup requires comparing the causal structure of findings, not just their surface text. — [https://dl.acm.org/doi/10.1145/3460120.3485364]

### Universal Constraints

- **Cross-artifact consistency verification is semantically harder than syntactic checking, and no tool achieves zero false positives.** TUM research on LLM-based multi-artifact consistency verification confirms that identifying inconsistencies requires deep understanding of implicit relationships across artifacts, not just syntactic matching. Different artifact types (specs vs. code vs. docs) present varying reliability levels. This means landscape-scan findings will inherently include false positives, and the downstream refinement-qa step is not optional -- it is the necessary human-in-the-loop filter. — [https://aet.cit.tum.de/research/publications/dietrich2025koli.pdf]

- **Pre-filtering heuristics must be recall-optimized, not precision-optimized.** When using mgrep or textual proximity to reduce pairs (TC-03 medium tier), a false negative (missing a real conflict) is worse than a false positive (analyzing a pair that turns out fine). Fast approximate hierarchical clustering literature confirms that quality degrades when the heuristic prunes too aggressively. The pre-filter should err on the side of including marginal pairs. — [First principles: in a two-stage pipeline (pre-filter then analyze), the first stage sets an upper bound on recall. A pair excluded by the pre-filter can never be discovered in the analysis stage. False positives only cost LLM calls; false negatives cost missed findings.]

- **Checkpoint-resume requires idempotent units of work.** For resumable scanning (EU-02), each pair analysis must be a self-contained, idempotent operation -- re-running a pair with the same inputs must produce equivalent output. LangGraph's Time Travel pattern demonstrates this: capturing each state transition transforms execution into a replayable state machine. The checkpoint granularity (per-pair) is correct because pairs are the natural atomic unit. — [https://dev.to/sreeni5018/debugging-non-deterministic-llm-agents-implementing-checkpoint-based-state-replay-with-langgraph-5171; First principles: idempotency is required for any resumable pipeline -- without it, partial results from a crashed run may be inconsistent with results from the resumed run.]

- **LLMs exhibit recency bias in conflict detection.** Conflicting constraints appearing later in an instruction are more likely to be satisfied. For landscape-scan, this means the order in which capability contents are presented to the scan-pair agent (A then B, vs B then A) may systematically bias which capability's constraints are "preferred." Mitigation: present both capabilities symmetrically or explicitly instruct the agent to treat both equally. — [https://arxiv.org/html/2511.14342]

### Validated Assumptions

- **A fixed taxonomy of finding types is viable and aligns with prior work.** The FEATURE.md taxonomy (CONFLICT, GAP, OVERLAP, DEPENDS_ON, ASSUMPTION_MISMATCH, ALIGNMENT) maps well to established categories in requirements engineering and process coherence checking. aProCheCk (Springer 2025) uses similar categories for multi-level process documentation coherence. The ConInstruct benchmark validates that intra-constraint and inter-constraint conflicts are distinct categories requiring different detection strategies. Evidence: both academic and industrial tools converge on a small, fixed set of relationship types rather than open-ended classification. — [https://link.springer.com/article/10.1007/s44311-025-00024-6; https://arxiv.org/html/2511.14342]

- **Categorical confidence (HIGH/MEDIUM/LOW) is more actionable than numeric scores for human review.** The decision to use categorical confidence (Decision 2026-03-05) aligns with evaluation literature showing that LLM-generated numeric confidence scores are poorly calibrated and difficult for humans to interpret. Categorical ratings reduce the illusion of precision and map directly to action thresholds (HIGH = must address, MEDIUM = should review, LOW = informational). — [First principles: numeric confidence from LLMs lacks a calibrated probability interpretation -- a "0.7" from one pair analysis is not comparable to "0.7" from another. Categories enforce consistent interpretation.]

- **70-95% of software defects are triggered by pairwise interactions of just two variables.** This NIST finding from combinatorial testing validates the pairwise approach: analyzing capability pairs (not triples or higher-order combinations) captures the vast majority of coherence issues. Higher-order analysis (A x B x C) has diminishing returns and exponential cost. — [https://www.qatouch.com/blog/pairwise-testing-guide/; NIST combinatorial testing research]

### Domain Risks

- **Context window exhaustion for large capabilities.** Per-pair analysis (FN-02) sends both capabilities' full contents plus accumulated prior findings to the agent. For capabilities with extensive specs (multiple features, detailed requirements), this payload can exceed practical context limits. The FEATURE.md spec says agent receives "contents, not paths" but does not address what happens when combined contents exceed the window. Mitigation needed: content summarization or selective inclusion for large artifacts. — [First principles: LLM context windows are finite; accumulated findings grow linearly with completed pairs. At pair N, context = cap_A_size + cap_B_size + sum(findings_1..N-1). This sum is unbounded without explicit management.]

- **Checkpoint invalidation is a deferred problem that will bite.** EU-02 explicitly marks "invalidating checkpoints when underlying artifacts change" as out of scope. But in practice, users will modify capability files between interrupted scan runs. A resumed scan will then mix stale findings (from checkpointed pairs analyzed against old content) with fresh findings. This creates silent inconsistency in the output. The risk is proportional to how often users actually interrupt and resume scans. — [Feature spec EU-02 out-of-scope note; First principles: any cache without an invalidation strategy will eventually serve stale data.]

- **mgrep pre-filtering quality determines the ceiling of medium-tier scan quality.** For 21-50 capability projects, mgrep textual proximity is the sole filter deciding which pairs reach Claude. If mgrep's semantic matching is weak for the project's domain vocabulary (e.g., capabilities use different terminology for related concepts), real conflicts will be silently excluded. There is no fallback or validation that the pre-filter is working correctly. — [First principles: in a funnel architecture, the narrowest stage determines throughput quality. mgrep operates on textual co-occurrence, not semantic understanding -- "authentication" and "access control" may have low textual overlap but high semantic relationship.]
