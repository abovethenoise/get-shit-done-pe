## Edge Cases Findings

### Failure Modes

| Failure | Likelihood | Severity | Mitigation | Source |
|---------|------------|----------|------------|--------|
| Malformed RECOMMENDATIONS.md breaks snapshot (FN-01) | common | blocking | Validate RECOMMENDATIONS.md structure before snapshotting; if parse fails, treat as "first run" (null snapshot) and warn user rather than crashing | FEATURE.md FN-01; `frontmatter.cjs:27` shows parse errors become `{}` but no analogous guard exists for markdown body parsing |
| Finding IDs renumbered between runs invalidates delta (FN-03) | common | degraded | Delta computation must match by finding ID string (`FINDING-001`). If IDs are renumbered (e.g., consolidation removes FINDING-003, next run assigns FINDING-003 to a new issue), delta will show a "changed" finding that is actually a different finding entirely. Mitigation: include a content hash or fingerprint alongside ID for identity verification | FEATURE.md FN-03 + TC-02: "Finding identity is by ID" |
| Orphaned finding files in `findings/` directory | common | degraded | If a previous run wrote FINDING-001 through FINDING-010, and the current run only produces FINDING-001 through FINDING-007, files FINDING-008 through FINDING-010 remain on disk since `writeFileSync` creates/overwrites but does not delete. FN-02 says "overwrite all files on each run" but does not specify deletion of stale files. Mitigation: `refinement-init` must clear `findings/` directory before writing new scan output | FEATURE.md FN-02: "Overwrite all files on each run (current state, not cumulative)" |
| Previous snapshot corrupted (partial write from killed process) | rare | blocking | If a prior run was interrupted mid-write (e.g., RECOMMENDATIONS.md is half-written), `readFileSync` returns truncated content. Snapshot will contain partial data, producing nonsensical delta. Mitigation: validate snapshot content has expected structure markers (e.g., contains expected sections) before using for delta; fall back to "first run" if validation fails | [First principles: `writeFileSync` is not atomic on all platforms -- process kill during write produces partial file] |
| Concurrent refinement runs race on `.planning/refinement/` | rare | blocking | Two simultaneous `refinement-init` calls both snapshot the same state, then both write different scan outputs. Second writer overwrites first's findings. Delta computed by either run reflects only its own changes, not the interleaved state. No locking mechanism exists in the codebase. Mitigation: create a lock file (`.planning/refinement/.lock`) during `refinement-init`, check/remove during completion. Simple PID-based lock is sufficient for single-user CLI tool. | [First principles: no locking in codebase; `core.cjs` and `capability.cjs` use bare `writeFileSync` without locks] |
| Empty matrix (single capability project) | common | cosmetic | A project with 1 capability produces 0 pairs, which means matrix.md is a 1x1 grid with only the diagonal (`--`). Dependency graph has 0 rows. Delta computation on matrix/graph diffs must handle the "nothing to compare" case. This is not blocking but delta output should explicitly state "N/A - insufficient capabilities for comparison" rather than producing empty/broken tables | FEATURE.md FN-03; landscape-scan FEATURE.md FN-01: pairs are `A x B where A < B` -- 1 capability = 0 pairs |
| Landscape-scan produces zero findings | rare | cosmetic | FN-02 receives aggregated scan output. If landscape-scan finds no issues, findings/ directory is empty. Delta computation (FN-03) must handle "0 previous findings, 0 current findings" without error. coherence-report already handles this (produces "clean bill of health"). refinement-artifact must not produce a DELTA.md that says "0 added, 0 resolved, 0 changed" on second+ clean runs -- this is valid but noisy | coherence-report FEATURE.md FN-02: "Zero findings: write clean bill of health" |
| Finding changes both type AND severity between runs | rare | degraded | FN-03 categorizes changes as "same ID but different severity, type, or recommendation." If a finding changes from `GAP/HIGH` to `OVERLAP/LOW`, the delta must report this as a single "changed" entry, not two separate changes. The delta format needs to show both dimensions of change. No explicit handling specified. | FEATURE.md FN-03: "findings with same ID but different severity, type, or recommendation" |
| Very large project delta computation (100+ findings) | rare | degraded | Delta diff of 100+ findings is O(n) for ID-based comparison (manageable), but the markdown output of DELTA.md could become very large. Matrix diff for 50+ capabilities is a 50x50 grid comparison (2500 cells). Mitigation: summarize large deltas ("42 findings added, 7 resolved") with detail sections collapsed or paginated | [First principles: 100 findings * structured comparison is computationally trivial, but output volume is the concern] |
| `refinement-write` called with path traversal in artifact name | theoretical | blocking | TC-01 specifies a `refinement-write` CLI route that writes a specific artifact file. If the artifact name parameter is not sanitized, a caller could write to `../../etc/something`. Mitigation: `refinement-write` must validate that the target path resolves within `.planning/refinement/` | [First principles: existing `generateSlugInternal` in `core.cjs:359-366` rejects paths with `/` or `\`, but `refinement-write` accepts full filenames not slugs] |

### Boundary Conditions

- **Zero capabilities in project**: `capability-list` returns `{ capabilities: [] }`. `scan-discover` returns empty list. `refinement-init` should create the directory structure but snapshot is null. No pairs, no findings, no matrix, no graph. Delta is meaningless. Must gracefully produce a "nothing to scan" message rather than empty artifacts. -- `capability.cjs:52-55` returns empty array when no capabilities dir exists
- **First run (no previous state)**: `.planning/refinement/` does not exist. FN-01 must return null snapshot. FN-03 must skip DELTA.md entirely (not create an empty one). `refinement-init` must create the directory. -- FEATURE.md FN-01: "store null -- delta computation will be skipped"; FN-03: "skip DELTA.md creation entirely"
- **Second run after first run had zero findings**: Previous snapshot has empty findings list. Current run also has zero findings. Delta should be skipped or show "no changes" -- not an error. -- [First principles: both states are valid; empty-to-empty diff is a no-op]
- **Finding ID gap (non-sequential)**: If consolidation in landscape-scan removes findings during dedup, IDs may have gaps (FINDING-001, FINDING-003, FINDING-007). Delta must not treat missing sequential IDs as "resolved" findings. Identity is by explicit ID string, not by sequence. -- FEATURE.md TC-02: "Finding identity is by ID (FINDING-001)"
- **Maximum path length**: `.planning/refinement/findings/FINDING-001.md` is 46 chars relative. On macOS HFS+ the component limit is 255 chars. Finding IDs like `FINDING-999` are fine, but the path is safe up to `FINDING-{209 digits}` which is not a realistic concern. -- [First principles: path length]
- **Stale `.planning/refinement/` from failed prior run**: Directory exists with partial files (e.g., matrix.md exists but findings/ is empty, RECOMMENDATIONS.md is missing). FN-01 must handle: snapshot what exists, treat missing files as absent (not errored). `existsSync` + `safeReadFile` pattern from `core.cjs:51-57` handles this. -- `core.cjs:51-57`: `safeReadFile` returns null on any read error
- **RECOMMENDATIONS.md exists but has no frontmatter**: `extractFrontmatter` returns `{}` for content without `---` delimiters. Snapshot will capture content but metadata will be empty. Delta should still work since it diffs content/findings, not metadata. -- `frontmatter.cjs:16`: returns `{}` when no match
- **Pairs checkpoint files from landscape-scan still present**: `pairs/{A}-{B}.complete` files persist across runs (by design for resumability). `refinement-init` must not clear these -- they belong to landscape-scan. The refinement-artifact feature writes to different files (RECOMMENDATIONS.md, DELTA.md, matrix.md, etc.) -- FEATURE.md TC-01: "Checkpoint management reuses landscape-scan's `scan-checkpoint` route"
- **Dependency graph row with pipe characters in capability names**: If a capability slug contains text that looks like a markdown table delimiter, the dependency-graph.md table parsing breaks. Slugs are sanitized by `generateSlugInternal` to `[a-z0-9-]` only, so this is safe. -- `core.cjs:362-363`: slug sanitization strips non-alphanumeric

### Integration Failure Scenarios

- **landscape-scan fails mid-scan** -> refinement-artifact receives partial scan output. Some findings written, some not. Matrix may be incomplete. `refinement-init` already snapshotted the previous state. If the orchestrator still triggers FN-02/FN-03, the delta will compare full previous state against partial current state, producing false "resolved" findings. Mitigation: orchestrator must not trigger refinement-artifact if landscape-scan did not complete successfully. -- FEATURE.md FN-02 receives "aggregated scan output from landscape-scan"
- **coherence-report fails** -> RECOMMENDATIONS.md is not written or partially written. refinement-artifact's FN-02 expects RECOMMENDATIONS.md to exist (written by coherence-report). If missing, the `refinement-write` for RECOMMENDATIONS.md has nothing to write. Delta computation in FN-03 will diff against a missing current RECOMMENDATIONS.md. -- FEATURE.md FN-02: "RECOMMENDATIONS.md is written by coherence-report (not this feature)"
- **change-application fails partway** -> DELTA.md from change-application (per change-application FN-04) is partially written. refinement-artifact's delta computation consumes this. If change-application's DELTA.md conflicts with refinement-artifact's DELTA.md (same filename, same path!), there is a naming collision. -- change-application FEATURE.md FN-04 writes to `.planning/refinement/DELTA.md`; refinement-artifact FEATURE.md FN-03 also writes DELTA.md to same path
- **`gsd-tools capability-list` returns stale data** -> If capabilities are created/deleted between `refinement-init` snapshot and `refinement-write`, the artifact directory structure may reference capabilities that no longer exist. Delta will show findings for non-existent capabilities. -- [First principles: no filesystem watch or cache invalidation]
- **Disk full during `refinement-write`** -> `writeFileSync` throws `ENOSPC`. No error handling exists in codebase (`core.cjs` has zero try/catch around file writes outside of `safeReadFile`). Process crashes with unhandled exception. Mitigation: wrap `refinement-write` in try/catch, report clear error. -- [First principles: `writeFileSync` throws on disk full; no try/catch in write paths per grep of codebase]

### Existing Error Handling (gaps)

- **`core.cjs`**: `safeReadFile` (line 51) handles read errors by returning null. But no equivalent `safeWriteFile` exists -- all writes use bare `writeFileSync` with no error handling. New `refinement-write` route will need its own error handling. -- `core.cjs:35`, `capability.cjs:39`, `feature.cjs:44`
- **`capability.cjs`**: `cmdCapabilityList` (line 49) handles missing capabilities directory gracefully (returns empty array). But does not handle corrupt CAPABILITY.md files beyond `safeReadFile` returning null (line 66: `if (!content) continue`). Finding card files may have similar corruption scenarios. -- `capability.cjs:52-55, 66`
- **`frontmatter.cjs`**: `extractFrontmatter` (line 14) handles malformed YAML by returning `{}` and writing a warning to stderr. But the warning is not surfaced to the caller -- callers get `{}` with no indication that parsing failed vs. empty frontmatter. Finding cards with frontmatter parse errors would silently lose metadata. -- `frontmatter.cjs:27-29`
- **No write atomicity anywhere**: The codebase uses `writeFileSync` directly. If the process is killed mid-write, files are left in a corrupted state. For refinement artifacts that span multiple files (matrix + findings + graph + recommendations), a partial write leaves the directory in an inconsistent state. No rollback mechanism exists. -- [First principles: `writeFileSync` is not atomic; grep confirms no temp-file-then-rename pattern except `core.cjs:34-36` for large JSON output]

### Known Issues in Ecosystem

- **Node.js `fs.writeFileSync` is not atomic**: On POSIX systems, `writeFileSync` calls `open` + `write` + `close`. A crash between `open` and `close` produces a truncated file. The standard mitigation is write-to-temp-then-rename (rename is atomic on most filesystems). This is relevant because refinement-artifact writes multiple interdependent files. -- [Node.js fs docs](https://nodejs.org/api/fs.html)
- **Node.js `fs.mkdirSync({ recursive: true })` ENOENT on older Node**: On Node.js v10.15.3, `mkdirSync` with `recursive: true` could throw `ENOENT` in race conditions when parent directories are being created concurrently. Fixed in later versions but worth noting for minimum Node version requirements. -- [nodejs/node#27293](https://github.com/nodejs/node/issues/27293)
- **Markdown table parsing without a library**: The codebase does not use a markdown parsing library (zero runtime deps). Delta computation on matrix.md and dependency-graph.md requires parsing markdown tables. Hand-rolled table parsing is fragile: trailing pipes, inconsistent spacing, escaped pipes in cell content, and missing separator rows are all edge cases. Since the tables are machine-generated (by this system), format is controlled, but any manual edits to these files could break parsing. -- [First principles: zero runtime deps constraint from core context; no markdown parser in `package.json`]
- **`fs.readdirSync` does not guarantee order**: The order of entries returned by `readdirSync` is filesystem-dependent. On HFS+ (macOS) it is typically alphabetical, on ext4 (Linux) it is inode order. Finding file enumeration (`findings/FINDING-*.md`) must sort explicitly after reading. -- [Node.js fs docs](https://nodejs.org/api/fs.html)

### DELTA.md Naming Collision (Critical Design Issue)

Both `refinement-artifact` (FN-03) and `change-application` (FN-04) specify writing DELTA.md to `.planning/refinement/DELTA.md`. These are different artifacts:

- **change-application's DELTA.md**: execution log of applied/failed/skipped mutations
- **refinement-artifact's DELTA.md**: semantic diff of findings/matrix/graph between runs

This is a naming collision that will cause one to overwrite the other depending on execution order. The FEATURE.md specs are inconsistent. The planner must resolve this -- either rename one (e.g., `EXECUTION-LOG.md` for change-application) or merge them into a single artifact with sections.

Source: refinement-artifact FEATURE.md FN-03 vs. change-application FEATURE.md FN-04 + TC-02.
