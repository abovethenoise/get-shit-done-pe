## Domain Truth Findings

### First Principles

- **Semantic diff requires stable identity anchors.** A delta between two runs is only meaningful if entities can be matched across snapshots. Without a stable key (e.g., FINDING-001), the diff degenerates into "everything removed, everything added" -- which is noise, not signal. This is the fundamental problem JaVers solves with entity ID configuration and why its Set algorithm produces only ValueAdded/ValueRemoved against matched keys. — [First principles: identity is prerequisite to comparison; without it, set difference is undefined. Validated by JaVers diff design: https://javers.org/documentation/diff-configuration/]

- **Snapshots must be immutable and complete at capture time.** Martin Fowler's Snapshot pattern establishes that a snapshot is a non-temporal, immutable view of state at a specific moment. If the snapshot can be mutated after capture (or is captured mid-write), the delta becomes unreliable. For refinement-artifact, FN-01's pre-scan snapshot must capture state atomically before any writes begin. — [https://martinfowler.com/eaaDev/Snapshot.html]

- **Three-category delta (added/removed/changed) is the minimal complete diff taxonomy for entity sets.** Given two sets of keyed entities, exactly three things can differ: keys present only in the new set (added), keys present only in the old set (removed), and keys present in both but with different attribute values (changed). This is mathematically exhaustive -- no fourth category exists. The spec's FN-03 correctly adopts this taxonomy. — [First principles: set theory -- symmetric difference partitions into left-only, right-only, and intersection-with-differences. Confirmed by data diff literature: https://www.thedataops.org/data-diff/]

- **Delta is only as precise as the identity granularity allows.** If two conceptually different findings share an ID (collision), the delta will show a "changed" finding when it should show "removed + added." If one finding splits into two across runs, the delta cannot express this without a convention for ID lineage. The ID scheme (FINDING-{id}) must be injected at creation time and remain stable through the finding's lifecycle. — [First principles: identity resolution determines diff quality. If f(key) is not injective, diffs lose precision. Supported by EF Core identity resolution documentation: https://learn.microsoft.com/en-us/ef/core/change-tracking/identity-resolution]

### Universal Constraints

- **Only-current-plus-delta is a lossy representation.** The spec explicitly scopes out historical archives (EU-01 out-of-scope). This means after N runs, only the last two states are recoverable (current + delta from previous). If a finding was added in run 2, removed in run 3, and re-added in run 5, there is no record of its history. This is a deliberate tradeoff but it is a constraint that cannot be worked around at the artifact layer without changing the retention model. — [First principles: without storing snapshots or a log of all deltas, intermediate states are irrecoverable. This is the fundamental event-sourcing vs. snapshot tradeoff: https://microservices.io/patterns/data/event-sourcing.html]

- **Convention-over-configuration for directory layout removes a class of integration bugs.** When multiple agents/features write to the same directory structure, a fixed layout (`.planning/refinement/`) eliminates path negotiation. Maven and Rails proved this pattern at scale -- predictable paths mean consumers never need to ask "where is X?" The constraint is: every consumer must hard-code the same path convention, and changing the convention is a breaking change. — [https://en.wikipedia.org/wiki/Convention_over_configuration ; https://books.sonatype.com/mvnref-book/reference/installation-sect-conventionConfiguration.html]

- **Markdown tables are the correct format for LLM-agent-consumed structured data.** Empirical testing shows Markdown-KV format achieves ~60.7% accuracy for LLM comprehension of tabular data, significantly outperforming CSV and approaching JSON. Mermaid diagrams, by contrast, are unreliable for LLM generation (syntax errors are common) and harder to diff textually. The spec's "no Mermaid, all markdown tables" constraint is well-grounded. — [https://www.improvingagents.com/blog/best-input-data-format-for-llms/]

- **Semantic diff cannot rely on textual diff tools.** A textual diff (e.g., Unix diff) operates on lines, not entities. Reordering rows in a markdown table produces a large textual diff but zero semantic change. Conversely, changing a single cell value in a large table produces a small textual diff that may be hard to locate semantically. TC-02's constraint to diff by entity identity (finding IDs, matrix cell coordinates, dependency tuples) rather than raw text is a necessary correctness requirement, not a preference. — [https://dl.acm.org/doi/10.1145/3122975.3122976 (type-directed diffing); https://semanticdiff.com/docs/what-is-semanticdiff/]

### Validated Assumptions

- **First-run special case (no delta) is correct and unavoidable.** A delta requires two states. On the first run, only one state exists. Generating a DELTA.md with "everything added" would be technically accurate but informationally useless -- the user already has the full report. The spec's decision to skip DELTA.md on first run aligns with how every snapshot-comparison system handles the cold-start case. — [First principles: delta(null, S1) = S1 itself, which adds no information beyond the report. Supported by event sourcing snapshot initialization patterns: https://microservices.io/patterns/data/event-sourcing.html]

- **Overwrite-on-each-run for artifacts (FN-02) is correct for current-state reporting.** Since the goal is "what is true now" rather than "what was true historically," overwriting matrix.md, dependency-graph.md, and findings/ on each run avoids stale data accumulation. The delta captures what changed; the artifacts capture what is. This separation of concerns (state vs. change) is fundamental to temporal data design. — [https://martinfowler.com/eaaDev/timeNarrative.html]

- **Centralizing file I/O in CLI routes (TC-01) rather than having agents write directly is a sound isolation pattern.** When multiple agents could write to the same directory, a single writer (gsd-tools.cjs via refinement-init / refinement-write) prevents race conditions, ensures consistent formatting, and provides a single point of validation. This is the same principle behind centralized audit log infrastructure. — [https://www.inscopehq.com/post/audit-trail-requirements-guidelines-for-compliance-and-best-practices]

### Domain Risks

- **Finding ID instability across runs will corrupt deltas.** If the coherence-report feature generates findings with non-deterministic IDs (e.g., based on discovery order, which may vary), the same conceptual finding could get different IDs across runs, making the delta show spurious adds/removes. The ID generation strategy must be deterministic or content-derived (e.g., hash of capability pair + finding type). This is the single highest-risk design decision for delta quality. — [First principles: identity stability is prerequisite to meaningful diff; unstable keys produce noise. Same problem as entity identity resolution in change tracking: https://learn.microsoft.com/en-us/ef/core/change-tracking/identity-resolution]

- **Matrix cell comparison assumes consistent matrix dimensions.** If capabilities are added or removed between runs, the matrix changes shape. The delta must handle not just changed cell values but also added/removed rows and columns. A naive "compare cell (i,j)" approach fails when row/column indices shift. The comparison must be keyed by capability names, not positions. — [First principles: positional comparison breaks under structural change; key-based comparison is required. Same issue as list diffing with the Set algorithm vs. index-based Simple algorithm in JaVers: https://javers.org/documentation/diff-configuration/]

- **Snapshot atomicity gap.** FN-01 snapshots before the scan, but if the scan partially fails and leaves some artifacts written and others not, the post-scan state is inconsistent. The delta would then compare a clean pre-scan state against a partially-written post-scan state, producing a misleading delta. The write step (FN-02) should be all-or-nothing, or the delta computation should validate artifact completeness before running. — [First principles: comparing a consistent snapshot against an inconsistent state produces undefined results. This is the same problem as dirty reads in database isolation levels.]
